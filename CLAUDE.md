# 全局 Schema — LLM Wiki 知识库

本仓库是一个基于 Karpathy [LLM Wiki](llm-wiki.md) 规范的多领域个人知识库。由 **LLM 维护**、**Obsidian 浏览**、**Git 管理**。

> Obsidian 是 IDE；LLM 是程序员；Wiki 是代码库。

---

## 一、仓库架构
```
knowledge/
├── CLAUDE.md              # ← 本文件：全局规则（所有领域通用）
├── llm-wiki.md            # LLM Wiki 规范原文（Karpathy）
├── index.md               # 顶层领域目录 — 各领域的轻量索引与路由
├── assets/                # 共享媒体资产（跨领域共用）
├── <领域>/                # 每个领域独立目录
│   ├── CLAUDE.md          # 领域规则：名称约定、分类体系、qmd collection 名等
│   ├── raw/               # 不可变层 — 来源文档（只读）
│   │   ├── articles/      # 网络文章、博客
│   │   ├── papers/        # 学术论文
│   │   ├── books/         # 书籍章节
│   │   ├── videos/        # 视频转录
│   │   ├── podcasts/      # 播客转录
│   │   ├── others/        # 其他来源
│   │   └── archive/       # 已归档（ingest 后移动）
│   ├── wiki/              # 编译输出层 — LLM 专属工作区
│   │   ├── index.md       # 总目录：wiki 页面索引（按分类组织）
│   │   └── log.md         # 操作日志：仅追加记录（ingest/query/lint）
│   └── notes/             # 个人笔记区 — 用户手写，LLM 不修改
│
└── <新领域>/              # 新增领域时复用上述结构
```

### 三层分离（每个领域内）

| 层 | 目录 | 权限 | 说明 |
|---|---|---|---|
| **不可变层** | `raw/` | **绝对只读** | 来源文档按类型分子目录（articles、papers、books、videos、podcasts、others）。ingest 后移动到 `archive/`。LLM 从中读取但**绝不修改内容**。 |
| **编译输出层** | `wiki/` | **LLM 专属** | LLM 创建、更新、提炼知识页面，解决矛盾，维护交叉引用。人类可阅读，按需手动修改。 |
| **个人笔记区** | `notes/` | **禁止修改** | 用户手写的个人笔记、日记、思考。LLM **绝不写入或修改**此目录。 |

### Schema 分层

- **全局规则**（本文件 `CLAUDE.md`）：所有领域通用的架构约定、文件规范、操作原则。
- **领域规则**（`<领域>/CLAUDE.md`）：该领域特有的名称约定、分类体系、qmd collection 名称、领域术语等。新增领域时必须创建。

### 顶层 `index.md`

`knowledge/index.md` 不是领域内容的一部分，而是**跨领域的轻量路由目录**。LLM 在判断问题涉及哪些领域时优先读取它，再决定深入哪个领域的 `wiki/index.md`。

**内容示例**：

```markdown
## 领域目录

- [[AI/wiki/index|AI]] — AI 编程工具、方法论与实践
- [[心理学/wiki/index|心理学]] — 认知科学、行为经济学
- [[投资/wiki/index|投资]] — 价值投资、量化策略
```

**维护者**：由 LLM 在新增/删除领域时更新，和人类 wiki 页面一样遵循标准 frontmatter。

## 二、核心操作（概述）

知识库有三种操作。它们的**完整工作流由独立的 skill 实现**，此处只说明概念：

| 操作 | 含义 |
|---|---|
| **Ingest** | 将 `raw/` 中的新素材阅读、提取、整合到 `wiki/`。可能触及 10+ 页面，更新 `index.md` 和 `log.md`。 |
| **Query** | 基于 `wiki/` 内容回答问题。有价值的答案可归档为新 wiki 页面，使探索成果复合增长。 |
| **Lint** | 定期健康检查：孤立页面、矛盾主张、陈旧内容、缺失交叉引用、数据空白。 |

## 三、导航文件

每个领域的 `wiki/` 中有两个特殊文件，帮助 LLM 和人类在 wiki 增长时导航：

### `wiki/index.md` — 领域内容总目录

- 按类别组织（entities、concepts、summaries、comparisons、overviews、syntheses 等）。
- 每个页面列出 `[[维基链接]]`、一行摘要、可选元数据（日期、素材数量）。
- LLM 在每次 ingest 后更新它。
- Query 时，LLM 先阅读 `index.md` 定位相关页面，再深入阅读。
- 中小规模下（~100 素材、~数百页面）足够有效，无需嵌入 RAG。

### `log.md` — 时间导向操作日志

- **仅追加**，按时间顺序记录发生了什么、何时发生。
- 条目格式统一前缀，便于解析：

```markdown
## [2026-05-08] ingest | 文章标题
- 关键要点摘要...

## [2026-05-08] query | 问题简述
- 回答摘要...

## [2026-05-08] lint | 健康检查
- 发现问题与处理...
```

## 四、Obsidian 生态与 Skill 使用

本仓库的底层是本地 Markdown 文件，但使用 **Obsidian** 作为浏览和阅读层。因此文件操作应遵循 Obsidian 生态规范。

### 默认使用 Obsidian Skills

当用户要求创建、编辑、读取、搜索、管理笔记或文件时，**优先调用对应的 Obsidian skill**，而非直接使用通用文件操作工具。

| 场景 | 触发的 Skill |
|---|---|
| 创建/编辑 `.md` 文件 | `obsidian-markdown` |
| 创建/编辑 `.canvas` 文件 | `json-canvas` |
| 创建/编辑 `.base` 文件 | `obsidian-bases` |
| 与 Obsidian 应用交互（读取/创建/搜索笔记、管理任务属性等） | `obsidian-cli` |
| 提取网页内容 | `defuddle` |

### 例外情况

仅在以下情况可直接使用通用文件操作：
- 用户明确要求不使用 skill
- 操作目标明确不是 Obsidian 文件（如纯代码文件、配置文件）
- skill 不可用时回退到通用工具

### 搜索规范

- **标签搜索**：优先使用 `obsidian tag` 命令，比全文搜索更规范。
- **wiki 页面搜索**：小规模使用 `index.md`；规模增大后使用 qmd CLI（每个领域独立 collection，具体名称见该领域 `CLAUDE.md`）。

## 五、文件规范

### Markdown 页面规范

所有 wiki 页面的 `.md` 笔记应包含标准 frontmatter：

```markdown
---
title: 页面标题
date: 2026-05-08
tags: [tag1, tag2]
source_count: 3
---
```

- `title`：页面标题
- `date`：创建或最后更新日期
- `tags`：主题标签（可用 frontmatter 或正文 `#标签`）
- `source_count`：该页面基于多少个 raw 素材（可选）

### 链接与引用

- **内部链接**：使用 `[[维基链接]]` 语法，Obsidian 中可点击跳转。
- **图片/附件**：使用 `![[文件名]]` 嵌入语法。
- **附件存放**：附件统一存放顶层 `assets/`

### 标签系统

- 正文内标签：`#标签`
- frontmatter 标签：`tags: [标签1, 标签2]`
- 领域特有标签体系在 `<领域>/CLAUDE.md` 中定义。

## 六、Wiki 页面分类约定

`wiki/` 内的页面按内容类型分类，建议统一在 `index.md` 中组织：

| 类型 | 说明 | 示例 |
|---|---|---|
| **summaries** | 单个来源的摘要 | `[[某篇文章摘要]]` |
| **entities** | 实体页面（人、组织、产品、框架） | `[[Claude Code]]` |
| **concepts** | 概念页面（方法论、技术、理论） | `[[RAG]]` |
| **comparisons** | 对比分析 | `[[RAG vs Fine-tuning]]` |
| **overviews** | 领域概览 | `[[AI 编程工具概览]]` |
| **syntheses** | 综合结论、最终判断 | `[[LLM 编程最佳实践]]` |

页面命名使用中文或英文，保持领域 `CLAUDE.md` 中约定的命名风格。

| 职责 | 人类 | LLM |
|---|---|---|
| **策展来源** | 选择、收集、审核 raw 素材 | — |

| **交叉引用与一致性** | — | 维护链接、标记矛盾、更新陈旧内容 |
| **机械劳动** | — | 总结、分类、链接、簿记 |

## 八、扩展工具

- **qmd**：wiki 本地搜索引擎（BM25 + 向量搜索 + LLM 重排序），每个领域独立 collection。具体 collection 名称和配置见该领域 `CLAUDE.md`。
- **Obsidian Web Clipper**：浏览器插件，快速将网页转为 markdown 存入 `raw/`。
- **Marp**：基于 markdown 的幻灯片格式，可直接从 wiki 内容生成演示文稿。
- **Dataview**：Obsidian 插件，基于 frontmatter 运行查询，生成动态表格和列表。

## 九、多领域查询与搜索策略

### 规模阈值

限制不是"页面数量"，而是 **LLM 的上下文窗口**。每次 query 时，LLM 需要先读 `index.md` 来定位相关页面。如果 `index.md` 太长，会占用大量上下文，留给实际页面内容的空间就少了。

| 指标 | 小规模（用 index.md） | 大规模（用 qmd） |
|---|---|---|
| **单领域 wiki 页面数** | < 200 页 | ≥ 200 页 |
| **index.md 行数** | < 500 行 | ≥ 500 行 |
| **领域总数** | < 5 个 | ≥ 5 个 |

**判断标准**：如果读一遍 `index.md` 再深入读 2-3 个具体页面后，上下文明显紧张（回答开始遗漏细节），就该切换。

### 小规模：index.md 模式

1. LLM 快速扫一眼目录结构，了解有哪些领域
2. 根据用户问题内容判断涉及哪些领域（用户也可能显式指定）
3. 并行读取相关领域的 `wiki/index.md`
4. 从 `index.md` 中定位最相关的页面链接
5. 读取具体页面内容
6. 综合回答，带 `[[引用]]`

### 大规模：qmd 模式

[qmd](https://github.com/tobi/qmd) 提供三种搜索命令：

| 命令 | 原理 | 速度 | 适用场景 |
|---|---|---|---|
| `qmd search` | BM25 关键词匹配 | **即时** | 默认首选 |
| `qmd vsearch` | 向量语义搜索 | **慢**（约1分钟冷启动加载本地模型） | 关键词不确定、概念模糊时 |
| `qmd query` | 混合搜索 + LLM 重排序 | **最慢** | 复杂问题，要最高质量时 |

**检索流程**：
1. 先用 `qmd search "问题" --json -n 20` 做 BM25 快速筛选（默认**跨所有 collection**，天然支持跨领域）
2. 结果质量不够时，用 `qmd query "问题" --json -n 10` 做混合精排
3. 根据结果中的文件路径读取具体内容（`qmd get <path>` 或通用文件工具）
4. 综合回答

**限制**：`-c` 只能限定**单个** collection。若用户明确只想查某几个领域（不是所有），需分别搜索再合并结果，或搜全部后由 LLM 按文件路径过滤。

**索引维护**（由 ingest skill 自动调用）：

```bash
qmd update      # 增量更新索引（快，每次 ingest 后执行）
qmd embed       # 更新向量嵌入（慢，按需执行）
```

## 十、领域 CLAUDE.md 模板

新增领域时，在 `<领域>/CLAUDE.md` 中至少包含：

```markdown
# <领域名称> 领域规则

## 领域概述

用 1-2 句话描述该领域的核心主题和边界。例如："AI 编程工具、方法论与实践"或"认知科学、行为经济学和决策心理学"。

## 分类体系

- 该领域 wiki 页面的主要分类（替换或扩展默认分类）
- 每个分类对应的子目录名称和说明

## 标签体系

领域初始标签（统一使用英文，便于检索和 Dataview 查询）。标签是动态扩展的——以下只是种子标签，LLM 在 ingest 时会根据素材内容自动补充：

- `#tag1` — 标签说明
- `#tag2` — 标签说明

**标签添加原则**：
- 初始标签作为 ingest 时的参考基准
- 当素材涉及新的子主题时，自动创建新标签
- 定期 review 标签使用情况，合并过于细分的标签

## qmd 配置

- collection 名称：`knowledge-<领域小写>`
- 索引路径：`./wiki/`

## 特殊约定

- 该领域特有的格式、模板或规则
```

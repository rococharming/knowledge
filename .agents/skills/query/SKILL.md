# Query — 知识库查询

基于知识库 `wiki/` 中的内容回答用户问题。优先使用 wiki 中的信息，
而非模型的训练数据，因为 wiki 包含用户个人的知识积累。

## 查询策略

根据领域规模自动选择查询方式：

| 场景 | 阈值 | 策略 |
|---|---|---|
| 小规模 | 涉及领域 wiki 页面数总和 ≤ 300 | index.md 模式 |
| 大规模 | 涉及领域 wiki 页面数总和 > 300 | qmd 搜索模式 |

> 页面数从顶层 `index.md` 中读取，每个领域标注了当前 wiki 页面数。

## 查询流程

### 1. 识别涉及领域

快速扫一眼知识库目录结构（`knowledge/` 下的领域文件夹）。

根据用户问题判断涉及哪些领域：
- 用户**显式指定**了领域（如"在 Rust 知识库里..."）→ 直接使用该领域
- 用户**未指定** → 根据关键词推断（如"所有权"→Rust，"RAG"→AI）
- 涉及**多个领域** → 同时查询多个
- 不确定 → 询问用户，不要猜测

### 2. 判断规模并选择模式

**必须先读取顶层 `knowledge/index.md`**，获取各领域的 wiki 页面数，计算涉及领域的页面总和：

```
小规模 (≤300 页):  index.md 模式
大规模 (>300 页):  qmd 搜索模式
```

**关键原则**：
- **小规模下绝对不使用 qmd 搜索**。wiki/index.md 通常只有几十行，人工浏览快速高效，qmd 反而增加不必要的延迟（尤其缺少 embeddings 时）。
- **大规模下不要直接读取 wiki/index.md 来手动筛选**。页面数过多时人工浏览效率极低且容易遗漏，应先用 qmd 搜索精准定位。

### 3. 执行查询

#### index.md 模式（小规模，≤300 页）

**此模式下禁止使用 qmd 任何命令。**

1. **并行读取涉及领域的 `wiki/index.md`**
   - 单领域：直接读取该领域的 `wiki/index.md`
   - **多领域：使用 Agent 工具并行读取**。每个子 agent 只读一个领域的 `wiki/index.md`，返回该领域下与问题相关的页面列表
   - 读取 index.md 的目的是**按分类浏览**，定位最相关的页面链接

2. **从 index.md 中定位最相关的页面**
   - 按分类（summaries、entities、concepts、recipes 等）浏览
   - 根据问题关键词匹配页面标题和摘要
   - 如果多个页面都相关，全部记下

3. **读取具体页面内容**
   - 并行读取所有定位到的页面
   - 如果页面之间存在引用关系，一并读取被引用的页面

4. 综合信息，形成回答

#### qmd 模式（大规模，>300 页）

**重要**：大规模领域下 wiki/index.md 可能包含数百个页面链接，不要直接读取 wiki/index.md 来人工筛选。

1. **执行 `qmd update`** — 确保索引最新（跨机器切换后索引可能过期）
2. **确定 collection** — 从各领域的 `CLAUDE.md` 中读取 qmd collection 名称
3. **执行搜索**（用用户问题中的关键词）：
   ```bash
   # 优先使用 qmd query（混合搜索，效果更好）
   qmd query "用户问题中的关键词" -c <collection1> -c <collection2> --full -n 10

   # 如果 query 不可用，回退到 search（纯 BM25）
   qmd search "用户问题中的关键词" -c <collection1> -c <collection2> --full -n 10
   ```
4. **读取搜索结果中的页面**
5. 如果 qmd 搜索结果不足，再考虑读取 wiki/index.md 作为补充
6. 综合信息，形成回答

> 搜索默认覆盖所有涉及的 collection。如果用户问题明确限定在某领域，只搜索该 collection。

### 4. 组织回答

回答结构：

```markdown
## [问题核心要点]

[综合回答，整合多个来源的信息]

### 关键要点
- ...
- ...

### 相关页面
- `子目录/[[页面名]]` — 简要说明相关点
- `子目录/[[页面名]]` — ...

> 标注页面所在的 wiki 子目录（如 `entities/[[rustup]]`、`concepts/[[所有权]]`、`recipes/[[RsProxy 镜像源配置]]`），方便用户快速定位文件位置。
```

**检索路径透明化**：
- 回答应体现检索过程，简要说明"从 [[Rust Wiki 索引]] 中定位到 [[RsProxy 镜像源配置]]"等
- 让用户感知到答案是基于知识库查询而非模型记忆

**引用规范**：
- 回答中涉及的事实、观点必须标注来源，使用 `[[页面名]]` 格式
- 如果多个页面支持同一观点，引用最关键的 1-2 个
- 引用应自然融入句子，不要堆砌

**回答原则**：
- 基于 wiki 内容，不要引入训练数据中的信息（除非 wiki 中确实没有，可补充说明"知识库中未涉及，以下是补充..."）
- 如果 wiki 内容有矛盾，指出矛盾并说明各来源的立场
- 如果信息不足，明确说明"知识库中关于 X 的信息有限"
- 跨领域问题时，显式说明各领域的视角差异

### 5. 可选：归档为 wiki 页面

如果回答具有长期价值（如系统性的对比分析、综合结论、最佳实践总结），
可询问用户是否需要归档为新 wiki 页面：

- 概念解释 → `concepts/`
- 对比分析 → `comparisons/`
- 最佳实践/综合结论 → `syntheses/`
- 领域概览 → `overviews/`

归档时：
1. 创建新页面，包含标准 frontmatter（title, date, tags, source_count）
2. 内容精炼为可独立阅读的 wiki 页面
3. **wiki 页面内部链接格式**：归档后的 wiki 页面中，所有内部链接使用纯 `[[页面名]]` 格式（**禁止**使用 `子目录/[[页面名]]` 或 `[[子目录/页面名]]` 等带路径前缀的写法）
   - ❌ 错误：`entities/[[rustup]]`、`recipes/[[RsProxy 镜像源配置]]`
   - ✅ 正确：`[[rustup]]`、`[[RsProxy 镜像源配置]]`
4. 更新该领域的 `wiki/index.md` 添加链接
5. 更新顶层 `index.md` 中该领域的页面计数
6. 在 `wiki/log.md` 追加 Query 记录

**后续交互中归档的处理**

query 回答后，用户可能在后续对话中才要求归档（如"根据刚才的结果生成 wiki 页面"、"把这个整理成 wiki"）。这种情况下：

1. **仍然触发此 skill**（description 中已包含归档触发条件）
2. **复用之前的查询结果**，不需要重新读取 wiki（除非需要补充验证）
3. **直接执行归档流程**：创建页面 → 更新索引 → 追加日志
4. **追加新日志**，绝不修改之前任何记录（即使之前已写过日志）

**日志规则（重要）**：

- **只读查询不写日志**。单纯的问答没有修改 wiki 状态，不记录。
- **只有归档（创建/更新 wiki 页面）时才写日志**。日志记录的是"查询成果被固化到 wiki"这一事件。
- **绝不修改已有日志条目**。如果用户在后续交互中才决定归档，追加一条新日志，不回头修改之前的任何记录。

**log.md 记录格式**：

```markdown
## [YYYY-MM-DD] Query | <问题摘要>

- 基于: `[[来源页面1]]`, `[[来源页面2]]`, ...
- 创建: `[[新页面名]]`（如归档了新页面；如未归档新页面则省略此行）
- 类型: <concepts/comparisons/syntheses/overviews/recipes>
```

**示例**：
```markdown
## [2026-05-11] Query | rustup 命令用法

- 基于: `[[rustup]]`, `[[Rust 安装与开发环境配置]]`
- 创建: `[[rustup 命令速查]]`
- 类型: recipes
```

## 规模阈值速查

| 查询类型 | 小规模 | 大规模 |
|---|---|---|
| 单领域 | ≤ 300 页 | > 300 页 |
| 多领域 | 总和 ≤ 300 页 | 总和 > 300 页 |

当前各领域的页面数见 `knowledge/index.md`。

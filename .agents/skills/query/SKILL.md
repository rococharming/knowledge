# Query — 知识库查询

基于知识库 `wiki/` 中的内容回答用户问题。优先使用 wiki 中的信息，而非模型的训练数据。

## 查询策略

根据领域规模自动选择查询方式：

| 场景 | 阈值 | 策略 |
|---|---|---|
| 小规模 | 涉及领域 wiki 页面数总和 ≤ 300 | index.md 模式 |
| 大规模 | 涉及领域 wiki 页面数总和 > 300 | qmd 搜索模式 |

页面数从顶层 `index.md` 中读取。关键原则：
- **小规模下绝对不使用 qmd**。wiki/index.md 通常只有几十行，人工浏览高效，qmd 反而增加延迟。
- **大规模下不要直接读取 wiki/index.md 来手动筛选**。页面过多时先用 qmd 精准定位。

## 查询流程

### 1. 识别涉及领域

快速扫一眼知识库目录结构，根据用户问题判断涉及领域：
- **显式指定**（如"在 Rust 知识库里..."）→ 直接使用
- **未指定** → 根据关键词推断（如"所有权"→Rust，"RAG"→AI）
- **多领域** → 同时查询多个
- **不确定** → 询问用户，不要猜测

### 2. 读取顶层 index.md 判断规模

**必须先读取顶层 `knowledge/index.md`**，获取各领域 wiki 页面数，计算涉及领域的页面总和，选择模式。

### 3. 执行查询

#### index.md 模式（小规模，≤300 页）

**此模式下禁止使用 qmd 任何命令。**

1. **并行读取涉及领域的 `wiki/index.md`**（多领域时用 Agent 并行读取）
2. **从 index.md 中定位最相关的页面**：按分类浏览，根据关键词匹配标题和摘要
3. **并行读取定位到的页面**，如果页面间有引用关系，一并读取被引用页面
4. 综合信息，形成回答

#### qmd 模式（大规模，>300 页）

1. **执行 `qmd update`** — 确保索引最新
2. **确定 collection** — 从各领域 `CLAUDE.md` 读取 qmd collection 名称
3. **执行搜索**：
   ```bash
   qmd query "关键词" -c <collection1> -c <collection2> --full -n 10
   ```
   若 `query` 不可用，回退到 `qmd search`
4. **读取搜索结果中的页面**
5. 若搜索结果不足，再读取 wiki/index.md 补充
6. 综合信息，形成回答

> 搜索默认覆盖所有涉及的 collection。若问题明确限定某领域，只搜索该 collection。

### 4. 组织回答

```markdown
## [问题核心要点]

[综合回答，整合多个来源的信息]

### 关键要点
- ...
- ...

### 相关页面
- `子目录/[[页面名]]` — 简要说明
- `子目录/[[页面名]]` — ...
```

**检索路径透明化**：简要说明从哪些索引定位到哪些页面，让用户感知答案是基于知识库查询。

**引用规范**：
- 事实、观点必须标注来源，使用 `[[页面名]]` 格式
- 多页面支持同一观点时，引用最关键的 1-2 个
- 引用自然融入句子，不要堆砌

**回答原则**：
- 基于 wiki 内容，不引入训练数据（除非 wiki 中没有，可补充说明）
- 若 wiki 内容有矛盾，指出矛盾并说明各来源立场
- 若信息不足，明确说明"知识库中关于 X 的信息有限"
- 跨领域问题时，显式说明各领域视角差异

### 5. 可选：归档为 wiki 页面

若回答具有长期价值（系统性对比分析、综合结论、最佳实践），可询问用户是否归档：

- 概念解释 → `concepts/`
- 对比分析 → `comparisons/`
- 最佳实践/综合结论 → `syntheses/`
- 领域概览 → `overviews/`

归档流程：
1. 创建新页面，包含标准 frontmatter（title, date, tags, source_count）
2. 内容精炼为可独立阅读的 wiki 页面
3. 内部链接使用纯 `[[页面名]]` 格式（不带路径前缀），且**仅指向同一领域 `wiki/` 内的页面**
4. 更新 `wiki/index.md` 和顶层 `index.md`
5. 在 `wiki/log.md` 追加 Query 记录

**后续交互中归档的处理**

用户在后续对话中要求归档（如"把刚才的回答整理成 wiki"）时：
1. **仍然触发此 skill**
2. **复用之前的查询结果**，不需要重新读取 wiki
3. **直接执行归档**：创建页面 → 更新索引 → 在 `log.md` 尾部追加日志

**日志规则**：
- **只读查询不写日志**
- **只有归档（创建/更新 wiki 页面）时才写日志**
- **绝不修改已有日志条目**。后续归档时追加新日志，使用 `Edit` 工具在文件末尾插入。

**log.md 记录格式**：

```markdown
## [YYYY-MM-DD] Query | <问题摘要>

- 基于: `[[来源页面1]]`, `[[来源页面2]]`, ...
- 创建: `[[新页面名]]`（如归档了新页面；如未归档则省略）
- 类型: <concepts/comparisons/syntheses/overviews/recipes>
```

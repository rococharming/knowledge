# Ingest Skill

## 概述

将 `<domain>/raw/` 下的 Markdown 素材阅读、提取、整合到 `<domain>/wiki/`，
并更新索引、日志和 qmd 搜索索引。

**核心职责边界**：ingest 只处理"新素材 vs 现有页面"的矛盾与补充。
全站性的陈旧扫描、孤立页面检测、跨页面矛盾排查由 `lint` 负责。

## 触发条件（必须使用的场景）

- "ingest 这篇文章"
- "处理 raw/ 里的新素材"
- "把 xxx.md 整合到知识库"
- "提炼 raw/articles/ 下的内容"
- "归档 raw/ 中的文件"
- "更新 wiki 内容"
- "补充 [[某页面]] 的内容"
- 任何涉及将 raw 素材转化为 wiki 页面的请求
- 用户要求"处理"、"整理"、"提炼" raw/ 目录下文件的场景

## 输入判断

### 用户指定了具体文件

解析用户提供的文件路径，确认文件存在于 `<domain>/raw/`（或子目录）下。

**单个文件：**
1. 读取文件内容
2. 询问用户是否需要**交互式提炼**：
   - "这篇文章需要我逐轮和你确认提炼角度、分类、重点吗？"
   - 如果用户回答需要（yes/y/是/需要 等），进入交互式模式
   - 如果用户回答不需要（no/n/否/不用 等）或未明确回复，进入自动处理模式
3. 交互式模式下，向用户展示素材的核心观点和结构，询问提炼角度、重点、分类偏好、关联页面等
4. 根据用户反馈调整提炼策略，确认后再写入 wiki

**多个文件（2个及以上）：**
- 默认自动批量处理，不逐轮交互
- 处理完所有文件后，统一报告结果（创建了哪些页面、更新了哪些页面、归档了哪些文件）

### 用户未指定具体文件

1. 扫描 `<domain>/raw/`（不包括 `archive/`），列出所有 `.md` 文件
2. 询问用户要处理哪些文件
3. 根据用户选择的数量，按上述"单个文件"或"多个文件"逻辑处理

## 核心处理流程

### 步骤 1：读取素材

读取用户指定的 raw Markdown 文件。注意：raw/ 内容只读，处理过程中绝不修改原文件。

### 步骤 2：分析内容并拆解

**核心原则：按需拆分，而非强制合并；但同级细粒度概念应优先聚合为上层主题页**

素材内容足够丰富时，应主动拆分为多个页面。但拆分不等于碎片化——当多个同级细粒度概念各自独立成页的内容量不足、检索价值有限时，应主动聚合为一个上层主题页面。

### 步骤 2a：识别知识点

1. **通读全文**：理解素材的整体结构和内容
2. **识别知识点**：从素材中识别出可以独立成页的知识点，包括：
   - **实体（entities）**：工具、产品、框架、组织（如 rustup、rustc、Cargo、VS Code）
   - **概念（concepts）**：方法论、技术概念、机制（如 所有权、生命周期、Edition、features、语义化版本）
   - **操作指南（recipes）**：具体的配置步骤、安装流程、操作命令（如 国内镜像源配置、VS Code 插件安装）
   - **摘要（summaries）**：整篇文章的高层次摘要

### 步骤 2b：概念聚合判断

**判断是否需要聚合的 4 个标准（满足 2 项及以上即应聚合）：**

| 标准 | 说明 | 示例 |
|---|---|---|
| **内容量不足** | 单个概念的核心内容少于 3-4 个关键要点（表格行、代码示例、特性描述） | 整型只有类型表和字面量写法，不足独立成页 |
| **检索价值有限** | 用户通常以更大的主题查询，而非细分概念 | 用户搜"Rust 数据类型"而非"Rust 整型" |
| **概念层级较低** | 属于某个上层主题下的同级细分 | 整型/浮点型/布尔型/字符型 都属于"标量类型" |
| **来源连续性** | 同一素材中连续介绍的一组同类概念，拆分后缺乏独立叙事 | 一篇文章连续介绍所有基础类型 |

**聚合示例：**

| 情况 | 处理方式 | 结果页面 |
|---|---|---|
| ✅ 素材中连续介绍了整型、浮点型、布尔型、字符型 | 聚合为「数据类型」页面 | `concepts/数据类型` |
| ✅ 素材中介绍了数组、元组、切片等复合类型 | 聚合为「复合类型」或归入「数据类型」 | `concepts/数据类型` 或 `concepts/复合类型` |
| ❌ 素材中详细介绍了所有权系统 | 保持独立 | `concepts/所有权` |
| ❌ 素材中深入解释了生命周期机制 | 保持独立 | `concepts/生命周期` |
| ❌ 素材中分别介绍了 Edition 和语义化版本 | 保持独立（概念层级高，检索价值高） | `concepts/Edition`、`concepts/语义化版本` |

**聚合页面的内容结构：**

聚合页面应使用清晰的内部标题组织各子概念，每个子概念保留足够的细节：

```markdown
# 数据类型

## 标量类型

### 整型
类型表、字面量写法、溢出处理...

### 浮点型
f32/f64、IEEE 754、特殊值...

### 布尔型与字符型
bool 严格检查、char Unicode...

## 复合类型

### 数组
定长同构、索引访问...

### 元组
异构元素、模式匹配...
```

### 步骤 2c：判断独立成页标准

经过聚合判断后，剩余的知识点按以下标准决定是否独立成页：

- 有**实质性描述**（不仅是名字提及，有功能说明、原理解释或使用方法）
- 有**独立检索价值**（用户单独搜索该知识点时，这个页面能提供完整信息）
- **不重复**：独立页面的内容不与其他页面重复

**决策输出**：列出计划创建的所有页面，包括分类、标题、内容范围。对于聚合页面，注明聚合了哪些子概念。

**拆分 vs 不拆分的判断**：
- 一篇文章同时介绍 rustup 安装、rustc 编译和 Cargo 包管理 → 拆分为 rustup（entities）、rustc（entities）、Cargo（entities），同时创建整篇文章的 summaries
- 一篇文章只简单提到"使用 cargo build 构建" → 不需要单独创建 Cargo 页面，在 summaries 或对应实体页面中提及即可
- 一篇文章详细介绍了 Edition 机制和语义化版本规则 → 分别创建 Edition（concepts）和语义化版本（concepts）
- 一篇文章包含具体的 RsProxy 镜像源配置步骤 → 创建 RsProxy 镜像源配置（recipes）

**页面内容范围**：
- summaries/ 页面：整篇文章的高层次摘要，可以引用拆分的其他页面
- entities/ 页面：该工具/产品的功能概述、核心命令/用法、与其他工具的关系
- concepts/ 页面：概念定义、原理解释、使用场景、与其他概念的关系
- recipes/ 页面：问题描述、前置条件、具体步骤、验证方法

**分类自检问题（写入前必须回答）：**
- 这篇文章是单篇来源的摘要吗？→ 创建 summaries/ 页面
- 这篇文章介绍的是一个人、产品、框架或组织吗？→ 创建 entities/ 页面
- 这篇文章解释的是一个方法论、技术或理论吗？→ 创建 concepts/ 页面
- 这篇文章对比了两个或多个事物吗？→ 创建 comparisons/ 页面
- 这篇文章梳理了一个子领域的全景吗？→ 创建 overviews/ 页面
- 这篇文章给出的是最佳实践或决策建议吗？→ 创建 syntheses/ 页面
- 这篇文章提供的是某个具体问题的操作步骤吗？→ 创建 recipes/ 页面

**聚合自检问题（写入前必须回答）：**
- 识别出的概念中，是否有多个属于同一上层主题的同级细分？→ 考虑聚合
- 每个概念页面是否有足够的内容量（≥3-4 个核心要点）？→ 不足则聚合
- 用户是否会单独搜索这个细分概念？→ 不会则聚合
- 聚合后是否保留了每个子概念的完整细节？→ 确保聚合不是简单合并标题

**注意**：上述问题不是互斥的。同一篇 raw 素材可以**同时**满足多个条件，从而产生多个不同分类的页面。例如一篇安装指南既包含整篇文章摘要（summaries），也包含工具介绍（entities）和配置步骤（recipes）。

### 步骤 2d：匹配现有页面

**目标**：判断新素材产生的候选页面是应更新现有页面，还是创建新页面。

1. **读取现有索引**：读取 `<domain>/wiki/index.md`，获取所有现有页面标题与分类
2. **标题预筛**：将步骤 2c 产出的候选页面标题与现有页面比对：
   - **强匹配**：标题相同或语义等价（如"数据类型"≈"基础类型"）→ 进入步骤 2e
   - **弱匹配**：标题部分重叠、同义词、或属于同一上层主题 → 进入步骤 2e，标记为"可能聚合/并入"
   - **无匹配**：全新主题 → 直接创建新页面
3. **qmd 辅助匹配**（可选）：若标题未直接匹配但候选概念较复杂，可用 `qmd search -c knowledge-<domain> "<候选标题关键词>"` 辅助发现相关页面

**分类隔离原则**：即使主题相同，不同分类的页面也应分别维护。例如 `entities/Cargo` 与 `recipes/Cargo 安装配置` 是两个独立页面，不应合并。

### 步骤 2e：矛盾验证与修正

**目标**：确保 wiki 页面呈现的是已验证的准确知识，而非素材原文的机械搬运。

**职责边界**：本步骤只处理"新素材 vs 现有页面"的矛盾。全站性的陈旧扫描、现有页面之间的相互矛盾由 `lint` 负责。

对步骤 2d 匹配到的现有页面：

1. **读取现有页面内容**
2. **主张比对**：提取新素材中的关键事实、数据、定义、版本信息，与现有页面逐条比对
3. **矛盾处理**（若发现不一致）：
   - **立即启用联网搜索**，验证双方主张（优先检索官方文档、权威来源、最新 release note、规范文档）
   - **以验证结果为准，只写入正确版本**：
     - 新素材经核实正确 → 用新素材内容覆盖旧内容
     - 现有页面经核实正确 → 保留现有内容，丢弃新素材中的错误信息
     - 双方均不完全准确 → 采用搜索验证后的最准确表述
   - **wiki 页面中不呈现矛盾过程、判断逻辑、分歧对比或"某来源认为"的并列句式**
4. **内容互补判断**（若无矛盾）：
   - 新素材覆盖的方面与现有页面不重叠 → 作为新章节补充到现有页面
   - 高度重叠（>70%）→ 仅增量更新（补充新细节、更新 source_count）
   - 新素材是现有页面的子主题，且内容量 ≥ 3-4 个核心要点、有独立检索价值 → 创建子页面，在现有页面添加 `[[子页面]]` 链接

### 步骤 3：提炼内容（交互式或自动）

**交互式模式（单个文件 + 用户选择交互）：**
1. 向用户展示素材的核心观点、结构和识别出的知识点清单，**同时标注聚合建议和匹配结果**：
   - 列出所有识别出的知识点，标注哪些是"建议聚合"、哪些是"建议独立"
   - 标注哪些知识点匹配到现有页面（"将更新 [[现有页面]]"）
   - 标注哪些是新主题（"将创建新页面"）
   - 示例："以下知识点中，'数据类型' 匹配到现有页面 [[Rust 数据类型]]，将补充新内容；'Cargo Workspace' 为新主题，将创建新页面"
2. 询问以下问题（可合并提问）：
   - 知识点确认：我识别出了以下可独立成页的知识点，是否遗漏或多余？
   - **聚合确认**：哪些知识点你认为应该聚合或拆分？（可覆盖默认建议）
   - **匹配确认**：对自动匹配的现有页面是否有异议？
   - 提炼角度：你希望从哪个角度总结这篇文章？（技术细节、方法论、实践指导等）
   - 重点标注：有哪些内容你认为特别重要，需要重点呈现？
   - 关联页面：是否需要关联到 wiki 中已有的某个页面？
3. 根据用户反馈生成 wiki 页面草稿（所有计划创建和更新的页面）
4. 向用户展示草稿，确认后再写入 wiki

**自动模式（单个文件用户不交互，或多个文件批量）：**
1. 自主分析素材内容，提取关键信息
2. 自主识别知识点并决定分类
3. 自主匹配现有页面（步骤 2d）并验证矛盾（步骤 2e）
4. 自主决定关联页面
5. 直接生成所有 wiki 页面并写入

### 步骤 4：创建/更新 wiki 页面

创建或更新 `<domain>/wiki/` 下的页面。使用**严格的 frontmatter 格式**：

```markdown
---
title: 页面标题
date: YYYY-MM-DD
tags: [tag1, tag2]
source_count: N
---
```

**frontmatter 规范（必须遵守）：**
- `title`：页面标题，简洁明了
- `date`：创建或最后更新日期，格式 `YYYY-MM-DD`
- `tags`：标签数组，使用英文小写，遵循各领域 `CLAUDE.md` 中的标签体系
- `source_count`：整数，该页面基于多少个 raw 素材（**新建为 1，更新则递增**）
- **禁止**添加任何其他 frontmatter 字段（如 `source`、`author`、`url` 等）

**页面内容规范：**
- 内容必须提炼而非照搬原文，突出结构化和可检索性
- 使用 `[[维基链接]]` 语法创建交叉引用
- wiki 页面中的 `[[维基链接]]` **必须仅指向同一领域 `wiki/` 目录内的页面**，禁止链接到 `raw/`、`notes/` 或其他目录中的文件
- 如果 wiki 中已有相关页面，新页面应至少引用 1-2 个现有页面
- 标签使用英文，遵循各领域 `CLAUDE.md` 中的标签体系
- **同一素材产生的多个页面之间应互相引用**，形成知识网络
- **素材不等于事实**：raw 素材中的主张需经交叉验证后方可写入 wiki。关键事实、版本信息、数据应做快速验证

### 步骤 5：更新领域索引（wiki/index.md）

更新 `<domain>/wiki/index.md`。使用**严格的格式**：

```markdown
---
title: <Domain> Wiki 索引
date: YYYY-MM-DD
---

# <Domain> Wiki 索引

## summaries

- [[页面名]] — 一句话摘要

## entities

- [[页面名]] — 一句话摘要

## concepts

- [[页面名]] — 一句话摘要

## comparisons

## overviews

## syntheses

## recipes
```

**索引规范：**
- 按分类组织，每个分类下只列出属于该分类的页面
- 使用 `[[页面名]]` 格式的 wiki 链接（**不要**使用 `[[路径/文件名.md|页面名]]` 格式）
- 每个链接后附 1-2 句话的摘要
- 如果某个分类下没有页面，保留空分类标题
- 不要重复列出同一个页面

### 步骤 5.5：更新顶层索引（index.md）

**重要：顶层 index.md 位于仓库根目录，文件路径是 `index.md`（不是 `knowledge/index.md` 或任何子目录下的 index.md）。**

读取顶层 `index.md`，统计当前领域 `wiki/` 下所有 `.md` 页面的数量（**排除** `wiki/index.md` 和 `wiki/log.md`），更新该领域条目中的页数标注。

**条目格式：**
```markdown
- [[<domain>/wiki/index|<domain>]] — <领域概述>（<N> 页）
```

**示例：**
```markdown
- [[Rust/wiki/index|Rust]] — Rust 编程语言学习与实践（8 页）
```

**如果该领域不在顶层 index.md 中**，添加新条目并标注页数。

### 步骤 6：更新日志（wiki/log.md）

在 `<domain>/wiki/log.md` 中追加记录。使用**严格的格式**：

```markdown
## [YYYY-MM-DD] Ingest | <文件名>

- 创建: `[[页面名1]]`, `[[页面名2]]`, ...
- 更新: `[[页面名3]]` (source_count: N→M), `[[页面名4]]` (source_count: N→M), ...
- 修正: 经搜索验证，更新 xxx 说明（简述修正内容）
- 类型: <articles/papers/books/videos/podcasts/others>
- 来源: `raw/<类型>/<文件名>.md`
```

**日志规范：**
- 每个 raw 文件一条独立记录，以 `## [YYYY-MM-DD] Ingest | <文件名>` 为标题
- 标题中的 `<文件名>` 为实际处理的 raw 文件名（不含路径）
- 记录中引用 wiki 页面使用 `[[页面名]]` 格式
- 单个 raw 素材产生多个页面时，在一条记录中列出所有创建和更新的页面
- 若发生实质性内容修正（矛盾验证后更新），在 `修正:` 字段中简述
- 若只有内容补充无矛盾，只记录 `更新:` 和 source_count 变化
- 不要在单条记录中合并多个文件的信息
- **统一前缀格式**使得 log 可用简单 Unix 工具解析：`grep "^## \[" log.md | tail -5`

**批量处理 log 示例：**

```markdown
## [2026-05-10] Ingest | rust-setup-guide.md

- 创建: `[[Rust 安装与开发环境配置]]`, `[[rustup]]`, `[[rustc]]`, `[[Cargo]]`, `[[RsProxy 镜像源配置]]`
- 类型: articles
```

### 步骤 8：更新 qmd 索引

执行 qmd 索引更新：

```bash
qmd update -c knowledge-<domain>
```

其中 `<domain>` 为领域目录名的小写形式（如 `knowledge-rust`、`knowledge-ai`）。

如果该领域尚未创建 qmd collection，先执行：

```bash
qmd collection add ./wiki/ --name knowledge-<domain> --mask "**/*.md"
qmd context add qmd://knowledge-<domain>/ "<领域描述>"
qmd update -c knowledge-<domain>
```

领域描述从该领域的 `CLAUDE.md` 中提取。

## 完成检查清单

每轮 ingest 完成后，逐项验证：

- [ ] 所有新 wiki 页面都有正确的 frontmatter（title, date, tags, source_count）
- [ ] 更新的 wiki 页面 source_count 已正确递增
- [ ] frontmatter 中没有额外字段（如 source、author 等）
- [ ] 页面分类符合"页面分类决策表"
- [ ] 同一素材产生的多个页面之间互相引用
- [ ] 新/更新页面与现有页面有合理的交叉引用
- [ ] wiki/index.md 按分类正确组织，使用 `[[页面名]]` 格式
- [ ] **顶层 index.md 已更新页数**（仓库根目录的 `index.md`，非子目录）
- [ ] wiki/log.md 中每个 raw 文件有独立的 Ingest 记录，包含创建/更新/修正信息
- [ ] 所有已处理的 raw 文件已移动到 archive/
- [ ] qmd update 已执行

## 关键原则

- **绝不修改 `notes/` 目录**
- **`raw/` 内容只读**，处理完后移动到 `archive/`
- 素材不跨领域，在哪个领域 `raw/` 下就属于哪个领域
- 交互式提炼以用户意图为导向，自动处理以内容质量为导向
- 每次 ingest 后必须更新 qmd 索引
- **主动拆解**：内容丰富的素材应拆分为多个页面，而非压缩成单一 summaries
- **优先聚合**：同级细粒度概念优先聚合为上层主题页，避免 wiki 碎片化
- **素材不等于事实**：raw 素材中的主张需经交叉验证后方可写入 wiki。与新素材矛盾时优先联网搜索；无矛盾时，对关键事实、版本信息、数据也应做快速验证
- **分类隔离**：同一主题跨越不同分类时，优先分页面而非合并（如 `entities/Cargo` vs `recipes/Cargo 安装`）

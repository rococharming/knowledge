# Ingest Skill

将 `raw/` 中的新素材编译进 `wiki/`，使其成为持久、相互链接的知识网络的一部分。

## 核心理念

Ingest 不是简单的"存文件"，而是**编译**：LLM 读取素材、提取结构化知识、整合到现有 wiki 中，可能同时创建或更新 10+ 个页面。素材只被读一次，但产生的知识页面会被反复查询和交叉引用。

## 触发条件

以下用户表达应触发本 skill：

- "ingest `raw/某文章.md`"
- "把这篇文章加入 wiki"
- "消化这个素材"
- "处理 raw 下的新文件"
- "自动 ingest 这 5 篇文章"
- "批量 ingest"
- "更新 wiki 关于 xxx 的内容"（如果涉及新素材）

## 执行流程

### 1. 解析指令与定位素材

从用户输入中提取：
- **素材路径**：相对路径（如 `AI/raw/文章.md`）或文件名
- **模式**：自动模式（用户明确说"自动"）或讨论模式（默认）
- **批量**：单素材还是多素材

**路径解析规则**：
- `raw/` 下的素材按来源类型分子目录存放（`articles/`、`papers/`、`books/`、`videos/`、`podcasts/`、`others/`）
- 如果用户提供的是文件名，在 `raw/` 及其子目录中搜索匹配文件（排除 `archive/`）
- 如果提供的是相对路径（如 `AI/raw/articles/文章.md`），直接使用
- 如果路径指向目录而非文件，列出目录中的文件让用户选择
- 已归档的素材在 `raw/archive/`，不应被重复 ingest

### 2. 读取素材

使用 `obsidian-markdown` skill 或通用文件工具读取素材内容。

**素材类型处理**：
- **纯文本/Markdown**：直接读取
- **含图片的 Markdown**：先读文本，再单独查看引用的图片获取额外上下文
- **PDF/其他格式**：提取文本内容（如需要转换，告知用户）

**标签参考**：读取素材前，先查看 `<领域>/CLAUDE.md` 中的标签体系，作为 ingest 时为页面打标签的参考基准。

### 3. 可选讨论（默认启用）

如果用户没有明确说"自动"或"批量"，展示素材的关键要点供用户确认：

```
我读完这篇素材了，以下是我提取的关键要点：

1. ...
2. ...
3. ...

你希望我：
- 强调某个特定角度？
- 补充什么背景？
- 直接开始写入 wiki？
```

用户确认后进入下一步。如果用户说"自动"，跳过此步骤。

### 4. 分析与规划

先读取 `<领域>/CLAUDE.md`，了解该领域的分类体系和标签体系。然后确定需要创建/更新的 wiki 页面：

1. **摘要页面**：素材本身的摘要（几乎总是创建），放在领域 CLAUDE.md 规定的合适分类子目录下
2. **实体页面**：素材中提到的具体实体（人、产品、框架、公司等），放入领域 CLAUDE.md 规定的实体类目录
3. **概念页面**：素材中涉及的概念、方法论、理论，放入领域 CLAUDE.md 规定的概念类目录
4. **其他分类页面**：根据领域 CLAUDE.md 中的分类体系，识别是否需要创建其他类型的页面
5. **现有页面更新**：素材中提及的已有页面是否需要更新

**避免过度创建**：不是每个提到的人名都需要实体页面。判断标准：该实体是否在知识库的主题范围内、是否会在未来被反复引用。

**标签规划**：根据领域 CLAUDE.md 中的标签体系，为即将创建的页面预选标签。如果素材涉及新的子主题，可以创建新标签。

### 5. 写入 wiki

#### 5.1 创建 Summary 页面

在领域 CLAUDE.md 规定的摘要类分类子目录下创建：

```markdown
---
title: "素材标题"
date: 2026-05-09
source_count: 1
tags: [tag1, tag2]
---

# 素材标题

来源：[[原始文件名]]（链接到 raw/ 中的素材）

## 核心观点

...

## 关键要点

...

## 相关页面

- [[相关实体 1]]
- [[相关概念 2]]
```

#### 5.2 创建/更新 Entity 页面

每个新实体在领域 CLAUDE.md 规定的实体类目录下创建独立页面：

```markdown
---
title: "实体名称"
date: 2026-05-09
source_count: 1
tags: [entity, 领域标签]
---

# 实体名称

## 定义

...

## 关键信息

...

## 相关素材

- [[某篇文章摘要]]
```

如果实体页面已存在，在页面末尾追加新的素材引用，并更新 `source_count`。

#### 5.3 创建/更新 Concept 页面

每个新概念在领域 CLAUDE.md 规定的概念类目录下创建独立页面。与 Entity 类似，但侧重于概念而非具体实体。

#### 5.4 更新现有页面

检查素材中提及的已有页面：
- 是否有新信息需要补充？
- 是否有矛盾需要标记？
- 是否需要添加反向链接？

**标记矛盾时使用**：
```markdown
> 注意：新素材 [来源] 提出不同观点... 
```

### 6. 更新 index.md

在对应领域的 `wiki/index.md` 中：

1. 添加新创建的页面条目（链接 + 一行摘要）
2. 更新现有页面的 `source_count`（如有变化）
3. 按分类组织——分类列表参考该领域 `CLAUDE.md` 中的分类体系，不固定为某个标准集合

**index.md 格式**：
```markdown
## <分类名1>

- [[页面名称]] — 一句话摘要（日期，来源数）

## <分类名2>

- [[页面名称]] — 一句话定义（日期，来源数）
```

（具体分类名和顺序遵循该领域 CLAUDE.md 的分类体系）

**链接规范**：
- index.md 中的链接使用**纯文件名**，不带路径前缀。例如文件在 `wiki/某分类/文章.md`，链接写作 `[[文章]]` 而非 `[[某分类/文章]]`
- Obsidian 会自动解析同名文件，只要各页面文件名唯一即可
- 如果不同子目录下出现同名文件，优先保证文件名唯一（通过调整命名），而非使用路径前缀

### 7. 更新 log.md

在对应领域的 `wiki/log.md` 末尾追加：

```markdown
## [2026-05-09] ingest | 素材标题

- 创建/更新页面：Summary、Entity1、Concept2...
- 关键要点：...
```

### 8. 归档素材

将已 ingest 的素材从 `raw/` 的当前位置移动到 `raw/archive/`：

```
raw/articles/某文章.md  →  raw/archive/某文章.md
```

**归档规则**：
- 只移动已完整处理的素材文件本身
- 如果素材有同目录下的关联附件（图片、数据文件等），一并移动到 archive/
- 在 log.md 中追加归档记录：`- 归档：raw/articles/某文章.md → raw/archive/`

### 9. 报告结果

向用户报告完成了什么：

```
已完成 ingest `素材标题`：

- 新建页面：3 个（Summary、Entity: X、Concept: Y）
- 更新页面：2 个（现有 Entity A、现有 Concept B）
- 更新：index.md、log.md
- 归档：raw/articles/素材.md → raw/archive/

你可以通过 [[素材标题摘要]] 查看完整摘要。
```

## 批量 Ingest

当用户要求批量处理多个素材时：

1. **列出所有素材**：确认用户要处理的文件列表
2. **逐个处理**：对每个素材执行上述流程（步骤 2-8，包括归档）
3. **合并报告**：最后统一报告所有创建/更新的页面和归档的文件
4. **只更新一次 index.md 和 log.md**：所有素材处理完成后统一更新，避免重复写入

**批量模式自动跳过讨论环节**，除非用户在过程中明确要求查看某个素材的要点。

## 文件操作规范

- 使用 `obsidian-markdown` skill 创建/编辑 `.md` 文件
- 所有 wiki 页面必须包含标准 frontmatter
- 内部链接使用 `[[维基链接]]` 语法
- 附件统一使用 `![[文件名]]` 嵌入语法，存放于 `assets/`
- **绝不修改 `raw/` 中的素材内容**（仅允许在 ingest 完成后将素材移动到 `archive/` 归档）
- **绝不写入 `notes/` 目录**

## 约束与注意事项

1. **raw/ 是只读的**：只从中读取，绝不修改或删除
2. **不要重复创建**：写入前先检查同名页面是否已存在
3. **保持领域隔离**：不同领域的素材分别 ingest 到对应领域的 `wiki/` 中
4. **source_count 准确**：反映该 wiki 页面实际基于多少个 raw 素材
5. **讨论环节有价值**：即使是自动模式，如果素材特别重要或有歧义，主动询问用户
6. **关注矛盾**：新素材与现有 wiki 内容矛盾时，在页面中显式标记，不要默默覆盖

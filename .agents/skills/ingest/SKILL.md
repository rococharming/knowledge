# Ingest Skill

## 概述

将 `<domain>/raw/` 下的 Markdown 素材阅读、提取、整合到 `<domain>/wiki/`，并更新索引、日志和 qmd 搜索索引。

**核心职责边界**：ingest 只处理"新素材 vs 现有页面"的矛盾与补充。全站性的陈旧扫描、孤立页面检测、跨页面矛盾排查由 `lint` 负责。

## 触发条件（必须使用的场景）

- "ingest 这篇文章"
- "处理 raw/ 里的新素材"
- "把 xxx.md 整合到知识库"
- "提炼 raw/articles/ 下的内容"
- "归档 raw/ 中的文件"
- "更新 wiki 内容"
- "补充 [[某页面]] 的内容"
- 任何涉及将 raw 素材转化为 wiki 页面的请求
- 用户要求"处理"、"整理"、"提炼" raw/ 目录下文件的场景

## 输入判断

### 用户指定了具体文件

解析用户提供的文件路径，确认文件存在于 `<domain>/raw/`（或子目录）下。

**单个文件：**
1. 读取文件内容
2. 询问用户是否需要**交互式提炼**：
   - "这篇文章需要我逐轮和你确认提炼角度、分类、重点吗？"
   - 需要（yes/y/是/需要）→ 进入交互式模式
   - 不需要（no/n/否/不用）或未明确回复 → 进入自动处理模式
3. 交互式模式下，向用户展示素材的核心观点、结构和识别出的知识点清单，同时标注聚合建议和匹配结果，确认后再写入 wiki

**多个文件（2个及以上）：**
- 默认自动批量处理，不逐轮交互
- 处理完所有文件后，统一报告结果（创建/更新/归档清单）

### 用户未指定具体文件

1. 扫描 `<domain>/raw/`（不包括 `archive/`），列出所有 `.md` 文件
2. 询问用户要处理哪些文件
3. 根据用户选择的数量，按上述逻辑处理

## 核心处理流程

### 步骤 1：读取素材

读取用户指定的 raw Markdown 文件。注意：raw/ 内容只读，处理过程中绝不修改原文件。

### 步骤 2：分析拆解

**核心原则：按需拆分，而非强制合并；但同级细粒度概念应优先聚合为上层主题页。**

#### 2.1 识别知识点

通读全文，识别可独立成页的知识点：

| 类型 | 说明 | 示例 |
|---|---|---|
| **实体（entities）** | 工具、产品、框架、组织 | rustup、Cargo、VS Code |
| **概念（concepts）** | 方法论、技术概念、机制 | 所有权、生命周期、Edition |
| **操作指南（recipes）** | 配置步骤、安装流程 | 镜像源配置、IDE 环境搭建 |
| **摘要（summaries）** | 整篇文章的高层次摘要 | 某篇文章的系统梳理 |

#### 2.2 概念聚合判断

多个同级细粒度概念若**满足 2 项及以上**即应聚合为上层主题页：

| 标准 | 说明 | 示例 |
|---|---|---|
| **内容量不足** | 单个概念核心内容少于 3-4 个关键要点 | 整型只有类型表和字面量写法 |
| **检索价值有限** | 用户通常以更大主题查询 | 搜"Rust 数据类型"而非"Rust 整型" |
| **概念层级较低** | 属于某上层主题的同级细分 | 整型/浮点型/布尔型都属于"标量类型" |
| **来源连续性** | 同一素材中连续介绍的同类概念 | 一篇文章连续介绍所有基础类型 |

**聚合示例**：
- ✅ 连续介绍整型、浮点型、布尔型、字符型 → 聚合为 `concepts/数据类型`
- ✅ 连续介绍数组、元组、切片 → 聚合为 `concepts/复合类型` 或归入 `数据类型`
- ❌ 详细介绍所有权系统 → 保持独立 `concepts/所有权`
- ❌ 深入解释生命周期 → 保持独立 `concepts/生命周期`

聚合页面使用清晰的内部标题组织各子概念，每个子概念保留足够的细节。

#### 2.3 独立成页判断

经过聚合判断后，剩余知识点按以下标准决定是否独立成页：

- 有**实质性描述**（不仅是名字提及，有功能说明、原理解释或使用方法）
- 有**独立检索价值**（用户单独搜索时，页面能提供完整信息）
- **不重复**：内容不与其他页面重复

**决策输出**：列出计划创建的所有页面，包括分类、标题、内容范围。对于聚合页面，注明聚合了哪些子概念。

**分类决策**：根据内容特征决定页面分类：

| 素材内容特征 | 目标分类 |
|---|---|
| 单篇文章/论文/视频的整体摘要 | `summaries/` |
| 人、组织、产品、框架 | `entities/` |
| 方法论、技术、理论 | `concepts/` |
| 工具对比、方法对比 | `comparisons/` |
| 子领域全景梳理 | `overviews/` |
| 最佳实践、决策建议 | `syntheses/` |
| 具体问题的操作步骤 | `recipes/` |

同一篇 raw 素材可以同时满足多个条件，产生多个不同分类的页面。例如一篇安装指南可同时产生 summaries、entities 和 recipes。

#### 2.4 拆分 vs 不拆分的判断

- 文章同时介绍 rustup、rustc、Cargo → 拆分为 3 个 entities + 1 个 summaries
- 文章只简单提到"使用 cargo build" → 在 summaries 或对应实体页面中提及，不单独创建 Cargo 页面
- 文章详细介绍 Edition 和语义化版本 → 分别创建两个 concepts 页面
- 文章包含 RsProxy 镜像源配置步骤 → 创建 recipes 页面

### 步骤 3：匹配现有页面与矛盾验证

#### 3.1 匹配现有页面

1. 读取 `<domain>/wiki/index.md`，获取所有现有页面标题与分类
2. 标题预筛：
   - **强匹配**：标题相同或语义等价 → 进入矛盾验证
   - **弱匹配**：部分重叠、同义词、同一上层主题 → 进入矛盾验证，标记"可能聚合/并入"
   - **无匹配**：全新主题 → 直接创建
3. **qmd 辅助匹配**（可选）：若标题未直接匹配但概念较复杂，可用 `qmd search -c knowledge-<domain> "<关键词>"` 辅助发现相关页面

**分类隔离原则**：即使主题相同，不同分类的页面也应分别维护。如 `entities/Cargo` 与 `recipes/Cargo 安装配置` 是两个独立页面。

#### 3.2 矛盾验证与修正

**职责边界**：只处理"新素材 vs 现有页面"的矛盾。全站性的陈旧扫描、现有页面之间的相互矛盾由 `lint` 负责。

1. 读取现有页面内容
2. **主张比对**：提取新素材中的关键事实、数据、定义、版本信息，与现有页面逐条比对
3. **矛盾处理**（若发现不一致）：
   - **立即启用联网搜索**，验证双方主张（优先官方文档、权威来源、最新 release note）
   - **以验证结果为准，只写入正确版本**：新素材正确 → 覆盖；现有页面正确 → 保留；双方均不准 → 采用搜索验证后的最准确表述
   - **wiki 页面中不呈现矛盾过程、判断逻辑或"某来源认为"的并列句式**
4. **内容互补判断**（若无矛盾）：
   - 新素材覆盖方面与现有页面不重叠 → 作为新章节补充
   - 高度重叠（>70%）→ 仅增量更新（补充细节、更新 source_count）
   - 新素材是现有页面的子主题，且内容量 ≥ 3-4 个核心要点、有独立检索价值 → 创建子页面，在现有页面添加 `[[子页面]]` 链接

### 步骤 4：提炼内容

**交互式模式（单个文件 + 用户选择交互）：**
1. 向用户展示素材的核心观点、结构和知识点清单，标注聚合建议和匹配结果
2. 询问：知识点确认、聚合确认、匹配确认、提炼角度、重点标注、关联页面
3. 根据反馈生成 wiki 页面草稿，展示给用户确认后再写入

**自动模式（单个文件用户不交互，或多个文件批量）：**
1. 自主分析素材内容，提取关键信息
2. 自主识别知识点、决定分类、匹配现有页面、验证矛盾
3. 自主决定关联页面，直接生成所有 wiki 页面并写入

### 步骤 5：创建/更新 wiki 页面

创建或更新 `<domain>/wiki/` 下的页面。

**frontmatter 格式（必须遵守）：**

```markdown
---
title: 页面标题
date: YYYY-MM-DD
tags: [tag1, tag2]
source_count: N
---
```

- `title`：页面标题，简洁明了
- `date`：创建或最后更新日期
- `tags`：标签数组，使用英文小写，遵循各领域 `CLAUDE.md` 中的标签体系
- `source_count`：整数，该页面基于多少个 raw 素材（新建为 1，更新则递增）
- **禁止**添加任何其他 frontmatter 字段（如 `source`、`author`、`url` 等）

**页面内容规范：**
- 内容必须提炼而非照搬原文，突出结构化和可检索性
- 使用 `[[维基链接]]` 语法创建交叉引用
- wiki 页面中的 `[[维基链接]]` **必须仅指向同一领域 `wiki/` 目录内的页面**，禁止链接到 `raw/`、`notes/` 或其他目录中的文件
- 新页面应至少引用 1-2 个现有页面
- **同一素材产生的多个页面之间应互相引用**，形成知识网络
- **素材不等于事实**：raw 素材中的主张需经交叉验证后方可写入 wiki。关键事实、版本信息、数据应做快速验证
- **图片处理**：raw 素材中的图片已统一存放于仓库顶层 `assets/` 目录。提炼时原样保留与知识点直接相关的图片（`![[xxx.png]]`），不要修改文件名、不要添加路径前缀；纯装饰性图片可丢弃；图片位置贴合语义

**来源章节（必须）：**

每个 wiki 页面底部必须包含 `## 来源` 章节，列出该页面所基于的原始材料，使用 Obsidian wiki 链接格式：

```markdown
## 来源

- [[<文件名>]]
```

**规范：**
- 放在页面最底部，在所有内容之后
- 使用 `[[<文件名>]]` 格式的 Obsidian 链接（**不含路径前缀和 `.md` 后缀**，Obsidian 会全局解析）
- 新建页面时，列出当前 ingest 的 raw 文件名（不含路径和扩展名）
- 更新已有页面时，读取现有来源章节，在列表末尾追加新来源（不重复已有的）
- 示例：`[[Rust基础语法]]`、`[[HTTP基础概述]]`

**命名冲突规避**：wiki 页面文件名（不含路径）不得与原始素材文件名相同。若素材标题恰好是某个核心概念名称（如 `所有权.md`），wiki 页面应使用更精确的命名（如 `所有权系统.md`、`所有权规则.md`），避免 `[[<文件名>]]` 来源链接在 Obsidian 中产生歧义、无法跳转。

### 步骤 6：更新领域索引（wiki/index.md）

更新 `<domain>/wiki/index.md`：

```markdown
---
title: <Domain> Wiki 索引
date: YYYY-MM-DD
---

# <Domain> Wiki 索引

## summaries

- [[页面名]] — 一句话摘要

## entities

- [[页面名]] — 一句话摘要

## concepts

- [[页面名]] — 一句话摘要

## comparisons

## overviews

## syntheses

## recipes
```

- 按分类组织，每个分类下只列出属于该分类的页面
- 使用 `[[页面名]]` 格式的 wiki 链接（**不要**使用 `[[路径/文件名.md|页面名]]` 格式）
- 每个链接后附 1-2 句话的摘要
- 空分类保留标题
- 不要重复列出同一个页面

### 步骤 7：更新顶层索引（index.md）

**重要：顶层 index.md 位于仓库根目录，文件路径是 `index.md`。**

读取顶层 `index.md`，统计当前领域 `wiki/` 下所有 `.md` 页面的数量（**排除** `wiki/index.md` 和 `wiki/log.md`），更新该领域条目中的页数标注：

```markdown
- [[<domain>/wiki/index|<domain>]] — <领域概述>（<N> 页）
```

若该领域不在顶层 index.md 中，添加新条目并标注页数。

### 步骤 8：更新日志（wiki/log.md）

在 `<domain>/wiki/log.md` **尾部追加**记录。使用 `Edit` 工具在文件末尾插入新记录，**禁止**重写整个文件。

**追加方法**：读取 `log.md` 文件末尾的 2-3 行内容作为 `old_string`，确保匹配的是文件真正的末尾。`new_string` 为原有末尾内容后紧跟新日志记录：

```markdown
## [YYYY-MM-DD] Ingest | <文件名>

- 创建: `[[页面名1]]`, `[[页面名2]]`, ...
- 更新: `[[页面名3]]` (source_count: N→M), `[[页面名4]]` (source_count: N→M), ...
- 修正: 经搜索验证，更新 xxx 说明（简述修正内容）
- 类型: <articles/papers/books/videos/podcasts/others>
- 来源: `raw/<类型>/<文件名>.md`
```

**日志规范：**
- 每个 raw 文件一条独立记录，以 `## [YYYY-MM-DD] Ingest | <文件名>` 为标题
- 标题中的 `<文件名>` 为实际处理的 raw 文件名（不含路径）
- 记录中引用 wiki 页面使用 `[[页面名]]` 格式
- 单个 raw 素材产生多个页面时，在一条记录中列出所有创建和更新的页面
- 若发生实质性内容修正（矛盾验证后更新），在 `修正:` 字段中简述
- 若只有内容补充无矛盾，只记录 `更新:` 和 source_count 变化
- 不要在单条记录中合并多个文件的信息
- **统一前缀格式**使得 log 可用简单 Unix 工具解析：`grep "^## \[" log.md | tail -5`

### 步骤 9：归档 raw 文件

将已处理的文件移动到 `<domain>/raw/archive/<类型>/`：
- `raw/articles/foo.md` → `raw/archive/articles/foo.md`
- 如果 `archive/<类型>/` 目录不存在，先创建

### 步骤 10：更新 qmd 索引

执行 qmd 索引更新：

```bash
qmd update -c knowledge-<domain>
```

其中 `<domain>` 为领域目录名的小写形式。

如果该领域尚未创建 qmd collection，先执行：

```bash
qmd collection add ./wiki/ --name knowledge-<domain> --mask "**/*.md"
qmd context add qmd://knowledge-<domain>/ "<领域描述>"
qmd update -c knowledge-<domain>
```

领域描述从该领域的 `CLAUDE.md` 中提取。

## 完成检查清单

每轮 ingest 完成后，逐项验证：

- [ ] 所有新 wiki 页面都有正确的 frontmatter（title, date, tags, source_count）
- [ ] 更新的 wiki 页面 source_count 已正确递增
- [ ] frontmatter 中没有额外字段（如 source、author 等）
- [ ] 页面分类符合分类决策表
- [ ] 同一素材产生的多个页面之间互相引用
- [ ] 新/更新页面与现有页面有合理的交叉引用
- [ ] 每个 wiki 页面底部都有 `## 来源` 章节，使用 Obsidian 链接格式 `[[<文件名>]]`
- [ ] 更新页面时，来源章节已追加新素材（不重复已有条目）
- [ ] wiki/index.md 按分类正确组织，使用 `[[页面名]]` 格式
- [ ] **顶层 index.md 已更新页数**（仓库根目录的 `index.md`）
- [ ] wiki/log.md 中每个 raw 文件有独立的 Ingest 记录，包含创建/更新/修正信息
- [ ] 所有已处理的 raw 文件已移动到 archive/
- [ ] qmd update 已执行

## 关键原则

- **绝不修改 `notes/` 目录**
- **`raw/` 内容只读**，处理完后移动到 `archive/`
- 素材不跨领域，在哪个领域 `raw/` 下就属于哪个领域
- 交互式提炼以用户意图为导向，自动处理以内容质量为导向
- 每次 ingest 后必须更新 qmd 索引
- **主动拆解**：内容丰富的素材应拆分为多个页面
- **优先聚合**：同级细粒度概念优先聚合为上层主题页，避免 wiki 碎片化
- **素材不等于事实**：raw 素材中的主张需经交叉验证后方可写入 wiki
- **分类隔离**：同一主题跨越不同分类时，优先分页面而非合并
- **避免命名冲突**：wiki 页面文件名（不含 `.md`）不得与归档的 raw 素材文件名相同。若核心概念名称与素材文件名重合，wiki 页面应使用更精确的命名（如 `所有权系统.md`），否则 Obsidian 的 `[[...]]` 来源链接会产生歧义

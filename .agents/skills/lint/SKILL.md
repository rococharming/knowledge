# Lint — 知识库健康检查

对知识库 `wiki/` 进行系统性健康检查，检测结构缺陷和内容质量问题。
简单问题自动修复，复杂问题生成报告供用户决策。

## 触发条件（必须使用的场景）

- "lint 一下知识库"
- "检查 wiki 有没有问题"
- "知识库健康检查"
- "清理死链"
- "维护 wiki"
- "知识库体检"
- "排查 wiki 问题"
- "整理 wiki 索引"
- 任何涉及检查 wiki 完整性、链接有效性、索引同步的请求

## 检查范围

默认扫描**整个知识库所有领域**。

用户可指定单个或多个领域：
- "lint Rust" → 只检查 Rust 领域
- "lint Rust AI" → 检查 Rust 和 AI 两个领域
- "lint 全库" → 明确全库扫描（与默认行为一致）

## 检查项

### 结构检查（优先执行）

#### 1. 死链（Broken Links）

**检测逻辑**：扫描所有 wiki 页面中的 `[[链接名]]`，检查目标页面是否存在于对应领域的 `wiki/` 目录中。

**修复策略**：仅报告，不自动修复。
- 死链可能因页面重命名、删除或拼写错误导致
- 修复需要人工判断：是修正链接、创建目标页面，还是移除链接

**报告格式**：
```markdown
### 死链

| 来源页面 | 死链 | 建议 |
|---------|------|------|
| [[页面A]] | `[[不存在的页面]]` | 页面已删除，建议移除链接或创建页面 |
```

#### 2. 孤立页面（Orphan Pages）

**检测逻辑**：扫描 `wiki/` 中所有 `.md` 文件，找出未被任何其他页面引用、也未在 `wiki/index.md` 中列出的页面。

**修复策略**：**自动修复**。
- 将孤立页面添加到 `wiki/index.md` 的合适分类下
- 分类推断优先级：
  1. 页面文件路径中的目录名（如 `wiki/concepts/foo.md` → concepts）
  2. 页面 frontmatter 中的 `tags` 或内容主题
  3. 如果无法推断，归入最通用的分类并标注"待分类"

#### 3. 索引同步问题

**检测逻辑**：对比 `wiki/index.md` 中列出的页面与 `wiki/` 目录中实际存在的页面。

**三类问题**：
- **index.md 中有，但文件不存在** → 文件已删除但索引未更新
- **index.md 中无，但文件存在** → 新页面未加入索引（通常已被"孤立页面"检查覆盖）
- **index.md 中的分类与实际不符** → 页面移动了但索引未更新

**修复策略**：**自动修复前两类**。
- 移除 index.md 中已不存在的页面链接
- 添加遗漏的页面到对应分类
- 分类不符的仅报告，由用户确认后修复

#### 4. 顶层索引计数偏差

**检测逻辑**：对比顶层 `index.md` 中标注的各领域 wiki 页面数与实际文件数。

**修复策略**：**自动修复**。
- 更新顶层 `index.md` 中的页面计数

### 内容质量检查

#### 5. 陈旧内容（Stale Content）

**检测逻辑**：读取每个 wiki 页面的 frontmatter `date` 字段，计算距今天数。

**阈值**：默认 **180 天**（半年）。用户可在 lint 时指定其他阈值：
- "lint，陈旧阈值 90 天"
- "lint，只标超过一年的"

**修复策略**：仅报告，不自动修复。
- 不修改 `date` 字段（掩盖真实新旧程度）
- 不自动更新内容（可能引入错误）

**报告格式**：
```markdown
### 陈旧内容（建议回顾）

| 页面 | 最后更新 | 距今 | 建议 |
|------|---------|------|------|
| [[Rust 所有权]] | 2025-11-10 | 180 天 | 内容可能随版本迭代而更新 |
```

#### 6. 数据空白（Data Gaps）

**检测逻辑**：
- frontmatter 缺失或字段不完整（缺少 `title`、`date`、`tags` 等）
- `source_count` 为 0 或缺失（summaries 类型页面应有 source_count ≥ 1）
- 页面内容过短（少于 100 字，可能是未完成的草稿）

**修复策略**：仅报告。
- 标注缺失的字段和具体页面
- 建议用户补充

#### 7. 矛盾主张（Contradictory Claims）

**检测逻辑**：读取相关页面内容，识别同一概念在不同页面中的矛盾描述。

**适用场景**：
- 小规模知识库（全库 wiki 页面 ≤ 300）或指定单个领域时，可执行此检查
- 大规模全库 lint 时，此检查可能消耗过多 token，建议用户明确指定"深度检查"才执行

**修复策略**：仅报告。
- 列出矛盾的页面和具体主张
- 由用户判断哪一方更准确

#### 8. 缺失交叉引用（Missing Cross-References）

**检测逻辑**：检查页面中提到的概念是否已创建为 wiki 页面但未链接。

**适用场景**：同"矛盾主张"，小规模或指定领域时才执行。

**修复策略**：仅报告。
- 列出页面中提到的未链接概念
- 建议添加 `[[链接]]` 或创建新页面

#### 9. 重要概念缺页（Missing Important Concepts）

**检测逻辑**：
- 检查 `wiki/index.md` 的 overviews 或概念列表中提到的概念是否有独立页面
- 检查多个页面频繁提及但未创建独立页面的概念

**适用场景**：同"矛盾主张"，小规模或指定领域时才执行。

**修复策略**：仅报告。
- 列出建议创建页面的概念
- 由用户决定优先级

## 执行流程

### 步骤 1：确定范围

1. 判断用户是否指定了领域
   - 未指定 → 全库扫描（读取顶层 `index.md` 获取所有领域列表）
   - 指定了 → 只检查指定领域
2. 记录陈旧内容阈值（默认 180 天）

### 步骤 2：收集信息

对每个检查领域，并行执行：
1. 读取 `wiki/index.md`
2. 扫描 `wiki/` 目录中所有 `.md` 文件（获取文件列表和 frontmatter）
3. 如果领域规模小（wiki 页面 ≤ 50），读取所有页面内容用于深度检查

### 步骤 3：执行结构检查

按顺序执行：
1. 死链扫描
2. 孤立页面检测
3. 索引同步检查
4. 顶层索引计数检查

### 步骤 4：执行内容质量检查

1. 陈旧内容检测（基于 frontmatter date）
2. 数据空白检测（基于 frontmatter 完整性）
3. 如果规模允许，执行深度检查（矛盾主张、缺失交叉引用、重要概念缺页）

### 步骤 5：自动修复

对标记为"自动修复"的问题执行修复：
1. 更新 `wiki/index.md`（添加孤立页面、移除已删除页面）
2. 更新顶层 `index.md` 页面计数

### 步骤 6：生成报告

报告文件：`lint-report-YYYYMMDD.md`（保存到知识库根目录或输出到对话）

报告结构：
```markdown
# 知识库健康检查报告 — YYYY-MM-DD

## 执行摘要

- 检查领域: <领域列表>
- 检查页面数: <N>
- 发现问题: <N> 个
- 自动修复: <N> 个
- 需人工处理: <N> 个

## 自动修复记录

- 将 [[孤立页面]] 添加到 index.md 的 <分类>
- 从 index.md 移除已删除页面 [[已删除页面]]
- 更新顶层 index.md 页面计数: Rust 350 → 352

## 需人工处理的问题

### 死链
...

### 陈旧内容
...

### 数据空白
...

### 矛盾主张（如执行了深度检查）
...

## 建议

1. ...
2. ...
```

### 步骤 7：更新日志

在每个被检查领域的 `wiki/log.md` 中追加记录：

```markdown
## YYYY-MM-DD

- **Lint**: 健康检查
  - 范围: <全库 / 本领域>
  - 问题: <N> 个
  - 自动修复: <N> 个
  - 报告: `lint-report-YYYYMMDD.md`
```

## 严重程度分级

| 级别 | 问题类型 | 说明 |
|-----|---------|------|
| 🔴 严重 | 死链、索引严重不同步 | 影响导航和可用性 |
| 🟡 警告 | 孤立页面、陈旧内容、数据空白 | 影响质量但可用 |
| 🟢 建议 | 缺失交叉引用、重要概念缺页 | 改进空间 |

## 规模适配

| 规模 | 页面数 | 执行策略 |
|-----|-------|---------|
| 小型 | ≤ 50 | 执行全部检查（含深度检查） |
| 中型 | 51-300 | 执行结构检查 + 基础内容检查，深度检查按需 |
| 大型 | > 300 | 优先结构检查，深度检查需用户明确要求 |

## 完成检查清单

每轮 lint 完成后，逐项验证：

- [ ] 所有孤立页面已添加到 `wiki/index.md`
- [ ] `wiki/index.md` 中已不存在的页面链接已移除
- [ ] 顶层 `index.md` 页面计数已更新
- [ ] 报告已生成，包含所有发现的问题
- [ ] 各领域的 `wiki/log.md` 已追加 Lint 记录
- [ ] 自动修复的内容已正确写入，未破坏文件格式

## 关键原则

- **绝不修改 `notes/` 目录**
- **绝不修改 `raw/` 目录中的内容**
- **绝不修改 wiki 页面的核心内容**（自动修复仅限索引层面）
- 陈旧内容只标记不篡改 date
- 死链只报告不猜测修复（不知道目标页面被删除还是重命名）
- 大规模 lint 优先保证结构健康，深度检查需额外授权

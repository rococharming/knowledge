# 一、AI 编程的四大坑

`Karpathy`在社交平台上列出了一堆关于AI编程是的各种问题，主要总结如下：

- **默默做假设**：模型会替你做出错误的假设，然后盲目地继续执行，而不进行检查
- **过度设计**：模型喜欢过度复杂化的代码和API，膨胀抽象层，有时候100行代码能解决的情况下，它会实现超过1000行的结构
- **顺手"优化"**：有时候仍然会修改/删除与任务无关的代码作为副作用
- **不验证结果**：任务可能还没完全解决，AI就告诉你完成，也没跑测试，没检查是否引入了新的bug等

# 二、解决方案

`Forrest Chang`在看到这些吐槽后，为`Claude Code`写了一份行为准则：`CLAUDE.md`，来尽可能规范 AI 编程时出现的上述问题。

主要可以提炼为4条原则：

**原则1：编码前先思考**

核心：不要做假设。不要隐藏困惑，把权衡摆出来。

在写代码前，AI必须：

- 明确说出自己的假设。如果不确定，就问。
- 如果有多种理解方式，列出来让你选择
- 如果有更简单的方式，说出来
- 如果有什么不清楚的，停下来，说明白哪里困惑

**原则2：简单优先**

核心：能解决问题的最少代码。不要投机性功能。

规则如下：

- 不要加没被要求的功能
- 单次使用的代码不要搞抽象
- 不要加没被要求的"灵活性"
- 不要为不可能的场景写错误处理
- 如果 200 行能缩写成 50 行，就重写

测试标准：资深工程师会不会说这太复杂了，如果会，就简化。

**原则3：外科手术式修改**

核心：只改必须改的。只清理自己弄乱的。

编辑现有代码时：

- 不要"改进"相邻的代码或格式
- 不要重构没坏的东西
- 匹配现有风格，即使你不喜欢
- 如果发现死代码，提一句——但别删

测试标准： 每一行改动都应该能直接追溯到用户的要求。

**原则4：目标驱动执行**

核心：定义成功标准。循环直到验证通过。

这是`Karpathy`认为最高杠杆的洞察：**把命令式任务转化为声明式目标**。

例如，当需要修复Bug时，不要说"帮我修复这个Bug"，而是说："帮我编写复现该Bug的测试，然后通过"；再例如，重构代码时，不要说"重构 X"，而是说"确保重构前后测试都能通过"。

> AI 编写代码必须加个验证，目标驱动方法。

# 三、完整的CLAUDE.md


````markdown
# CLAUDE.md

用于减少 LLM 在编码任务中常见错误的行为准则。

**权衡：** 这些准则更偏向谨慎，而不是速度。对于非常简单的任务，可以自行判断是否需要完整执行。

## 1. 编码前先思考

**不要假设。不要隐藏困惑。要明确指出权衡。**

在实现之前：

- 明确说明你的假设。如果不确定，就提问。
- 如果存在多种理解方式，要把它们列出来，不要默默选择其中一种。
- 如果有更简单的方案，要说出来。必要时要提出反对意见。
- 如果某件事不清楚，就停下来。指出哪里令人困惑，并提问。

## 2. 简单优先

**用最少的代码解决问题。不要做推测性的扩展。**

- 不要实现用户没有要求的功能。
- 不要为只使用一次的代码创建抽象。
- 不要添加用户没有要求的"灵活性"或"可配置性"。
- 不要为不可能发生的场景添加错误处理。
- 如果你写了 200 行代码，但其实 50 行就能解决，那就重写并简化。

问自己一句：**"资深工程师会不会觉得这太复杂了？"**  
如果答案是会，那就简化。

## 3. 精准修改

**只改必须改的地方。只清理你自己造成的问题。**

编辑现有代码时：

- 不要顺手"改进"相邻代码、注释或格式。
- 不要重构没有问题的代码。
- 保持与现有代码风格一致，即使你自己会用另一种写法。
- 如果发现无关的废弃代码，可以提出来，但不要删除。

当你的修改造成遗留问题时：

- 删除由你的修改导致不再使用的 import、变量或函数。
- 不要删除修改前就已经存在的废弃代码，除非用户明确要求。

检验标准：**每一行改动都应该能够直接对应到用户的请求。**

## 4. 目标驱动执行

**定义成功标准。持续迭代，直到验证通过。**

把任务转化为可验证的目标：

- "加个验证" → "先为非法输入写测试，再让测试通过"
- "修复 bug" → "先写一个能复现 bug 的测试，再让它通过"
- "重构 X" → "确保重构前后测试都通过"

对于多步骤任务，先给出简短计划：

```
1. [步骤] → 验证：[检查方式]
2. [步骤] → 验证：[检查方式]
3. [步骤] → 验证：[检查方式]
```

清晰的成功标准能让你独立循环推进。  
模糊的标准，例如"让它能用"，则需要不断向用户确认。

---

**如果这些准则有效，应该会看到：**  
diff 中不必要的改动更少，因为过度复杂而导致的重写更少，并且澄清问题会发生在实现之前，而不是出错之后。


````


# 四、使用该准则的方式

## 1、安装插件

（1）增加插件市场

```shell
/plugin marketplace add forrestchang/andrej-karpathy-skills
```

（2）安装插件

```
/plugin install andrej-karpathy-skills@karpathy-skills
```

## 2、单个项目的CLAUDE.md

- 如果是新项目：将`CLAUDE.md`复制粘贴到该项目
- 如果是已有项目：将`CLAUDE.md`内容追加到当前项目的`CLAUDE.md`

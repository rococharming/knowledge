
# 一、简介

`skill-creator`是一个用于创建、改进、测试和优化`Skill`的**元技能**。
 
它**不是用来解决某一单一业务问题**的普通 Skill，而是用来帮助用户生产 Skill、打磨 Skill、测试 Skill、优化 Skill 触发效果，并最终打包 Skill 的完整工作流 Skill。

因此，它的核心作用是帮助用户把某个固定工作流、方法论、工具使用方式或专业任务流程，整理成一个可以被反复调用的`Skill`。它可以帮助你完成：

- 从零创建一个新的 Skill
- 修改和改进已有 Skill
- 为 Skill 编写测试用例
- 运行测试并对比 Skill 使用前后的效果
- 根据用户反馈持续迭代 Skill
- 优化 Skill 的 `description`，提高触发准确率
- 最后将 Skill 打包成可安装文件

因此，它的工作流程可以概括为：

```text
┌─────────────────────────────────────────────────────┐  
│  1. 确定技能目标（做什么、何时触发、输出格式）             │  
└──────────────────┬──────────────────────────────────┘  
                   ▼  
┌─────────────────────────────────────────────────────┐  
│  2. 撰写 SKILL.md 初稿                               │  
└──────────────────┬──────────────────────────────────┘  
                   ▼  
┌─────────────────────────────────────────────────────┐  
│  3. 创建测试用例（2-3 个真实场景）                      │  
└──────────────────┬──────────────────────────────────┘  
                   ▼  
┌─────────────────────────────────────────────────────┐  
│  4. 并行运行测试（with-skill vs baseline）             │  
└──────────────────┬──────────────────────────────────┘  
                   ▼  
┌─────────────────────────────────────────────────────┐  
│  5. 用户审查结果（浏览器查看器 + 定量指标）               │  
└──────────────────┬──────────────────────────────────┘  
                   ▼  
┌─────────────────────────────────────────────────────┐  
│  6. 根据反馈改进技能                                   │  
└──────────────────┬──────────────────────────────────┘  
                   │  
                   └──────→ 不满意则回到第 4 步 ──────┐  
                                                   │

                   ▼  
┌─────────────────────────────────────────────────────┐  
│  7. 优化描述（可选，提高触发准确度）                     │  
└──────────────────┬──────────────────────────────────┘  
                   ▼  
┌─────────────────────────────────────────────────────┐  
│  8. 打包并交付 skill                                  │  
└─────────────────────────────────────────────────────┘

---
```



# 二、什么时候应该使用 skill-creator

当你有下面这些需求时，就可以使用 `skill-creator`。

## 1、从零创建一个新 skill

例如：

> 我想创建一个帮我写 Git 提交信息的 skill。  
> 我想创建一个能帮我整理 Obsidian 笔记的 skill。  
> 我想创建一个根据项目文档生成执行计划的 skill。  
> 我想创建一个专门处理 PDF 表格提取的 skill。

这种情况下，`skill-creator`会先帮你明确 skill 的目标、触发条件、输出格式，然后再生成 `SKILL.md`。

## 2、改进已有 skill

例如：

> 我这个 skill 有时候输出太泛泛。  
> 我的 PDF 处理 skill 经常漏掉表格。  
> 我的写作 skill 风格不稳定。  
> 这个 skill 触发了，但执行结果不符合预期。

这种情况下，`skill-creator` 会帮助你分析现有 skill 的问题，并通过测试和迭代改进它。


## 3、优化 skill 的触发准确度

例如：

> 我的 skill 有时候该触发时没触发。  
> 我的 skill 有时候不该触发却触发了。  
> 我想让 description 写得更准确。  
> 我想测试 Claude 到底什么时候会调用这个 skill。

这种情况下，`skill-creator` 会重点优化 `SKILL.md` 里的 `description` 字段。


## 4、给 skill 写测试用例

例如：

> 帮我为这个 skill 设计 eval.json
> 帮我写几个测试  case
> 帮我设计 assertions
> 帮我判断这个 skill 是否真的比 baseline 好

这种情况下，`skill-creator` 会帮你把“感觉好不好”变成可测试、可比较的评估流程。


## 5、打包 skill

例如：

> 帮我把这个 skill 打包成 .skill 文件
> 帮我检查这个 skill 是否符合规范
> 帮我准备发布 这个 skill

这种情况下，`skill-creator` 会在打包前检查 skill 结构，然后生成 `.skill` 文件。




# 三、使用skill-creator的基本方式

你可以直接用自然语言调用它，不需要记复杂命令。

常见说法包括：

> 使用 skill-creator，帮我创建一个用于整理会议纪要的 skill。

> 帮我用 skill-creator 改进这个现有 skill。

> 我想做一个 Claude Code skill，用来根据代码 diff 生成 PR 描述。

> 帮我为这个 skill 写测试用例和 assertions。

> 帮我优化这个 skill 的 description，让它更容易在正确场景触发。



# 四、从零创建 skill 的推荐流程

如果你要从零开始创建一个 skill，推荐按下面的流程走。

## Step 1：说明你想创建什么 skill

你可以先简单描述目标，不需要一开始就想得很完整。

例如：

> 我想创建一个 skill，用来帮我把英文技术文档翻译成中文，要求保持原意、术语准确、中文自然。

或者：

> 我想做一个 skill，专门帮我把零散的项目想法整理成可执行计划。

`skill-creator` 会根据你的描述继续追问或直接整理需求。


## Step 2：明确 skill 的三个核心问题

创建 skill 前，最重要的是明确三个问题：

### 1. 这个 skill 做什么？

也就是 skill 的核心任务。

例如：

> 把英文技术文档翻译成清晰、准确、自然的中文。

### 2. 什么时候应该触发？

也就是用户说什么话时，Claude 应该自动使用这个 skill。

例如：

> 当用户要求翻译英文技术文档、开发文档、API 文档、论文摘要时触发。


### 3. 输出应该是什么样？

也就是结果格式。

例如：

> 直接输出中文译文，不额外解释；保留原文 Markdown 结构；代码块不翻译；术语保持一致。

这三个问题越清楚，写出来的 skill 越稳定。


## Step 3：生成 SKILL.md 初稿

明确目标后，`skill-creator` 会帮你生成 skill 的主文件：

```
SKILL.md
```

一个标准的 `SKILL.md` 通常包括两部分：

1. YAML 前置元数据
2. Markdown 指令正文

示例结构：

```
---name: technical-document-translatordescription: Use this skill when the user asks to translate English technical documentation into clear, accurate, natural Chinese while preserving the original meaning and Markdown structure.---# Technical Document Translator## Purpose...## Workflow...## Output Requirements...
```

其中最重要的是 `description`。

Claude 会根据 `description` 判断什么时候应该加载这个 skill。


## Step 4：创建测试用例

不要只写 skill，不测试。

一个可靠的 skill 至少应该有 2 到 3 个真实测试场景。

例如，一个“技术文档翻译 skill”可以有这些测试用例：

1. 翻译 API 文档
2. 翻译命令行工具说明
3. 翻译包含代码块的 Markdown 文档

每个测试用例应该尽量贴近你真实使用时的输入。


## Step 5：设计 assertions

`assertions` 是用来判断输出是否合格的客观标准。

好的 assertions 应该可以被明确判断通过或失败。

例如：

好的 assertions：

- 输出保留原始 Markdown 标题层级
- 代码块内容没有被翻译
- 专有术语翻译保持一致
- 输出中没有遗漏原文段落
- 中文表达自然、没有明显机器翻译腔

不好的 assertions：

- 输出很好
- 翻译不错
- 看起来专业
- 内容让人满意

原因是这些说法太主观，不适合自动评分。



## Step 6：运行对比测试

`skill-creator` 的一个核心价值是做对比测试。

它通常会比较两种结果：

| 类型         | 含义                                    |
| ---------- | ------------------------------------- |
| baseline   | 不使用 skill 或 使用旧的 skill 时 Claude 的普通输出 |
| with-skill | 使用 skill 后 Claude 的输出                 |

这样你可以判断：

- skill 是否真的改善了结果
- 改善体现在哪里
- 是否引入了新的问题
- 哪些测试用例仍然失败

## Step 7：审查结果并反馈

测试完成后，你需要查看结果。

重点看三件事：

1. 哪些 assertions 通过了
2. 哪些 assertions 失败了
3. with-skill 是否明显优于 baseline

你的反馈可以很具体，例如：

> 第 2 个测试用例里，代码块虽然保留了，但命令参数说明被翻译得不够准确。

或者：

> 输出格式很好，但术语表没有被严格执行。

这些反馈会成为下一轮改进的依据。



## Step 8：迭代改进 skill

根据测试结果，`skill-creator` 会帮你改进 `SKILL.md`。

常见改进包括：

- 补充边界情况
- 加强输出格式要求
- 添加反例
- 修改 workflow
- 明确不要做什么
- 优化术语处理规则
- 改写 description

改完后继续测试。

这个过程可以循环多轮，直到结果稳定。


## Step 9：优化 description

当 skill 的主体逻辑已经比较稳定后，可以进一步优化 `description`。

`description` 的目标不是“介绍 skill”，而是帮助 Claude 判断：

> 当前用户请求是否应该触发这个 skill？

好的 description 应该：

- 聚焦用户意图
- 使用明确的触发场景
- 不要写太泛
- 不要只描述内部实现
- 可以稍微主动一点，避免该触发时没触发

例如：

较弱的 description：

> Helps with translation.

较好的 description：

> Use this skill when the user asks to translate English technical documentation, API references, developer guides, CLI instructions, or Markdown-based engineering content into accurate and natural Chinese while preserving structure, code blocks, and terminology consistency.


## Step 10：打包 skill

当 skill 测试通过、效果稳定后，就可以打包。

最终会生成：

```
.skill
```

这个文件可以用于安装、分发或分享。

打包前通常会检查：

- 是否存在 `SKILL.md`
- YAML 元数据是否正确
- `name` 是否符合命名规范
- `description` 是否存在
- 文件结构是否合理
- 是否包含不必要的大文件

# 五、改进已有 skill 的使用方法

如果你已经有一个 skill，不需要从头开始。

你可以直接说：

> 帮我改进这个 skill。

然后提供：

1. 现有的 `SKILL.md`
2. 你遇到的问题
3. 一个或多个失败案例
4. 你期望的输出效果

例如：

> 这个 skill 用来生成 PR 描述，但它经常忽略 breaking changes。请帮我改进。

更好的输入是：

> 这是我的 SKILL.md。问题是：当 diff 里有数据库 schema 变更时，它没有在 PR 描述中提醒风险。我希望它以后必须增加“风险与影响”小节，并明确指出 migration 相关风险。

这样 `skill-creator` 能更准确地修改指令。

# 六、优化触发准确度的使用方法

如果你的问题不是 skill 输出不好，而是“什么时候触发”不准，那么重点应该优化 `description`。

你可以这样说：

> 帮我优化这个 skill 的 description。它现在有时候不该触发也触发。

或者：

> 这个 skill 该在用户请求整理项目计划时触发，但普通闲聊时不要触发。

优化时，`skill-creator` 通常会关注两类情况：

|类型|含义|
|---|---|
|应该触发|用户请求确实属于这个 skill 的范围|
|不应该触发|用户请求看起来相关，但其实不需要用这个 skill|

例如，对于“项目计划 skill”：

应该触发：

> 帮我把这个项目想法整理成可执行计划。

不应该触发：

> 项目管理是什么意思？

前者是执行任务，后者只是概念解释。


# 七、如何写出好的 skill 需求

为了让 `skill-creator` 更好地帮你，建议你描述需求时包含这些信息。

## 1. 任务目标

说明这个 skill 要完成什么任务。

例如：

> 用来把零散想法整理成项目执行计划。

---

## 2. 触发场景

说明用户说什么时应该使用它。

例如：

> 当用户要求制定计划、拆解任务、形成路线图、整理执行步骤时触发。

---

## 3. 不触发场景

说明哪些类似请求不应该使用它。

例如：

> 如果用户只是问某个概念是什么意思，不需要触发。

---

## 4. 输入类型

说明用户通常会提供什么材料。

例如：

- 一段想法
- 一份文档
- 一个代码 diff
- 一个会议记录
- 一个需求描述
- 一组文件

---

## 5. 输出格式

说明结果应该长什么样。

例如：

- Markdown 文档
- 分步骤计划
- 表格
- JSON
- checklist
- `.docx` 文件
- 可执行命令

---

## 6. 质量标准

说明什么叫“做得好”。

例如：

- 结构清晰
- 不遗漏关键点
- 保留原始格式
- 输出可以直接执行
- 不编造不存在的信息
- 对不确定内容明确标注

# 九、写 assertions 的原则

`assertions` 是 skill 评估中非常重要的一环。

它们决定了你能否客观判断 skill 是否变好了。

## 好的 assertions

好的 assertions 通常有这些特点：

- 可以客观验证
- 名称清晰
- 通过或失败容易判断
- 对用户真实需求有意义
- 能在 benchmark viewer 中一眼看懂

例如：

- 输出包含“风险与阻塞”小节
- 生成了目标 `.docx` 文件
- CSV 中包含 `profit_margin` 列
- JSON 输出可以被正常解析
- 保留原始 Markdown 表格
- 每个任务都有负责人和截止时间
- 输出没有翻译代码块内容

---

## 不好的 assertions

不好的 assertions 通常太主观。

例如：

- 输出很好
- 内容专业
- 看起来不错
- 写得有水平
- 用户会满意

这些标准不适合自动评估。

---

## 主观型 skill 怎么办

有些 skill 天然偏主观，例如：

- 写作风格
- 设计质量
- 艺术创作
- 文案美感
- 品牌调性

这类 skill 不适合强行写过度客观的 assertions。

更好的方式是：

1. 设置少量基础客观要求
2. 其余部分交给用户做 qualitative evaluation，也就是人工质量判断

例如，对于“写作风格 skill”：

可以客观检查：

- 是否保留原文含义
- 是否输出指定格式
- 是否避免使用禁用词

但这些就更适合人工判断：

- 是否有感染力
- 是否高级
- 是否有品牌感

---

# 十、用户在流程中的角色

使用 `skill-creator` 时，用户不是旁观者，而是评审者和决策者。

你需要参与这些环节：

## 1. 提供真实场景

skill 的质量取决于测试场景是否真实。

不要只给抽象描述，最好提供你真实会输入的例子。

---

## 2. 判断输出是否符合预期

自动评分可以发现一部分问题，但最终还是你最清楚什么结果有用。

---

## 3. 提供具体反馈

反馈越具体，skill 改得越快。

较弱反馈：

> 还是不太好。

较好反馈：

> 第 1 个输出结构对了，但内容太泛，没有结合我提供的项目背景。请让 skill 必须引用输入中的具体事实。

---

## 4. 决定何时停止迭代

skill 不需要无限改。

当它在主要测试用例上表现稳定，并且明显优于 baseline，就可以进入打包阶段。

---

# 十一、常见问题

## 1. skill-creator 会自动帮我完成所有事吗？

它会尽量自动化，但不是完全替代用户判断。

尤其是下面这些事情，仍然需要你参与：

- 判断输出是否真的有用
- 提供真实测试案例
- 决定某些主观结果是否满意
- 确认最终 skill 是否可以交付

---

## 2. 我一开始需求不清楚，可以用吗？

可以。

你可以先说一个大概想法，例如：

> 我想做一个帮我整理学习笔记的 skill，但具体还没想清楚。

`skill-creator` 会帮助你把目标、触发场景和输出格式逐步明确。

---

## 3. description 为什么这么重要？

因为 Claude 是否调用 skill，主要取决于 `description`。

如果 description 太窄，可能该触发时不触发。

如果 description 太宽，可能不该触发时也触发。

所以 description 不是普通简介，而是 skill 的“触发规则”。

---

## 4. skill 写得越长越好吗？

不是。

好的 skill 应该清晰、结构化、可执行。

过长的指令可能带来三个问题：

1. Claude 难以抓住重点
2. 增加上下文负担
3. 更容易产生互相冲突的规则

更推荐的做法是：

- 主体指令保持精炼
- 复杂资料放到 references 或 scripts
- 需要时再按需读取

---

## 5. 每个 skill 都必须写 evals 吗？

不是强制，但强烈推荐。

如果只是个人临时使用，可以简单测试。

如果这个 skill 会长期使用、分享给别人，或者用于重要工作，就应该写 evals。

---

## 6. 所有 skill 都适合自动评分吗？

不是。

偏格式、数据、文件处理、代码生成的 skill 更适合自动评分。

例如：

- JSON 生成
- CSV 处理
- 文档转换
- PR 描述生成
- 报告结构生成
- 测试文件生成

偏审美、风格、创意的 skill 更适合人工审查。

例如：

- 海报设计
- 文案风格
- 小说创作
- 品牌表达
- 视觉审美

---

# 十二、完整使用流程总览

下面是使用 `skill-creator` 的完整流程：

```text
1. 说明你想创建或改进什么 skill       
    ↓
2. 明确 skill 目标、触发场景和输出格式      
    ↓
3. 生成或修改 SKILL.md        
    ↓
4. 创建真实测试用例        
    ↓
5. 编写客观 assertions       
    ↓
6. 运行 with-skill 和 baseline 对比测试       
    ↓
7. 查看测试结果和 benchmark       
    ↓
8. 根据结果反馈并改进 skill        
    ↓
9. 重复测试与改进，直到表现稳定        
    ↓
10. 优化 description，提高触发准确度        
    ↓
11. 验证 skill 结构        
    ↓
12. 打包为 .skill 文件
```

---

# 十三、最佳实践

## 1. 先做小，再做大

不要一开始就创建一个“什么都能做”的 skill。

更好的方式是先聚焦一个明确任务。

例如：

不推荐：

> 创建一个帮我处理所有文档的 skill。

推荐：

> 创建一个把英文技术文档翻译成中文，并保留 Markdown 结构的 skill。

---

## 2. 用真实案例测试

测试用例越真实，skill 越可靠。

不要只用简单、理想化的输入。

---

## 3. 明确“不做什么”

很多 skill 的问题不是不会做，而是做太多。

所以应该写清楚边界。

例如：

> 不要主动扩写用户没有提供的信息。  
> 不要改变代码块内容。  
> 不要把概念解释任务误判为项目计划任务。  
> 不要输出与用户请求无关的建议。

---

## 4. 输出格式要具体

不要只写：

> 输出清晰结果。

更好的是：

> 使用 Markdown 输出，包含：背景、目标、任务拆解、风险、下一步行动。每个任务必须包含负责人、截止时间和完成标准。

---

## 5. 把主观偏好变成具体规则

例如，不要只说：

> 写得高级一点。

可以改成：

> 避免口号式表达；减少形容词堆叠；优先使用具体事实；每段只表达一个核心意思；语气克制、清晰、专业。

---

## 6. 不要过度依赖 description

`description` 只负责触发。

真正决定 skill 行为的是正文指令。

所以：

- description 负责“什么时候用”
- 正文负责“怎么做”
- evals 负责“做得好不好”

---

# 十四、最终交付物

使用 `skill-creator` 后，通常会得到这些产物：

```
your-skill/
├── SKILL.md
├── LICENSE.txt
├── references/
├── scripts/
├── evals/│   
└── evals.json
└── your-skill.skill
```

其中最重要的是：

|文件|作用|
|---|---|
|`SKILL.md`|skill 的核心指令文件|
|`evals.json`|测试用例和 assertions|
|`.skill`|可安装、可分享的打包文件|
|`references/`|额外参考资料|
|`scripts/`|可执行脚本或辅助工具|


# 一、简介

`Agent Harness`可以理解为让模型真正能干活的一整套外部工程环境。

一个完整的 `Agent` 产品不只有模型，而是：**模型 + Harness**。

其中：

- 模型负责理解任务、推理目标、决定下一步行动
- Harness负责提供工具、上下文、知识、权限和执行环境

一个形象的比喻：

> 模型是驾驶者，Harness是载具

模型本身具备一定的感知、推理和行动决策能力，但如果没有外部工具和环境，它只能生成文本，不能真正读文件、执行命令、调用和修改代码。

所以，`Harness`的作用不是创造智能，而是让模型的智能可以落地。



# 二、Agency

`Agency`可以理解为一个系统具备以下能力：

- 感知环境；
- 理解目标；
- 推理下一步；
- 执行行动；
- 根据结果继续调整。

例如，一个编程 Agent 收到用户需求后，会阅读代码、分析项目结构、修改文件、运行测试、根据报错继续修复。

这个过程中，真正负责“判断下一步该做什么”的是模型，而不是外层代码。

很多人容易将`Agent`理解为一堆外部流程：

```text
先总结需求 -> 再生成计划 -> 再调用工具  -> 再检查结果 -> 再输出答案
```

这种流程本身只是编排。它可以帮助模型工作，但不能凭空产生真正的推理和行动能力。

模型之所以能根据任务选择行动，是因为它在训练过程中学到了大量语言、代码、任务过程和推理模式。

因此：

- 模型负责产生决策能力
- Harness负责提供行动条件

外部代码可以让模型“有手有眼”，但不能替代模型本身的理解和推理。


# 三、Agent不等于工作流编排

现在很多产品会把多个 `LLM API` 调用串起来，然后称为 `Agent`。

例如：

```text
用户输入
  ↓
提示词 A 总结需求
  ↓
提示词 B 生成计划
  ↓
提示词 C 执行某一步
  ↓
提示词 D 输出结果
```

这种方式更接近**工作流自动化**，不一定是真正意义上的 Agent。

它的问题是：

- 行动路径主要由外部代码写死；
- 模型只是流程中的文本处理节点；
- 任务一旦超出预设路径，就容易失效；
- 系统看起来复杂，但泛化能力有限。

真正的 Agent 系统应该尽量让模型根据当前环境自己判断下一步，而不是用大量固定规则替模型做决定。也就是说，Agent 产品的重点不是写很多 `if-else`，而是构建一个适合模型工作的环境。



# 四、Harness

## 1、概念

`Harness`是模型完成任务所需的外部基础设施。

简单理解：

```text
Harness = Tools + Knowledge + Observation + Action Interfaces + Permissions
```

即工具、知识、观察能力、行动接口、权限边界。

这些东西共同构成了模型工作的环境。

## 2、Tools

`Tools`是模型可以调用的工具。例如在编程 Agent 中，常见工具包括：

- 读取文件
- 编辑文件
- 搜索代码
- 执行 Shell 命令
- 运行测试
- 调用浏览器
- 查询数据库
- 调用外部 API

没有工具，模型只能输出建议；有了工具，模型才能真正操作环境。



## 3、Knowledge

`Knowledge`是模型可以使用的领域知识。

例如：

- 项目说明
- API文档
- 编码规范
- 架构设计
- 业务规则
- 使用手册
- 团队规范

这些知识不一定要一次性全部塞进上下文。更好地方式是让模型知道有哪些资料可用，然后在需要时按需读取。

这样可以减少无关信息干扰，也可以节省上下文空间。


## 3、Observation

`Observation`是模型观察环境状态的方式。

例如：

- 命令执行结果
- 错误日志
- 测试输出
- `git diff`
- 文件内容
- 浏览器页面状态
- API 返回结果

模型每次行动之后，都需要看到环境反馈，才能判断下一步该怎么做。如果没有观察能力，模型就不知道自己的操作是否成功。

## 4、Action Interfaces

`Action Interfaces`是模型影响外部环境的接口。

例如：

- 执行 CLI 命令；
- 修改文件；
- 调用 HTTP API；
- 操作网页 UI；
- 创建任务；
- 提交代码；
- 发送消息。

这些接口决定了 Agent 能实际做什么。

不同领域的 Harness，行动接口也不同。编程 Agent 的接口可能是终端和文件系统；酒店运营 Agent 的接口可能是预订系统、客户沟通系统和房态管理系统。

## 5、Permissions

`Permissions`是 Agent 的权限边界。因为 Agent 可以调用工具，所以必须限制它能做什么、不能做什么。

例如：

- 读取文件是否需要确认；
- 修改文件是否需要确认；
- 是否允许执行 Shell 命令；
- 是否允许访问网络；
- 是否允许删除文件；
- 是否允许操作生产环境；
- 是否需要沙箱隔离。

权限控制不是可有可无的功能，而是 Agent 产品安全运行的基础。


# 五、Harness工程师职责

如果不是训练模型，而是在开发 Agent 产品，那么大多数时候做的其实是 `Harness 工程`。

## 1、实现工具

Harness 工程师需要给模型设计工具。

好的工具应该：

- 名称清楚；
- 功能单一；
- 参数明确；
- 返回结果可读；
- 可以和其他工具组合使用。

例如，文件读取工具只负责读取文件，命令执行工具只负责执行命令，搜索工具只负责查找内容。不要把太多职责塞进一个工具里。


## 2、管理上下文

`Agent`工作时会不断产生上下文，如果上下文不管理，就会越来越长、越来越乱，最终影响模型判断。

因此 Harness 需要处理：

- 上下文压缩
- 长任务摘要
- 子任务隔离
- 历史信息清理
- 任务状态持久化

这也是为什么很多 Agent 产品会有 `/clear`、`/compact`、子 Agent、任务列表等机制。


## 3、组织知识

模型需要知道当前项目或领域的规则。

例如在编程场景中，它可能需要知道：

- 项目结构
- 代码风格
- 测试方式
- 构建命令
- 分支规范
- 团队约定
- API使用方式

这些知识可以来自 `README`、项目文档、`CLAUDE.md`、技能文件、接口文档等。

Harness 要解决的问题是：什么时候给模型什么知识。

不是所有知识都应该一开始就塞进去。对于复杂项目，更合理的方式是按需加载。


## 4、控制权限

Agent 能力越强，权限设计越重要。

例如在 Claude Code 中，不同权限模式控制了它在编辑文件、执行命令或访问系统资源时是否需要用户确认。你给的模板中提到的 `default`、`acceptEdits`、`plan`、`bypassPermissions`，本质上就是不同程度的 Harness 权限策略。

因此，好的 Harness 不应该只有“全开”或“全关”，而应该根据任务和风险提供不同的权限模式。


## 5、反馈执行结果

模型调用工具后，Harness需要把执行结果返回给模型。

例如：

```text
模型决定运行测试
  ↓
Harness 执行测试命令
  ↓
Harness 收集测试输出
  ↓
结果返回给模型
  ↓
模型根据报错继续修复

```

这个反馈循环非常关键。

如果工具结果不清楚，模型就无法准确判断下一步。

这里最关键的一点是：

> 模型决定是否调用工具，Harmess负责执行工具


## 2、Agent Loop和Harmess的关系

`Agent Loop`是最小运行机制，`Harmess`是围绕这个循环建立的一整套工程能力。

因此，可以这样理解：

- `Agent Loop`负责让模型与工具循环交互
- `Harmess`负责让这个循环安全、稳定、可扩展

最小循环只能让模型调用工具。

但真实产品还需要：

- 工具注册
- 权限控制
- 错误处理
- 上下文压缩
- 长任务管理
- 多Agent协作
- 后台任务
- 工作区隔离
- 日志记录

这些都是`Harness`工程的一部分。



# 七、Harness设计原则

1. 不要过度替模型作决策
2. 工具要清晰可组合
3. 知识要按需加载
4. 权限要分层设计
5. 执行结果要及时反馈


## 一、概述

`LLM Wiki`是一种由**大语言模型持续维护**的个人知识库方法。它的核心目标不是让`LLM`在每次提问时重新开始检索资料、拼接答案，而是让`LLM`把**原始资料逐步整理、综合、更新成持久的Markdown Wiki**。

简单说：

> LLM Wiki = 原始资料 + LLM持续整理 + 结构化、相互链接的Markdown 知识库

它把`LLM`从 **临时问答工具** 变成 *长期知识库维护者*。

传统的 RAG、文件上传、ChatGPT 文件问答等方式，通常是这样的流程：

1. 用户上传资料
2. 系统对资料建立索引
3. 用户提问
4. LLM检索相关片段
5. LLM根据片段生成答案

这种方式很有用，但问题是：

`LLM`每次都像第一次接触这些资料一样，需要重新检索、重新理解、重新综合。如果问题比较简单，这没有太大问题。但如果问题需要综合多份文档、多个时间带你、多个观点时，`LLM`就必须在每次回答时重新组装知识。

结果就是：

- 知识没有持续沉淀
- 交叉关系不会自然积累
- 矛盾和版本变化被忽略
- 每次复杂问题都像重新研究一遍
- 用户很难形成长期可浏览、看演化的知识结构

因此，`LLM Wiki`用于解决该问题。

## 二、Obsidian笔记软件

`LLM Wiki`的核心是：Agent + Markdown 文件目录。因此，理论上完全可以用 Agent 直接操作本地文件夹。

但普通文件夹更像存储层，适合`Agent`读写，但不适合人长期浏览和审阅。

因此，使用`Obsidian`来管理本地知识库。它主要由三大优势：

- **更好浏览**：Agent 写出\[\[双向连接\]\]，在 Obsidian 里可以直接点击跳转。
- **更好审阅**：可以查看反向链接、关系图谱、孤立页面
- **不锁数据**： Obsidian 的底层还是本地 Markdown 文件，Agent 可以直接改，Git 可以直接管，也可以随时迁移

此外，`Obsidian`还推出了`Obsidian CLI`，使得`Agent`更加方便操作`Obsidian`仓库。

> 用`Karpathy`的话来说：`Obsidian`是IDE，`LLM`是程序员，`Wiki`是代码库，

## 三、架构

`LLM Wiki`分为三层：

- 原始资料（`raw/`）：来源文档集合，可以是文章、技术博客、论文、图片、视频等，它们是*不可变的*。`LLM`从它们读取但从不修改。
- Wiki页面（`wiki/`）：一个由`LLM`生成的`Markdown`文件目录，可以再细分为 summaries（摘要）、entities（实体）、concepts（概念）、comparisons（比较）、overviews（概览）、syntheses（综合结论）等等。
- Schema（例如 Claude Code 的`CLAUDE.md`）：Wiki维护规范文档，用来告诉 LLM wiki的构建方式、约定规范，以及在摄入素材、回答问题、维护wiki时应该遵循什么工作流。这是**关键的配置文件**，它让 LLM 变成有纪律的 wiki 维护者，而不是普通的聊天机器人。用户和`LLM`在探索适合的领域过程中不断演进这份文档。

## 四、操作

管理`LLM Wiki`知识库主要有三个操作，分别是`Ingest`（摄取）、`Query`（查询）、`Lint`（校验）。

- **Ingest**：将新的素材放入`/raw`原始资料集合并告诉`LLM`处理它。一个示例流程：`LLM`阅读素材，与你讨论关键要点，在 wiki 中写一个 summary（摘要页面），更新`index`，更新 wiki 中相关的 entity（实体）和 concept（概念）页面，并在`log`中追加条目。单个素材可能触及 10～15 个页面。建议一次`ingest`编译一个素材并让你保持参与——你阅读摘要、检查更新，并引导`LLM`重点强调什么。你也可以批量 `ingest` 多个素材，减少监督。
- **Query**：你向 wiki 提问。`LLM`搜索相关页面，阅读它们，并综合出一个带引用的答案。答案可以根据问题采取不同方式——`Markdown`页面、对比表格、幻灯片（Marp）、图表（matplotlib）、画布。**重要的洞察：好的答案可以作为新页面归档回 wiki 页面**。你要求的一次对比、一次分析、一个你发现的关联——这些都有价值，不应消失在聊天记录中。这样，你的探索就像 ingest 的素材一样，在知识库中复合增长。
- **Lint**：定期让`LLM`对 wiki 进行健康检查。查找：页面中的矛盾、被新素材取代的陈旧说法、没有入站链接的孤立页面、被提及但缺少独立页面的重要概念、缺失的交叉引用、可以通过网络搜素填补的数据空白、亦或是可以通过网络搜索推翻的错误说法。LLM 很擅长建议要调查的新问题和要寻找的新来源。这有助于 wiki 在增长过程中保持健康。

## 五、index 与 log

两个特殊文件帮助`LLM`和你 在 wiki 增长时进行导航。它们服务于不同目的：

- `index.md`：该文件以内容为导向，它是 wiki 中所有内容的目录——每个页面都列出链接、一行只要，以及可选的元数据如日期或素材数量。按类别组织（entities、concepts、source等）。`LLM`在每次`ingest`时更新它。回答`query`时，`LLM`先阅读`index.md`找到相关页面，然后深入阅读。这在中等规模（~100个素材，~数百个页面）下出奇地有效，避免了基于嵌入的 RAG 基础设施的需求。
- `log.md`：该文件以时间为导向。它是按时间顺序的、**仅追加**的记录，记录发生了什么、何时发生——`ingest`、`query`、`lint`。一个有用的技巧：如果每个条目以一致的前缀开头（例如`## [2026-04-02] ingest | 文章标题`），log 就可以用简单的 Unix 工具解析 —— `grep "^## \[" log.md | tail -5`就能给出最近 5 条条目。log 提供了 wiki 演进的时间线，并帮助`LLM`理解最近做了什么。


## 六、可选：CLI工具

在某些时候，你可能想构建一些小工具来帮助 LLM 更高效地操作 wiki。最明显的需求是 wiki 页面的搜索引擎——在小规模下 index 文件就足够了，但随着 wiki 增长，你需要效率更高的搜索。[qmd](https://github.com/tobi/qmd) 是一个不错的选择：它是一个本地 markdown 文件搜索引擎，带有混合 BM25/向量搜索和 LLM 重排序，全部在本地运行。它既有 CLI（因此 LLM 可以调用它），也有 MCP 服务器（因此 LLM 可以将其作为原生工具使用）。你也可以自己构建更简单的工具——LLM 可以在需要时帮你快速开发一个朴素搜索脚本。


## 七、技巧和窍门

- **Obsidian Web Clipper**：是一个浏览器扩展，可以将网页文章转换为 markdown。对于快速将素材放入你的 raw sources 集合非常有用。
- **本地下载图片**：在`Obsidian`设置 -> 文件与链接中，将“附件文件夹路径”设为固定目录（如`assets/`）。然后在设置 -> 快捷键中，搜索“下载”找到“下载当前文件内所有附件”，并将其绑定到快捷键（例如Ctrl + Shift + D）。这样，剪辑文章后，按快捷键，所有图片都会下载到本地磁盘。这是可选但有用的——它让 LLM 可以直接查看和引用图片，而非依赖可能失效的 URL。**注意 LLM 不能原生地在一次阅读中读取带有内嵌图片的 markdown——解决方法是让 LLM 先阅读文本，然后单独查看部分或全部引用的图片以获得额外上下文**。在指定`CLAUDE.md`或`ingest`、`query`技能时，需要特殊处理包含图片的`Markdown`来源。
- **Obsidian 的查看关系图谱**： 关系图谱是查看 wiki 形状的最佳方式——什么与什么相连，哪些是枢纽页面，哪些是孤立页面。
- **Marp**：Marp是一种基于 markdown 的幻灯片格式。Obsidian 有它的插件。对于直接从 wiki 内容生成演示文稿很有用。
- **Dataview**：Dataview是一个 Obsidian 插件，可以在页面 frontmatter 上运行查询。如果你的`LLM`为 wiki 页面添加YAML frontmatter（标签、日期、素材数量），Dataview 可以生成动态表格和列表。
- **版本管理**：wiki 就是一个 markdown 文件的 git 仓库。你免费获得了版本历史、分支和协作功能。

# Claude Code 的 Prompt Caching 机制深度解析

## 引言

Claude Code 作为 Anthropic 推出的 AI 编程助手，在处理大型代码库时面临一个核心挑战：如何在保持上下文窗口充分利用的同时，避免重复计算带来的高昂成本。Prompt Caching（提示缓存）机制正是为了解决这一问题而设计的。

## 核心原理

### 什么是 Prompt Caching

Prompt Caching 允许 Claude 将之前处理过的提示内容缓存起来，当后续请求中包含相同或相似的前缀内容时，可以直接复用缓存结果，而不需要重新计算。

### 缓存的工作层级

1. **系统提示层**：包括 Claude 的角色定义、工具定义等，这部分内容在会话期间几乎不变
2. **上下文层**：包括代码库结构、文件内容等，这部分在单次任务中相对稳定
3. **用户输入层**：每次请求独有的内容，这部分无法缓存

## 实现细节

### 缓存键的生成

缓存键基于提示内容的哈希值生成。Anthropic 使用了一种滑动窗口哈希算法，可以在内容发生局部变化时仍然命中缓存。

### 缓存命中率优化

在实际使用中，缓存命中率可以达到 70-90%，这意味着：
- 处理相同代码库的不同文件时，大部分上下文可以复用
- 连续的多轮对话中，历史消息不需要重复编码

## 与 RAG 的对比

传统 RAG（检索增强生成）方案需要在每次请求时重新检索相关文档片段，而 Prompt Caching 则直接复用完整的上下文表示。

| 特性 | RAG | Prompt Caching |
|------|-----|----------------|
| 首次请求延迟 | 低 | 高 |
| 后续请求延迟 | 中 | 低 |
| 上下文完整性 | 片段化 | 完整 |
| 适用场景 | 问答 | 代码编辑 |

## 最佳实践

1. **保持提示前缀稳定**：将不变的内容放在提示开头
2. **合理组织上下文**：将相关的文件内容放在一起
3. **监控缓存命中率**：通过 API 响应中的 `cache_creation_input_tokens` 和 `cache_read_input_tokens` 字段

## 局限性

- 缓存内容有 TTL（生存时间），长时间不活跃会被清理
- 缓存命中只针对前缀匹配，中间内容变更会导致后面的缓存失效
- 需要支付缓存写入的成本（通常是标准输入 token 成本的 1.25 倍）

## 结论

Prompt Caching 是 Claude Code 能够高效处理大型代码库的关键技术。它代表了从"每次重新检索"到"一次编译，多次复用"的范式转变。

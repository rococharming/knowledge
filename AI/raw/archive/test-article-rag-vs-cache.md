# 从 RAG 到 Prompt Caching：知识检索范式的演进

## 背景

随着大语言模型上下文窗口的不断扩大（从 4K 到 200K+），我们对知识检索的理解也在发生变化。传统的 RAG 方案正在面临新的挑战。

## RAG 的局限性

### 片段化问题

RAG 将文档切分成小块，每次查询只检索相关片段。这导致了：
- 上下文断裂：无法看到完整的论证链条
- 重复检索：相同的问题每次都要重新搜索
- 知识不积累：检索结果不会随时间改善

### 检索质量瓶颈

即使使用向量搜索 + 重排序，检索准确率也存在上限。复杂的查询需要综合多份文档时，RAG 的效果会显著下降。

## Prompt Caching 的崛起

Anthropic 在 Claude 3.5 Sonnet 中引入的 Prompt Caching 机制，代表了另一种思路：与其每次都检索片段，不如一次性加载完整上下文并复用。

### 核心优势

1. **完整性**：整个代码库或文档集作为完整上下文存在
2. **一致性**：不会因为分块策略不同而丢失关联
3. **效率**：复用缓存后，后续请求成本大幅降低

## LLM Wiki 的启示

Karpathy 提出的 LLM Wiki 概念进一步扩展了这个思路：让 LLM 主动维护一个结构化的知识库，而不是被动地响应检索请求。

在这种范式下：
- 知识被"编译"一次，然后持续维护
- 交叉引用在编译阶段就建立好了
- 矛盾和新信息在 ingest 时就被标记和处理

## 未来展望

Prompt Caching 和 LLM Wiki 代表了从"即时检索"到"持久知识"的范式转变。随着上下文窗口继续扩大和缓存成本下降，这种趋势会越来越明显。

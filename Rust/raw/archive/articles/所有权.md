# 一、Rust内存管理

所有编程语言都必须管理程序运行时使用的内存，常见路线有三类：

**垃圾回收（GC）**：如Java、Python语言。**运行时会追踪不再使用的对象并回收**。优点是开发体验轻松；缺点是可能存在运行时开销、暂停时间和内存占用等问题，具体表现取决于语言实现和GC策略。

**手动管理**：如C/C++语言。开发者负责malloc/free或new/delete。优点是控制能力强；缺点是容易产生内存泄漏、悬垂指针、二次释放、越界访问等安全问题。

**编译期所有权系统：** Rust。Rust通过 **所有权 + 借用 + 生命周期** 在**编译阶段静态检查内存使用是否合法**。大多数内存安全问题会在编译期被发现，不需要依赖GC，也不要求开发者手动释放普通内存。

简单来说，C/C++将内存正确性交给程序员自律，但程序员总有犯错的时候，不可避免地会产生内存安全问题；GC把内存管理交给运行时，虽然降低了手动管理的负担，但会带来额外运行时成本。**Rust则另辟蹊径，将它交给编译器规则系统，在不依赖GC的前提下尽量兼顾性能和内存安全。**

需要注意的是，Rust并不是说“只要编译通过，程序就永远不会出错”。Rust主要保证的是内存安全，且这个保证默认建立在安全Rust代码之上。如果使用unsafe代码、外部FFI或者逻辑本身写错，仍然可能产生问题。


# 二、作用域与所有权

## 1、变量的作用域

变量作用域（Scope）是变量在程序中有效的范围，通常由 **{}块** 界定。

示例：

```rust
fn main() {
    { // a 还未声明

        let a = 1; // a 进入作用域

        println!("a: {}", a); // 使用a

    } // a离开作用域
}
```

变量进入作用域之后可以被使用；离开作用域之后，变量绑定失效，对应的值也会按照所有权规则被处理。

## 2、Drop

Rust中资源管理遵循`RAII`（资源获取即初始化）。

核心思想：

> 当值离开作用域，会自动调用其析构逻辑。

Rust中通常称为`drop`，由`Drop`Trait参与。

- 对栈上的纯数据：离开作用域后栈空间自然失效，不需要额外释放动作
- 对拥有堆内存或系统资源的值：在drop时释放堆内存、关闭文件句柄、释放锁等。

示例：

```rust
fn main() {
	{
	    let s = String::from("hello");
	}
	// s离开作用域，自动drop，释放String持有的堆内存
} 
```

> String是Rust的字符串类型

这里并不需要手动调用free/delete，Rust会在所有者离开作用域时自动完成清理。

## 3、所有权三大规则

Rust所有权规则可以概括为三条：

1. **每一个值都有一个所有者**
2. **同一时刻，一个值只有一个所有者**
3. **当所有者离开作用域，值被丢弃**

这三条规则配合编译器检查，使得Rust能在不依赖GC的情况下避免大量内存错误。

> 注意，这里的“一个所有者”指的是默认所有权模型。Rust也允许共享所有权，但需要通过Rc、Arc等特殊类型显式表达，而不是让随意变量共享同一份资源。


## 4、String与所有权

Rust的`String`类型是理解所有权的典型例子。String是Rust的字符串类型，本质是一个拥有**堆缓冲区**的类型，它在栈上保存三个字段：

- 指向堆缓冲区的指针ptr
- 当前长度len
- 当前容量cap

堆上存放的是**UTF-8字节序列**本体。

示例：

```rust
fn main() {

	let s = String::from("Hello");

} // s离开作用域 -> drop -> 释放堆缓冲区
```

下面给出了`String`的内存示意图：

![[Pasted image 20260513102821.png|300]]

`s`变量（`String`值本身）保存在栈上，其中的`ptr`指向堆区实际存放字符串字节的内存。

当`s`离开作用域，栈上的`String`值失效，同时触发drop，自动释放`ptr`指向的堆内存。这里就很好地体现了“当所有者离开作用域，值被丢弃”这一规则。


## 5、所有权体系中的几种核心机制

Rust的所有权规则很严格，但语言和标准库也提供了几种**受控机制**，用于表达更复杂的内存与共享场景：

- **移动（Move）**：值的所有权可以从一个绑定转移到另一个绑定，移动后原绑定不再可用，从而避免重复释放等问题。
- **复制（Copy类型）**：对实现了`Copy`的类型，如整型、浮点型、布尔型、字符型，赋值和传参会复制一份独立的值，原变量仍然可用。
- **借用（Borrow）**：通过`&T`或`&mut T`临时访问一个值，不取得其所有权。引用本身不负责释放值，值何时释放仍由所有者决定。
- **共享所有权（Rc/Arc）**：通过引用计数让多个所有者共同持有同一份数据。当最后一个所有者离开作用域时，数据才被释放。


# 三、移动

## 1、移动的基本概念

在Rust中，对于**大多数非Copy类型**来说，给变量赋值、作为参数传入函数或从函数返回，默认发生的是 **移动（Move）** ，而不是复制。

**移动会将值的所有权从源绑定转移到目标绑定** 。移动之后，源绑定在语义上被视为已经失效，编译器会禁止继续使用它。新的目标绑定负责该值的生命周期，并在适当时机触发drop。

Rust通过移动语义来避免重复释放和悬垂访问等问题。

仍然以String类型为例，来说明移动的过程。

示例：

```rust
fn main(){

let s1 = String::from("hello");

let s2 = s1; // Move

// println!("s1: {}", s1); // 编译报错，s1已经失效

println!("s2: {}", s2);

}
```

当定义s1时，内存示意如下：

![[Image 32.png|400]]

执行let s2 = s1之后，内存示意如下：

![[Image 33.png|400]]

Rust的移动不是运行时把堆上的字符串字节重新复制了一份。对于String来说，移动主要是转移栈上的String结构，即ptr、len、cap这几个字段。堆上的字符串数据仍然是同一份。

Rust之所以让`s1`失效，是为了避免`s1`和`s2`都以为自己拥有同一块堆内存。如果两个变量都负责释放同一块堆内存，就会产生二次释放问题。

## 2、Clone（显式克隆）

如果希望`s1`仍然可用，就不能使用普通赋值了，而应该显式调用`clone`。

示例：

```rust
fn main() {
    let s1 = String::from("Hello");

    let s2 = s1.clone(); // 克隆

    println!("s1: {}", s1);

    println!("s2: {}", s2);
}
```

内存示意图如下：

![[Image 35.png|400]]

此时`s1`和`s2`分别拥有两份独立的`String`数据。

对于String来说，clone通常会分配新的堆内存，并复制字符串内容，因此这是一个相对更重的操作。Rust要求你显式写出clone，就是为了让这种成本在代码中可见。

# 四、Copy类型

## 1、Copy是移动的例外

对于像整数这类实现了`Copy`特征的类型，赋值不会让原变量失效，而是复制一份独立的值。

示例：

```rust
fn main() {
    let num1: i32 = 36;

    let num2 = num1;

    println!("num1: {}", num1);

    println!("num2: {}", num2);
}
```

`i32`只是一个简单的固定大小数值，不拥有堆内存，也不需要特殊释放逻辑。因此释放之后，`num1`和`num2`彼此独立。

![[Image 36.png|400]]

## 2、常见Copy类型

标准库中常见的Copy类型包括：

- 各种整数类型
- 浮点型
- 字符型
- 布尔类型
- 不可变引用类型（&T）
- 元素全是Copy类型的数组
- 字段全部是Copy类型的元组

示例：

```rust
let a = 1;
let b = a; // a仍然可用

let x = true;  
let y = x; // x仍然可用

let p = (1, 2);  
let q = p; // p仍然可用
```

需要注意的是，不可变引用 `&T` 是Copy类型，但可变引用 `&mut T` 不是Copy类型。因为可变引用要求独占访问，如果随意复制多个可变引用，就会破坏借用规则。

## 3、不能成为Copy的类型

`String`不能是`Copy`类型，因为它拥有堆内存，需要在`drop`时释放。

`Box<T>`不能是`Copy`类型，因为它拥有堆上的`T`值，需要在`drop`时释放。

`File`不能是`Copy`类型，因为它代表操作系统文件句柄。复制一个File并不是简单复制几个字节就能正确表达的行为。

`MutexGuard`不能是`Copy`类型，因为它代表对互斥锁的持有，离开作用域时需要释放锁。

简单来说，**任何实现了Drop的类型都不能实现Copy。** 因为Copy意味着可以悄悄复制出多个值，而Drop意味着这些值离开作用域时需要执行释放逻辑。两者放在一起很容易导致重复释放或资源状态混乱。

后续介绍的[[6、结构体|结构体]]和[[7、枚举|枚举]]默认也不是`Copy`类型。如果它们的所有字段都是Copy类型，可以通过 `#[derive(Copy, Clone)]` 让它们成为Copy类型。


# 五、函数中的所有权转移

## 1、函数传参

当实参传递给函数形参时，对于非Copy类型会发生所有权转移。

示例：

```rust
fn main() {
    let s = String::from("Hello");

    take_ownership(s); // s移动进函数，此后s失效

    // println!("{}", s); // 错误
}

fn take_ownership(some_string: String) {
    println!("some_string: {}", some_string);
}
```

上述例子中，值的所有权从main函数中的s移动到了take_ownership函数中的some_string。当take_ownership函数结束时，some_string离开作用域，String被drop。

## 2、Copy类型传参

如果传入的是Copy类型，函数调用时会复制一份值，原变量仍然可用。

示例：

```rust
fn main() {
    let x = 5;

    makes_copy(x);

    println!("x: {}", x); // 正确，i32是Copy类型
}

fn makes_copy(some_integer: i32) {
    println!("some_integer: {}", some_integer);
}
```

这里x没有失效，因为i32实现了Copy。

## 3、函数返回值

当函数返回一个非Copy类型的值时，所有权会转移给调用者。

示例：

```rust
fn main() {
    let s1 = gives_ownership();

    let s2 = String::from("Hello");

    let s3 = takes_and_gives_ownership(s2);

    // println!("{}", s2); // 错误，s2已经移动给函数

    println!("{}", s1);

    println!("{}", s3);
}

fn gives_ownership() -> String {
    let s = String::from("Hello");

    s // s的所有权转移给调用者
}

fn takes_and_gives_ownership(s: String) -> String {
    s // s的所有权转移给调用者
}
```

函数返回值是Rust中非常常见的所有权转移方式。


# 六、移动与控制流

前面的例子大多都是顺序代码，但在真实场景中，移动往往会和条件分支（if/else）和循环等控制流交织在一起。

## 1、if/else中的移动

移动也会受到控制流影响。

```rust
fn main() {
    let s = String::from("hello");

    if true {
        consume(s);
    } else {
        consume(s);
    }

    // println!("{}", s); // 错误，s已经被移动
}

fn consume(s: String) {
    println!("{}", s);
}
```

在两个分支中，s都有可能被移动。因此if/else之后，s不再可用。即便只有某个分支移动了s，编译器也不允许后续继续使用s。

## 2、循环中的移动

Rust不允许在可能执行多次的循环中反复移动同一个非Copy值。

示例：

```rust
fn main() {
    let s = String::from("hello");

    loop {
        consume(s); // 错误，第一次循环后s已经被移动
    }
}

fn consume(s: String) {
    println!("{}", s);
}
```

第一次循环时，s已经被移动给consume。第二次循环时，s就已经无效了，因此编译器会直接拒绝这段代码。

如果只是想在循环中使用s，可以使用[[4、引用和切片|借用]]。


# 七、从复合类型中移动值
## 1、不能直接从Vec索引中移动非Copy值

移动会把值的所有权从源位置转移到目标位置。移动之后，源位置就不再拥有有效值。

当源位置是某个复合类型的一部分时，比如Vec中的某个索引，如果直接把值移动走，就会在原位置留下一个空洞。Rust不允许Vec处于这种不完整状态，因此**禁止从索引表达式中直接移动非Copy值**。

示例：

```rust
fn main() {
    let v = vec![
        String::from("hello"),
        String::from("world"),
        String::from("rust"),
    ];

    let first = v[0]; // 错误
}
```

编译报错：

![[Image 37.png]]

`v[0]`中存放的是String，而String不是Copy类型。如果允许直接移动，就会导致v的第一个位置没有有效元素，但v仍然以为自己有三个元素。

如果Vec中存放的是Copy类型，则没有这个问题：

```rust
fn main() {
    let v = vec![1, 2, 3];

    let first = v[0]; // 正确，i32是Copy类型

    println!("{}", first);

    println!("{}", v[0]);
}
```

这里移动并没有发生，`v[0]`只是被复制了一份。

## 2、从Vec中取出元素的常见方式

如果确实想从Vec中取出某个元素，可以使用标准库提供的方法。

1. 从末尾弹出（只能移动末尾的值，向量缩短）

示例：

```rust
fn main() {
    let mut v = vec![
        String::from("hello"),
        String::from("world"),
        String::from("rust"),
    ];

    let last = v.pop().expect("vector is empty");

    println!("last element: {}", last);
}
```

pop会取出最后一个元素，同时Vec长度减1。

2. 交换删除（将目标位置与末尾交换，再弹出目标，向量缩短，不保证顺序）

```rust
fn main() {
    let mut v = vec![
        String::from("hello"),
        String::from("world"),
        String::from("rust"),
    ];

    let second = v.swap_remove(1);

    println!("second element: {}", second);
}
```

swap_remove会把目标位置的元素移出来，并用最后一个元素填补该位置。它会改变元素顺序，但效率较高，Vec长度也会减1。

3. 替换元素（用替代值填充原槽位，将原值移动出来，向量不变）

示例：

```rust
fn main() {
    let mut v = vec![
        String::from("hello"),
        String::from("world"),
        String::from("rust"),
    ];

    let first = std::mem::replace(&mut v[0], String::from("hi"));

    println!("first element: {}", first);
}
```

这几种方式的共同点是：Rust允许你移动元素，但必须保证原来的容器仍然保持合法状态。


## 3、for循环消耗Vec

Vec这样的集合类型也可以通过for循环把所有元素逐个移动出来。

```rust
fn main() {
    let v = vec![
        String::from("hello"),
        String::from("world"),
        String::from("rust"),
    ];

    for s in v {
        println!("{}", s);
    }

    // println!("{:?}", v); // 错误，v已经被移动
}
```

`for s in v` 会调用v的into_iter，把v的所有权移动进迭代器中。循环过程中，每个String元素也会被依次移动出来。因此循环结束后，v不再可用。

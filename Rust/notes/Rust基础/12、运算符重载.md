# 一、运算符重载简介

在Rust中，运算符并不是“语法糖式的特殊机制”，而是 **建立在trait之上的一套统一接口** 。也就是说，像 `+` 、 `-` 、 `*` 、 `/` 、 `==` 、 `<` 、 `[]` 这些看起来像语言内建的操作，本质上都对应着标准库中的trait方法调用。 **为自定义类型实现这些trait，就等于为该类型赋予了对应运算符的能力** 。

例如下面的Complex类型：

```rust
#\[derive(Clone, Copy, Debug)\]

struct Complex<T> {

/// 复数的实部

re: T,

/// 复数的虚部

im: T,

}
```
对于自定义复数类型，希望像使用普通数值一样进行运算，例如：
```rust
z = z \* z + c;
```
想要这样使用，需要 **为该类型实现对应运算符的特征（trait）** ，从而支持 `+` 、 `*` 等运算符。
# 二、运算符重载的整体设计思想

Rust 的运算符重载并不是为了“语法炫技”，而是为了让抽象类型在 **不牺牲类型安全和可读性** 的前提下，拥有自然的使用方式。

从设计角度看，它有几个鲜明特点：

**每个运算符都对应一个明确的Trait**

Rust 不允许用户随意发明新运算符，也不允许把已有运算符改造成完全不相关的含义。你只能在语言允许的范围内，为某个 trait 提供实现。

**运算符行为受类型系统严格约束**

例如， `Add` 并不要求左右操作数必须同类型，也不要求结果类型仍然是原类型。它的定义是：

```rust
trait Add<Rhs = Self> {

type Output;

fn add(self, rhs: Rhs) -> Self::Output;

}
```
这说明 Rust 的运算符重载不是简单“贴语法”，而是和泛型、关联类型、trait约束深度结合的。

**与所有权和借用系统保持一致**

有些运算符按值接收参数，例如 `Add::add(self, rhs)` ，有些运算符按引用接收参数，例如 `PartialEq::eq(&self, &rhs)` 。这不是随意设计，而是与类型使用方式密切相关。例如：

加法通常生成新值，因此按值获取很自然

比较通常只是读取值，不应造成所有权转移，因此按引用更合理

**不是所有运算符都允许重载**

一些运算符直接关系到 Rust 的内存安全模型和语言语义，例如：

借用运算符 `&`

赋值运算符 `=`

逻辑短路运算符 `&&` 、 `||`

错误传播运算符 `?`

这些都不能由用户任意改写。这样做是为了保证语言的关键语义稳定可靠。

## 2、核心规则

`Neg` 和 `Not` 的定义形式如下：

```rust
trait Neg {

type Output;

fn neg(self) -> Self::Output;

}

trait Not {

type Output;

fn not(self) -> Self::Output;

}
```
说明：

返回类型通过关联类型 `Output` 指定

方法接收 `self` ，说明默认按值获取操作数

在内置类型中：

所有带符号整数和浮点数都支持 `Neg`

整数和 `bool` 支持 `Not`

需要注意的是，在Rust中，`!`同时兼具两种作用：

对 `bool` 表示逻辑取反

对整数表示按位取反

**这与其他语言把**`!`**和** `~` **分开的设计不同** 。
## 3、代码示例

下面为复数类型实现一元负号：

```rust
use std::ops::Neg;

#\[derive(Debug, Copy, Clone)\]

struct Complex<T> {

re: T,

im: T,

}

impl<T> Neg for Complex<T>

where

T: Neg<Output = T>

{

type Output = Self;

fn neg(self) -> Self::Output {

Complex {

re: -self.re,

im: -self.im,

}

}

}

fn main() {

let c = Complex { re: 3, im: -4 };

println!("{:?}", c);

println!("{:?}", -c);

}
```
注意事项：

`Neg` 的返回类型不一定必须是原类型，虽然很多场景下 `Output = Self` ，但从 trait 设计上讲，它完全可以返回别的类型。

实现泛型版本时，字段类型必须满足相应约束。如果 `T` 不支持 `-` ，那么 `Complex<T>` 也不可能支持 `-` 。

## 2、Add核心结构

这里主要以 `Add` 为例介绍， `std::ops::Add` 定义如下：

```rust
trait Add<Rhs = Self> {

type Output;

fn add(self, rhs: Rhs) -> Self::Output;

}
```
说明：

`Add<Rhs = Self>` ：表示加法右操作数类型默认是Self，即默认做同类型相加

`type Output` ：关联类型，表示加法结果的类型

`fn add(self, rhs: Rhs) -> Self::Output` ：表示加法操作接收左值 `self` 和右值 `rhs` ，并返回一个 `Output` 类型的结果。

`Add<T>` 表示某个类型具有加上一个 `T` 类型值的能力：

如果希望类型可以加上 `i32` ，就要实现 `Add<i32>`

如果希望类型可以加上 `u32` ，就要实现 `Add<u32>`

**因此，同一个类型可以实现多个不同版本的** `Add` 。如果需求只是同类型之间相加，由于 `Rhs` 默认就是 `Self` ，因此直接实现 `Add` 即可。
## 3、同类型相加

例如，若要让两个 `Complex<i32>` 相加，那么 `Complex<i32>` 必须实现 `Add` 。

示例：

```rust
use std::ops::Add;

#\[derive(Debug, Copy, Clone)\]

struct Complex<T> {

re: T,

im: T,

}

impl Add for Complex<i32> {

type Output = Self;

fn add(self, rhs: Self) -> Self::Output {

Self {

re: self.re + rhs.re,

im: self.im + rhs.im

}

}

}
```
注意：这里实现的是 `Complex<i32> + Complex<i32> -> Complex<i32>`
```rust
use std::ops::Add;

#\[derive(Debug, Copy, Clone)\]

struct Complex<T> {

re: T,

im: T,

}

impl<T> Add for Complex<T>

where

T: Add<Output = T>

{

type Output = Self;

fn add(self, rhs: Self) -> Self::Output {

Self {

re: self.re + rhs.re,

im: self.im + rhs.im,

}

}

}

fn main() {

let c1 = Complex { re: 1, im: 2 };

let c2 = Complex { re: 3, im: 4 };

let c3 = c1 + c2;

println!("{:?}", c3);

}
```
说明：这里要求字段类型 `T` 需要实现 `Add<Output = T>` 。因此只要 `T` 本身能和自身相加， `Complex<T>` 就也能相加。
## 4、左右操作数不同类型的情况

代码示例：

```rust
use std::ops::Add;

#\[derive(Debug, Copy, Clone, PartialEq)\]

struct Complex<T> {

re: T,

im: T,

}

impl<L, R> Add<Complex<R>> for Complex<L>

where

L: Add<R>,

{

type Output = Complex<L::Output>;

fn add(self, rhs: Complex<R>) -> Self::Output {

Complex {

re: self.re + rhs.re,

im: self.im + rhs.im,

}

}

}
```
说明：

这个实现表示： `Complex<L> + Complex<R> -> Complex<L::Output>` ，也就是说，只要 `L` 能和 `R` 相加，就允许对应的复数相加。例如，如果存在 `L + R -> X` ，那么就有 `Complex<L> + Complex<R> -> Complex<X>` 。

虽然这种写法在类型系统上完全成立，但在实际工程中并不常见。

## 2、AddAssign核心结构

以 `AddAssign` 为例，其定义如下：

```rust
trait AddAssign<Rhs = Self> {

}
```
需要注意，复合赋值表达式本身的值始终是 `()` ，而不是赋值后的结果。 `let z = (x += y);`，这里的 `z` 是 `()` 。

# 六、相等性比较

## 1、概述

Rust中的相等比较运算符： `==` 和`!=` 对应的trait是 `std::cmp::PartialEq` 和 `std::cmp::Eq` 。其中：

`std::cmp::PartialEq` 表示 **部分相等**

`std::cmp::Eq` 表示 **完全相等**

这是Rust中一个非常重要但也非常容易混淆的点。之所以要区分两者，是因为并不是所有类型都满足严格意义上的数学相等关系。

## 2、PartialEq

下面是 `std::cmp::ParialEq` 的定义：

```rust
trait PartialEq<Rhs = Self>

where

Rhs:?Sized,

{

fn eq(&self, other: &Rhs) -> bool;

fn ne(&self, other: &Rhs) -> bool {

!self.eq(other)

}

}
```
与 `Add` 这类 trait 不同， `PartialEq` 的参数是引用，原因是比较通常只是读取值，不应消耗所有权。

例如，比较两个 `String` ：
```rust
let a = String::from("hello");

let b = String::from("world");

println!("{}", a == b);
```
比较后， `a` 和 `b` 仍然可用。这正是按引用比较带来的好处。

下面是为自定义类型 `Complex` 的实现 `PartialEq` ：
```rust
impl<T> PartialEq for Complex<T>

where

T: PartialEq

{

fn eq(&self, other: &Self) -> bool {

self.re == other.re && self.im == other.im

}

}
```
两个复数相等，当且仅当实部相等且虚部相等。

实现了 `PartialEq` 特征，就可以使用 `assert_eq!`对两个 `Complex` 值进行断言了：
```rust
let c1 = Complex { re: 1, im: 2 };

let c2 = Complex { re: 2, im: 4 };

assert\_eq!(c1 + c2, Complex { re: 3, im: 6 });
```
**手动实现** `PartialEq` **比较枯燥，因为无非就是自定义类型的各个字段比较** 。由于相等性是一个常见的操作，可以使用派生属性宏为自定义类型自动实现 `PartialEq` 。

示例：
```rust
#\[derive(Debug, Copy, Clone, PartialEq)\]

struct Complex<T> {

re: T,

im: T,

}
```
这会让编译器自动按字段生成比较逻辑。
## 3、部分相等的由来

`PartialEq` 是部分相等的意思，而不是完全相等。

数学上的 **完全相等** 关系通常要求满足：

**对称性：** 如果 `x == y` ，则 `y == x`

**传递性：** 如果 `x == y` 且 `y == z` ，则 `x == z`

**自反性：** 任意 `x` 都应该满足 `x == x`

最后一个自反性看似显而易见不值一提，但却最容易出错。例如浮点数就不满足自反性。Rust 的 `f32` 和 `f64` 遵循 IEEE 浮点标准，其中存在一个特殊值： `NaN` 。

根据标准：

```rust
let x = 0.0f64 / 0.0;
```
得到的是 `NaN` ，并且 `x == x` 是 `false` ，所以浮点数不满足自反性。因此，浮点类型只能实现 `PartialEq` ，不能实现 `Eq` 。代码示例：
```rust
fn main() {

let x = 0.0f64 / 0.0;

assert!(x.is\_nan());

assert!(x!= x);

}

Rust 没有强行把所有“可比较”的类型都视作“完全相等”，而是把这种差异体现在 trait 层面。这种设计非常严谨，能避免泛型代码误以为“任何类型都满足 `x == x` ”。
```
## 4、Eq

`Eq` 的定义如下：

```rust
trait Eq: PartialEq<Self> {}

`Eq` 不增加新方法，它只是一个更强的语义承诺：如果一个类型实现了 `Eq` ，那么它的所有值都必须满足 `x == x` 。
```
**实际上，几乎所有实现了** `PartialEq` **的类型都实现了** `Eq` **，而** `f32` **和** `f64` **是标准库中仅有的两个属于** `PartialEq` **而不属于** `Eq` **的类型** 。

下面为自定义类型 `Complex` 实现 `Eq` ：
```rust
impl<T: Eq> Eq for Complex<T> {

}
```
一般情况下，通过派生属性宏自动实现更加简便：
```rust
#\[derive(Debug, Copy, Clone, PartialEq, Eq)\]

struct Complex<T> {

re: T,

im: T,

}
```
不过需要注意的是，只有当字段类型本身都满足 `Eq` 时，自定义类型才适合实现 `Eq` 。如果结构体包含 `f64` 字段 ，就不能安全地实现 `Eq` 。
# 七、有序性比较

## 1、概述

Rust的大小比较运算符 `<` 、 `>` 、 `<=` 、 `>=` 主要由 `std::cmp::PartialOrd` 决定。

如果一个类型不仅能比较，并且任意两个值之间都一定可以排出顺序，那么还可以进一步实现 `std::cmp::Ord` 。

这与前面的 `PartialEq` / `Eq` 是对应关系：

`PartialEq` 对应部分相等

`Eq` 对应完全相等

`PartialOrd` 对应部分有序

`Ord` 对应全序

## 2、PartialOrd

`PartialOrd` 特征定义如下：

```rust
trait PartialOrd<Rhs = Self>: PartialEq<Rhs>

where

Rhs:?Sized

{

fn partial\_cmp(&self, other: &Rhs) -> Option<Ordering>;

fn lt(&self, other: &Rhs) -> bool {... }

fn le(&self, other: &Rhs) -> bool {... }

fn gt(&self, other: &Rhs) -> bool {... }

fn ge(&self, other: &Rhs) -> bool {... }

}
```
说明：

`PartialOrd<Rhs = Self>` **扩展了** `PartialEq<Rhs>` **，只有可以比较相等性的类型才能比较有序性**

`PartialOrd` 中必须自行实现的唯一方法是 `partial_cmp` ，当 `partial_cmp` 返回 `Some(o)` 时， `o` 应该指出 `self` 与 `other` 的关系， `std::cmp::Ordering` 枚举定义如下：
```rust
enum Ordering {

Less, // self < other

Equal, // self == other

Greater, // self > other

}
```
如果 `partial_cmp` 返回 `None` ，意味着 `self` 和 `other` 相对于彼此是无序的，即两者都不大于对方，但也不相等。

返回 `Option<Ordering>` 的原因是并不是所有值都能比较出顺序。例如浮点数中的 `NaN` ，与任何值比较都是无序，因此必须能表达无法比较的情况，于是返回 `None` 。因此，此时的 `lt` 、 `le` 、 `gt` 和 `ge` 方法都返回 `false` 。

和其他二元运算符一样，如果要比较 `Left` 和 `Right` 这两种不同类型的值，那么 `Left` 就必须实现 `PartialOrd<Right>`

像 `x < y` 这样的表达式都是调用 `PartialOrd` 方法的简写，如下表所示：

## 3、为区间类型实现PartialOrd

考虑一个区间类型，表示左闭右开区间： `[lower, upper)`

```rust
#\[derive(Debug, PartialEq)\]

struct Interval<T> {

lower: T,

upper: T,

}
```
对于区间，顺序关系不是总能确定的：

`[10, 20)` 与 `[20, 40)` ：前者小于后者

`[30, 20)` 与 `[10, 20)` ：前者大于后者

`[10, 20)` 与 `[10, 20)` ：相等

`[10, 30)` 与 `[20, 40)` ：重叠，无法比较大小

这正是部分有序的典型例子。

现在为 `Interval<T>` 实现 `PartialOrd` ：
```rust
use std::cmp::Ordering;

#\[derive(Debug, PartialEq)\]

struct Interval<T> {

lower: T,

upper: T,

}

impl<T: PartialOrd> PartialOrd for Interval<T> {

fn partial\_cmp(&self, other: &Self) -> Option<Ordering> {

if self == other {

Some(Ordering::Equal)

} else if self.upper <= other.lower {

Some(Ordering::Less)

} else if self.lower >= other.upper {

Some(Ordering::Greater)

} else {

None

}

}

}

fn main() {

assert!(Interval { lower: 10, upper: 20 } == Interval { lower: 10, upper: 20 });

assert!(Interval { lower: 10, upper: 20 } < Interval { lower: 20, upper: 40 });

assert!(Interval { lower: 20, upper: 40 } > Interval { lower: 10, upper: 20 });

assert!(!(Interval { lower: 10, upper: 20 } < Interval { lower: 10, upper: 30 }));

}
```
## 4、Ord

如果知道两个值总能确定相对于彼此的顺序，那么就可以实现更为严格的 `std::cmp::Order` 特征：

```rust
trait Ord: Eq + PartialOrd<Self> {

fn cmp(&self, other: &Self) -> Ordering;

}
```
与 `partial_cmp` 不同， `cmp` 返回的不是 `Option<Ordering>` ，而是直接返回 `Ordering` 。这意味着： **任意两个值都必须能比较出结果** 。

注意： `Ord` **表示全序关系，它通常只定义在同一种类型之间的比较上，而** `PartialOrd` **更宽松，允许跨类型比较。所以** `Ord` **没有指定泛型参数** 。

**很多标准库排序接口都要求全序或需要用户显式提供比较逻辑** 。例如， `sort_by()` 方法需要传入一个闭包，这个闭包其实是一个比较器（comparator），它的作用不是直接排序，而是告诉 `sort_by` ：当拿到两个元素 `a` 和 `b` 时，它们谁应该排前面。sort\_by的闭包大致可以理解成这种形式：
```text
FnMut(&T, &T) -> Ordering
```
含义：

返回 `Less` ：表示a应该排在b前面

返回 `Equal` ：表示两者顺序无所谓

返回 `Greater` ：表示a应该排在b后面

示例：
```rust
let mut v = vec!\[10, 30, 20\];

v.sort\_by(|a, b| b.cmp(a) );

println!("{:?}", v);
```
正常的升序排序应该是 `a.cmp(b)` ，因为例如返回 `Less` ，a比b小，而a要排在b前面，即升序。但 `b.cmp(a)` 则是降序，因为例如返回 `Less` ，表示b比a小，a要排在b前面，即降序。
## 5、Reverse

`std::cmp::Reverse<T>` 是一个包装类型，用于把原本的比较顺序反过来。它最常见的用途就是把默认升序变成降序。

`Reverse` 是Rust标准库的一个包装类型，其作用很简单，就是将原本的比较顺序反过来。最常见用途是本来默认是升序，包一层Reverse之后就变成降序。

`Reverse` 的简化版本定义如下

```rust
struct Reverse<T>(T); // 包装类型

impl<T: PartialOrd> PartialOrd for Reverse<T> {

fn partial\_cmp(&self, other: &Reverse<T>) -> Option<Ordering> {

other.0.partial\_cmp(&self.0)

}

}

impl<T: Ord> Ord for Reverse<T> {

fn cmp(&self, other: &Reverse<T>) -> Ordering {

other.0.cmp(&self.0)

}

}
```
原先的 `T` 类型比较时左操作数是 `self` ，右操作数是 `other` ，但是Reverse<T>对调了顺序。

示例：
```rust
use std::cmp::Reverse;

fn main() {

let mut v = vec!\[40, 10, 30, 20, 50\];

v.sort\_by\_key(|x| Reverse(\*x));

println!("{:?}", v);

}
```
# 八、Index与IndexMut

## 1、概述

Rust支持类似数组那样的索引语法： `a[i]` 。如果要让自定义类型支持这种写法，血药实现：

`std::ops::Index`

`std::ops::IndexMut`

其中 `Index` 用于只读索引， `IndexMut` 用于可变索引。

对于大多数自定义类型来说，索引表达式并不是特殊魔法，本质上还是特征方法调用的简写：

（1）只读索引

```text
a\[i\]
```
通常等价于：
```text
\*a.index(i)
```
注意：

`index` 是 `std::ops::Index` 的方法，返回值是对某个值的 **共享引用**

前面的 `*` 号表示对返回的引用进行解引用

（2）可变索引
```text
a\[i\]
```
通常等价于：
```text
\*a.index\_mut(i)
```
注意：

`index_mut` 是 `std::ops::IndexMut` 的方法，返回值是某个值的可变借用
## 2、定义

`Index` 和 `IndexMut` 特征的定义如下：

```rust
trait Index<Idx> {

type Output:?Sized;

fn index(&self, index: Idx) -> &Self::Output;

}

trait IndexMut<Idx>: Index<Idx> {

fn index\_mut(&mut self, index: Idx) -> &mut Self::Output;

}
```
说明：

`Idx` 是泛型参数，表示索引参数类型，即索引表达式中括号里的类型。 索引参数不一定只是 `usize` ，例如切片既支持单个整数索引，也支持范围索引：

单个元素索引：切片实现了 `Index<usize>` ，索引可以写 `a[i]` 。

范围索引：切片实现了 `Index<Range<usize>>` ，索引可以写 `a[i..j]` 。

`Output` 是索引结果类型，表示索引访问得到的内容类型。例如切片用 `usize` 类型索引时， `Output` 是单个元素类型；切片用范围索引时， `Output` 得到的子切片类型。

`IndexMut` 继承自 `Index` ，表示如果一个类型支持可变索引，那么也必须先支持只读索引

示例：
```rust
let v = vec!\[1,2,3,4,5\];

let slice = &v\[..\];

let x = slice\[0\];

// 等价于let x = \*slice.index(0);

let y = &slice\[1..4\];

// 等价于let y = &\*slice.index(Range {start: 1, end: 4});

Rust的 `HashMap` 也支持索引，且它们的索引不一定是整数。只要键类型合适，就可以用键来索引。例如，下面的代码之所以能运行，是因为 `HashMap<&str, u32>` 实现了 `Index<&str>` ：
```
```rust
let mut scores = HashMap::new();

scores.insert("Tom", 90);

scores.insert("Bob", 80);

scores.insert("Helen", 60);

assert\_eq!(scores\["Tom"\], 90);

assert\_eq!(scores\["Bob"\], 80);

`IndexMut` 通过 `index_mut` 方法扩展了 `Index` ，它返回的是：
```
```text
&mut Self::Output
```
也就是返回索引结果的可变引用。当索引表达式出现在需要可变借用的上下文中，Rust会自动选择 `index_mut` 。

例如：
```rust
let mut v = vec!\["hello".to\_string(), "hi".to\_string()\];

v\[0\].push\_str(", world");

println!("{:?}", v);
```
等价写法：
```rust
use std::ops::IndexMut;

(\*v.index\_mut(0)).push\_str(", world");

`IndexMut` 的设计要求是必须返回某个现有值的可变引用。这会带来一个限制： **它不适合索引不存在时自动创建元素的场景** 。
```
例如 `HashMap` 就没有实现 `IndexMut` ，因为键可能不存在：
```text
scores\["Jack"\] = 70; // 错误

`index_mut` 必须返回一个已经存在的值的 `&mut` 引用，但如果 `"Jack"` 这个键不存在，则HashMap就先要创建一个默认值再返回该值的可变引用。但问题在于并非所有类型都有合适的默认值，即使有，创建出来立即被覆盖，也是浪费。
```
示例（按行索引的二维图像）
```rust
use std::ops::Index;

use std::ops::IndexMut;

struct Image<P> {

width: usize,

pixel: Vec<P>,

}

impl<P: Default + Clone> Image<P> {

fn new(width: usize, height: usize) -> Self {

Self {

width,

pixel: vec!\[P::default(); width \* height\],

}

}

}

impl<P> Index<usize> for Image<P> {

type Output = \[P\];

fn index(&self, index: usize) -> &Self::Output {

let start = index \* self.width;

&self.pixel\[start..start + self.width\]

}

}

impl<P> IndexMut<usize> for Image<P> {

fn index\_mut(&mut self, index: usize) -> &mut Self::Output {

let start = index \* self.width;

&mut self.pixel\[start..start + self.width\]

}

}
```
# 九、无法重载的运算符

并不是所有运算符都可以在Rust中重载。Rust对运算符重载是有边界的：只有一部分运算符对应标准库中的特征，可以通过特征来自定义行为；而另一部分运算符属于语言本身的固定机制，不可由用户自定义重载。

**（1）错误传播运算符**`?`

`?`将在错误处理中详细介绍，其仅适用于 `Result` 和 `Option` 。`?`**更接近于Rust语言内建的错误传播机制，而不是普通的运算符重载接口** 。

**（2）逻辑运算符** `&&` **和** `||`

`&&` 和 `||` 运算符只适用于 `bool` 值，不能重载。从行为上理解， `&&` 和 `||` 不只是计算两个值，还带有短路求值语义：

`a && b` ：如果a为false，则b不再求值

`a || b` ：如果a为true，则b不再求值

**（3）范围运算符**`..`**和**`..=`

`..`**和**`..=` 运算符也不能重载，它们的作用是直接构造表示边界范围的结构体值。例如 `a..b` 等价于构造结构体值 `Range {start: a, end: b}` 。

**（4）借用运算符** `&`

`&` 运算符始终表示借用一个引用，不能重载。这是Rust所有权和借用系统的基础语法，不允许用户改写其含义。

**（5）赋值运算符** `=`

`=` 运算符也不能重载，它始终表示对值进行赋值，在需要时进行移动或者在类型实现 `Copy` 时执行复制。

**（6）函数调用运算符** `()`

Rust不支持重载函数调用运算符，也就是不能像C++等语言一样，通过给普通类型实现 `()` 来让对象看起来像函数一样被调用。

如果需要一个可调用的值，通常直接闭包、函数项或者实现了 `Fn` 、 `FnMut` 、 `FnOnce` 相关调用能力的类型。

还有一类运算符虽然不在本节介绍的普通运算符列表里，但它们的行为可以通过特征 `Deref` 和 `DerefMut` 影响，那就是：

解引用运算符 `*val`

点运算符 `val.filed`

方法调用 `val.method()`

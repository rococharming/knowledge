## 所有权

## 一、Rust内存管理

所有编程语言都必须管理程序运行时使用的内存，常见路线有三类：

**垃圾回收（GC）** ：如Java、Python语言。 **运行时定期追踪不在可达的对象并回收** 。优点是开发体验轻松；缺点是存在运行时开销与暂停/延迟等问题（程序依实现而已）。

**手动管理：** 如C/C++。开发者负责malloc/free或new/delete。优点是控制能力强；缺点是容易产生内存泄漏、悬垂指针、二次释放、越界访问等安全问题。

**编译期所有权系统：** Rust。Rust通过 **所有权 + 借用+ 生命周期** 在编译阶段静态检查内存使用的合法性。只要编译能通过，Rust就在大范围内保证了内存安全。

简单来说，C/C++将内存正确性交给程序员自律，但程序员总有犯错的时候，不可避免地会产生内存安全问题；GC把它交给运行时，虽然无需手动管理，但降低了程序运行的性能。 **Rust则另辟蹊径，将它交给编译器规则系统，这样既兼顾性能又兼顾了运行时的安全** 。

## 二、作用域与所有权

## 1、变量的作用域

变量作用域（Scope）是变量在程序中有效的范围，通常由 **{}块** 界定。

示例：

展开

自动换行

fn main() {

{ // a 还未声明

let a = 1; // a 进入作用域

println!("a: {}", a); // 使用a

} // a离开作用域

}

## 2、Drop

Rust中资源管理遵循 **RAII** 思想： **当值离开作用域，会自动调用其析构逻辑（Rust术语叫drop，由Drop Trait参与）** 。

**对栈上的纯数据：** 离开作用域就是弹栈

**对拥有堆内存/系统资源的值：** 在drop时释放堆内存、关闭文件句柄、释放锁等。

## 3、所有权三大规则

Rust所有权规则可以概括为三条：

**每一个值（value）都有一个所有者（owner）**

**同一时刻，一个值有且仅有一个所有者（默认情形，共享需要特殊类型）**

**当所有者离开作用域，值会被丢弃（drop）**

这三条规则配合编译器检查，使得Rust能在不依赖GC的情况下避免大量内存错误。

Rust的 **String** 类型就是理解所有权的典型例子。String本质上是一个拥有 **堆缓冲区** 的类型，它在栈上保存三个字段：

指向堆缓冲区的指针ptr

当前长度len

当前容量cap

堆上存放的是 **UTF-8字节序列** 本体。

示例：

展开

自动换行

fn main() {

let s = String::from("Hello");

} // s离开作用域 -> drop -> 释放堆缓冲区

下面给出了内存示意图：

展开

自动换行

Stack (栈) Heap (堆)

┌──────────────────────────┐ ┌─────────────────────────────┐

│ s: String │ │ allocation for string bytes │

s存储在栈上，包含ptr、len、cap三个字段，其中ptr指向了堆区的实际字符串数据。

当s离开作用域，除了栈上这三个字段销毁，同时会触发drop，自动释放堆区内存。这里就很好诠释了当所有者离开作用域，值被丢弃这一条规则。

Rust的所有权规则很严格，但语言和标准库提供了几种“受控的放宽方式”，用于表达更复杂的内存与共享场景：

**移动（Move）：值的所有权可以从一个绑定转移到另一个绑定，移动后原绑定不再可用，从而避免重复释放等问题**

**复制（Copy类型）** ：对实现了 **Copy的类型** （如整型、浮点型、布尔型、字符型）， **赋值/传参会发生按位复制** ，原变量仍然可用。这遵守所有权模型，只是复制后两个值彼此独立，无需资源释放协调。

**借用（borrow)/引用（reference）：通过&T或&mut T临时访问一个值而不取得其所有权** 。引用本身不负责释放值，值何时释放仍由其所有者决定，且引用必须满足借用规则与生命周期约束。

**共享所有权（Rc/Arc）** ：标准库提供Rc<T>（单线程）和Arc<T>（多线程）实现引用计数， **在一定限制下允许多个所有者共享同一份数据** 。若需要在共享下修改数据，通常还需要配合Mutex等机制。

## 三、移动

## 1、移动的基本概念

在Rust中，对于 **大多数非Copy类型** 来说， **给变量赋值、作为参数传入函数或从函数返回** ， **默认发生的是移动（Move）而不是复制** 。

**移动会将值的所有权从源绑定转移到目标绑定** 。移动之后，源绑定在语义上被视为 **已被移动而不可再使用** （编译器会禁止继续使用它），由目标绑定负责该值的生命周期并在适当时机触发drop。

Rust程序通常通过这种所有权转移来构建、重组和拆解包含堆资源的复杂数据结构，同时避免重复释放和悬垂访问等问题。

这里仍然以String类型为例，来说明移动的过程。

示例：

展开

自动换行

fn main(){

let s1 = String::from("hello");

let s2 = s1; // Move

// println!("s1: {}", s1); // 编译报错，s1已经失效

println!("s2: {}", s2);

}

## 2、Copy类型：移动的例外

对于像整数这类实现了Copy特征的类型，赋值不会发生所有权移动，而是按位复制一份值，因此赋值后原变量仍然可用。

可以比较String赋值和i32赋值内存的差异：

示例：

展开

自动换行

let string1 = "somnambulance".to\_string();

let string2 = string1;

let num1: i32 = 36;

let num2 = num1;

运行这段代码后，内存示意如下：

![[Image 36.png]]

对于String，赋值会将string1转移给string2，同时string1无效。但对于i32，它只是内存中的几个字节，不拥有堆区资源，因此赋值时num2是与num1独立的副本，num1仍然有效。

标准的Copy类型包括：

各种标量类型（整型、浮点型、字符型和布尔型）

元素都是Copy类型的数组

元素都是Copy类型的元组

其他类型（例如不可变引用类型）

对于String，因为它拥有堆区分配的缓冲区，因此不能是Copy类型；对于Box<T>（智能指针，后续介绍），因此它拥有指向堆区分配的目标，因此不能是Copy；对于代表操作系统文件句柄的File，因为复制这样的值需要向操作系统申请另一个文件句柄，因此不能是Copy；对于MutexGurad互斥锁类型，因为同一时间只能有一个线程持有互斥锁，因此不能是Copy类型...。

简单来说，任何在丢弃值时需要做一些特殊操作的类型都不能是Copy类型：String需要释放堆区内存、Vec需要释放自身元素、File需要关闭自身文件描述符、MutexGuard需要释放自身互斥锁等。

**此外，后续会介绍的结构体struct和枚举enum默认也不是Copy类型** 。如果他们的字段类型都是Copy的，可以通过#\[derive(Copy, Clone)\]使得结构体或枚举成为Copy类型。

## 3、更多移动操作

前面的例子展示了 **变量赋值** 会发生移动，这只是一种情况。Rust将移动语义应用到了几乎所有对值的使用上，例如将参数传递给函数、函数返回值、构建元组将值转移给元组等。

### （1）函数传参

当实参传递给函数的形参时，对于非Copy类型会发生所有权的转移。

示例：

展开

自动换行

fn main() {

let s = String::from("Hello"); // s进入作用域

take\_ownership(s); // s移动进了take\_ownership函数，此时s失效

}

fn take\_ownership(some\_string: String) {

println!("some\_string: {}", some\_string);

}

上述例子中，值的所有权从s移动到了函数中的some\_string，其实就是变量的赋值。

### （2）函数返回值

当函数返回时，对于非Copy类型的返回值，所有权也会转移给调用者。

示例：

展开

自动换行

fn main() {

let s1 = gives\_ownership(); // s1获得所有权

let s2 = String::from("Hello"); // s2获得所有权

let s3 = takes\_and\_gives\_ownership(s2); // s2移动，失效

// s3获得了s2的所有权

}

fn gives\_ownership() -> String {

let s = String::from("Hello");

s // s的所有权转移给调用者

}

fn takes\_and\_gives\_ownership(s: String) -> String {

s // s的所有权转移给调用者

}

## 4、移动与控制流

前面的例子大多都是直线式的控制流，但在真实代码里，移动往往会和条件分支（if/else）和循环等控制流交织在一起。

在if/else中，如果一个变量在条件判断后仍然确定有值，那么可以在两个分支中分别将它移动出去：

展开

自动换行

let s = String::from("hello");

if flag {

f(s); // 可以在这里移动s

} else {

g(s); // 也可以在这里移动s

}

// h(s); // 错误，s变为未初始化状态，不可继续使用

但注意的是， **一旦某条路径移动了s，那么在if/else之后，编译器就会把s视为未初始化的状态，因此不能再使用** 。

出于类似原因， **Rust不允许在循环结构中直接移动同一个变量** 。因为在第一次迭代已经移动了变量，第二次迭代时该变量就失效了。

示例：

展开

自动换行

let s = String::from("hello");

while true {

f(s); // 错误

}

## 5、从复合类型中移动值

移动（move）会把值的所有权从源绑定转移到目标绑定。移动之后，源绑定在语义上被视为已被移动，因此不能再被使用。

但当源是某个复合类型的局部位置（例如向量Vec<T>的某个索引位置或结构体字段）时，若直接把值移动走，会在原位置上留下一个空洞。Rust禁止这种从索引表达式直接移动非Copy值的行为。

示例：

展开

自动换行

let v = vec!\[

String::from("hello"),

String::from("world"),

String::from("rust")

\];

let first = v\[0\]; // 错误

let third = v\[2\]; // 错误

编译报错：

![[Image 37.png]]

报错结果就说明了不能从Vec<String>中移动某个索引值，因为String没有实现Copy类型。如果Vec存放的是Copy类型，才不会报错。

对于Vec<T>类型，如果确实想要把某个索引处的值移动走，通常采用三种方法：

从末尾弹出（只能移动末尾的值，向量缩短）：

将目标位置与末尾交换，再弹出目标（向量缩短，不保证顺序）

用替代值填充原槽位，将原值移动出来（向量不变）

展开

自动换行

let mut v = vec!\[

String::from("hello"),

String::from("world"),

String::from("rust"),

String::from("C++")

\];

// 弹出末尾的值

let last = v.pop().expect("vector is empty!");

println!("last element: {}", last);

// 交换目标位置和末尾的值，在弹出

let second = v.swap\_remove(1);

println!("second element: {}", second);

// 替换

let first = std::mem::replace(&mut v\[0\], String::from("hi"));

println!("first element: {}", first);

上述每种方法都可以将元素移出向量，但仍然使得向量出于完全填充状态。

像Vec这样的集合类型通常也提供在循环中效果所有元素的方法：

展开

自动换行

let v = vec!\[

String::from("hello"),

String::from("world"),

String::from("rust"),

String::from("C++")

\];

for s in v {

println!("{}", s);

}

// 此时v已经被移动

for s in v的底层实现会将向量的所有权从v移动进循环中，因此之后v失效。

## 四、共享所有权

## 1、Rc和Arc

在典型Rust代码中，大多数值都有唯一的所有者：谁拥有，谁负责在离开作用域时drop释放资源。 **但有些场景下，很难为某个值找到足够长寿命的单一所有者，更希望该值存活到所有使用者都不再需要它为止** 。

为了解决这种多个所有者共同决定值的生命周期的需求，Rust标准库提供了两类引用计数指针：

**Rc<T>：** 引用计数（Reference counted）：用于单线程共享所有权

**Arc<T>：** 原子引用计数（Atomic reference counted）：用于多线程共享所有权

它们都遵循Rust的安全规则：引用计数的增减由类型本身保证正确。 **当最后所有者离开作用域时，值才会丢弃** 。

**Rc和Arc的核心差异只有一个：线程安全性** 。Arc使用原子操作维护计数，因此可以安全跨线程共享；Rc使用更轻量的非原子计数，因此更快但不能跨线程。只要不需要跨线程共享，则优先使用Rc，避免为Arc的原子操作开销买单。如果在多线程中误将Rc发送到其他线程，Rust编译器会报错。

## 2、使用方式

**以Rc为例** 说明共享所有权的使用方式，Arc的API与Rc完全一样。

**对于任意类型T，Rc<T>表示指向堆上T类型值的指针，并且堆内存块中还包含一个引用计数** 。 **克隆一个Rc<T>，并不会克隆T值本身，而是复制一个新的Rc指针同时将堆区的引用计数加1** （这里的clone语言有别于像String类型的clone，它不是深拷贝，同时也不是简单的浅拷贝，因为会更新堆区引用计数）。

示例：

展开

自动换行

use std::rc::Rc;

let s: Rc<String> = Rc::new("hello".to\_string());

let t: Rc<String> = Rc::clone(&s); // 等价于 s.clone()，只增加引用计数

let u: Rc<String> = Rc::clone(&s);

此时，s、t、u三个Rc<String>指针指向同一块内存，其中既有引用计数，也有String本体（三个字段）。

Rc指针本身仍然遵循常规所有权规则，当s、t、u陆续离开作用域时，引用计数递减。当最后一个Rc指针被丢弃时，才触发释放，并递归drop内部的String（String字段本身存储在堆区， String的ptr指向的字符数据也存储在堆区）。

Rc<String>还提供了类似 **共享借用** 的使用体验，因为它实现了Deref特征，因此可以像使用String引用一样直接调用方法：

展开

自动换行

assert!(s.contain("he"));

assert\_eq!(s.find("he"), Some(0));

println!("{}", s);

值的注意的是：Rc提供的是共享所有权，但默认不提供共享可变性。也就是说，多个所有者共享同一份数据时，不允许直接通过Rc<T>获得&mut T去修改它，这会破坏借用规则。

如果业务确实需要“多个所有者 + 可变”，通常需要配合内部可变性或同步原语获得受控的可变访问。

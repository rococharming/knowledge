## 11、错误处理

## 一、Rust错误处理分类

Rust的错误处理机制比较有特色，它将错误明确分成两大类： **可恢复错误（recoverable errors）和不可恢复错误（unrecoverable errors）** ，对应的错误处理机制分别是 `Result` 和 `panic!`。

这种分类方式让程序员能够根据错误的性质选择不同的处理策略，从而写出更安全、更清晰的程序。

## 1、可恢复错误

**可恢复错误是指：错误发生后，程序仍有机会采取补救措施，或将错误传递给上层继续处理，而不是立刻终止程序** 。这类错误通常 **来源于程序外部环境** ，而不是程序本身的逻辑缺陷。

常见场景：

打开文件时文件不存在

读取配置文件失败

网络请求超时

用户输入非法

没有足够的权限执行某项操作

例如，打开文件失败不一定说明程序写错了，也可能只是文件暂时不存在。此时程序可以通过 `Result` **捕获该错误** ，并根据需要选择提示用户、创建文件后重试、或将错误继续向上传递。

## 2、不可恢复错误

**不可恢复错误是指：程序遇到了严重问题，已经无法安全、合理地继续运行，因此必须终止程序** 。这类错误通常反映的是 **程序内部逻辑问题、前提条件被破坏，或者系统关键流程出错** 。

常见场景：

数组访问越界

核心数据结构被破坏

启动流程中关键步骤失败

例如，访问不存在的数组下标通常说明代码逻辑存在问题，此时继续运行程序已经没有意义，甚至可能掩盖更严重的错误，对于这类情况，Rust使用 `panic!`来处理这类错误。发生 `panic` 时，程序会立即停止执行，并输出相应的错误信息。

## 二、panic与不可恢复错误

在Rust中， `panic` 是程序崩溃的术语。 `panic!`宏用于处理不可恢复错误。当 `panic` 发生时，程序不会继续按正常流程执行，而是进入异常终止流程。

需要注意的是， `panic!`是一个宏，可以像 `println!`一样接收参数，用于构造错误信息。

## 1、panic的触发方式

在Rust中， `panic` 的触发方式分为 **被动触发** 和 **主动触发** 。

### （1）被动触发

**被动触发是指程序运行到某些非法或不合理的状态时，由Rust主动触发了** `panic` 。常见情况包括：

数组越界访问

整数除以0

在值为 `Err` 的 `Result` 上调用 `unwrap()` 或 `expect()`

断言失败（ `assert!`、 `assert_eq!`）

示例：

自动换行

fn main() {

let v = vec!\[1, 2, 3\];

v\[99\];

}

这类 `panic` 往往说明程序中存在bug。Rust选择在运行时立即将错误暴露出来，而不是让程序继续在错误状态下运行。

### （2）主动触发

Rust提供了 `panic!`宏，程序员可以在检测到某种严重错误时主动触发 `panic` 。

示例：

## 2、panic发生的程序行为

当 `panic` 发生时， **Rust的默认处理方式是展开调用栈（unwind）** 。整个过程通常包括以下几个步骤：

打印错误信息，指出 `panic` 的原因以及发生位置

如果启用回溯信息，还会输出调用栈

从当前函数开始沿调用栈逐层退出

在退出过程中，按创建顺序的逆序清理局部变量、参数和临时值。执行对应的 `drop` 方法，释放资源

最终结束当前线程：如果发生 `panic` 的是主线程，则整个程序退出

**因此** `panic` **并不等同于程序瞬间崩溃，而是一个有序的终止过程** 。

程序 `panic` 的行为有两种，除了默认的展开调用栈（unwind），还有直接终止（abort）。

## 3、backtrace：调用栈展开信息

Rust **默认采用展开调用栈** 的方式处理 `panic` 。 **所谓展开调用栈，就是程序从发生** `panic` **的位置开始，沿着函数调用链一层层返回，并在返回过程中清理当前作用域中的资源：**

释放字符串、向量等堆区内存

关闭已经打开的文件

执行用户自定义的 `drop` 方法

这意味着 `panic` 虽然会终止当前执行流程，但它仍然是安全的。它不会破坏Rust的内存安全规则，也不会留下悬空指针或半初始化状态的数据。

换言之， `panic` 不是未定义行为，而是一种行为明确、受控制的异常终止机制。

**需要注意的是，之前是在IDE（如RustRover）中直接运行程序的，这些IDE默认应该在cargo运行时设置了** `RUST_BACKTRACE` 。

因此，在命令行执行 `cargo run` 时，设置 `RUST_BACKTRACE=1` 来查看调用栈回溯信息：

**调用栈信息会显示函数调用的路径** ， **最近一次调用的函数排在列表的最上方，较早的调用排在了列表下方** 。例如 `main` 函数一般是最早调用的函数，所以排在接近最下面。从下往上看：

最底层的 `core::ops::function::FnOnce::call_once` 是运行时调用 `main` 的入口包装层，简单理解为Rust将 `main` 当作一个函数对象执行。

`hello_cargo::main at ./src/main.rs:5:6` 这一层就是我们写的代码，说明当前 `panic` 是从 `hello_cargo` 这个crate的 `main` 函数里触发的，具体位置在`./src/main.rs:5:6` 。 **这通常是排查问题是最该关注的一层，因为这是用户层代码，而不是标准库内部。**

`<alloc::vec::Vec<T,A> as core::ops::index::Index<I>>::index` 这一层说明对 `Vec` 执行了索引操作，对应代码里就是 `v[99]` 。

`core::slice::index::<impl core::ops::index::Index<I> for [T]>::index` ， `Vec` 的底层可以看作一个切片 `slice` ，所以继续往下转到切片的索引实现。

`<usize as core::slice::index::SliceIndex<[T]>>::index` 说明传入的索引类型是 `usize` ，这是更底层的按某个下标访问切片的实现。

`core::panicking::panic_bounds_check` 这一层非常关键，它表示标准库在做 **边界检查时发现越界** ，于是决定出发 `panic` 。也就是说，真正检测出错误的地方在这里：索引超出了合法的范围。

`core::panicking::panic_fmt` 这一层可以理解为Rust将 `panic` 信息格式化，准备输出错误内容。例如 `index out of bounds: the len is 3 but the index is 99` 就是这一阶段构造的。

`__rustc::rust_begin_unwind` 这是panic处理流程的更底层入口，它表示程序正式进入了 **unwind** （栈展开）阶段。也就是打印panic信息、沿调用栈逐层输出、清理局部变量，最终结束当前线程。

需要注意的是， **调用栈信息的具体内容会因操作系统、Rust版本和构建方式不同而略有差异** 。

调用栈信息对于定位问题非常重要，尤其是在错误涉及多层函数调用或第三方库时。

## 4、panic的另一种方式：直接终止

虽然展开调用栈是默认行为，但是Rust并不是所有情况下都可以这样做。两种典型情况：

第一种就是在展开过程中再次 `panic` ：如果程序正在处理第一次 `panic` ，并在清理某个值的 `drop` 过程中又触发了第二次 `panic` ，此时Rust会停止展开并直接终止整个进程。这个过程称为 **双重panic** ，属于致命错误。

还有一种就是编译时配置 `panic = 'abort'` 。这是Rust支持的另一种 `panic` 处理方式： **直接中止** （abort）。

**这种方式下，一旦发生** `panic` **，程序会立即终止，不再进行调用栈展开，也不会执行沿途的清理逻辑。善后工作交由操作系统负责** 。

可以在编译时指定这种行为，例如在 `Cargo.toml` 中配置：

展开

自动换行

\[profile.release\]

panic = 'abort'

这样在 `release` 构建中，程序发生 `panic` 时会直接终止。

**使用** `abort` **的好处是可执行文件体积更小，不需要为栈展开保留额外支持代码** ，但代价是 **不能获得完整的栈展开清理过程** 。

## 5、线程级别的panic

`panic` 是基于线程的。 **当某个线程发生** `panic` **时，默认只会终止当前线程；其他线程仍然可以继续运行。但如果发生** `panic` **的是主线程，那么整个进程通常会退出** 。  
  
这说明 `panic` 并不一定意味着整个程序立刻彻底结束，而是首先影响发生错误的那个线程。

## 6、捕获panic

虽然通常不建议把 `panic` 当作常规错误处理手段，但Rust也提高了在特定场景下捕获 `panic` 的能力。标准库中的：

展开

自动换行

std::panic::catch\_unwind()

可以捕获那些以栈展开方式传播的 `panic` ，从而避免线程直接终止。

这类机制主要用于测试框架中处理断言失败、某些需要隔离故障的场景、与其他语言交互时对 `panic` 进行边界控制。

不过要注意， `catch_unwind()` 只能捕获 **展开调用栈型** 的 `panic` ，不能捕获所有 `panic` 。

## 三、Result与可恢复错误

## 1、Result

Rust没有传统意义上的异常机制。对于执行过程中可能发生、但程序仍有机会处理的错误，Rust通常用 `Result<T,E>` 表示。 `Result` 是一个枚举类型：

展开

自动换行

enum Result<T, E> {

Ok(T),

Err(E),

}

其中：

`Ok(T)` 表示操作成功，内部保存成功值

`Err(E)` 表示操作失败，内部保存错误值

例如，打开文件就可能成功，也可能失败：

展开

自动换行

use std::fs::File;

let res = File::open("hello.txt");

`File::open()` 的返回类型是 `Result<File,std::io::Error>` ：

如果打开成功，返回 `Ok(File)`

如果打开失败，返回 `Err(std::io::Error)`

这类错误通常来自外部环境，例如文件不存在、权限不足、网络中断、输入不合法等，因此属于可恢复错误。

## 2、使用match处理Result

处理 `Result` 最基本、最完整的方式是使用 `match` ：

展开

自动换行

use std::fs::File;

fn main() {

let file = match File::open("hello.txt") {

Ok(f) => f,

Err(e) => {

panic!("打开文件失败：{}", e);

}

};

println!("{:?}", file);

}

结果：

![[Image 60.png]]

这种写法的优点是清晰、完整，适合在需要区分不同分支行为时使用。

如果希望针对不同错误做不同处理，还可以继续匹配错误类型。例如，如果是文件不存在的错误，可以先创建文件；如果是其他错误再出发 `panic` ：

展开

自动换行

use std::fs::File;

use std::io::ErrorKind;

fn main() {

let file = match File::open("hello.txt") {

Ok(f) => f,

Err(e) => match e.kind() {

ErrorKind::NotFound => match File::create("hello.txt") {

Ok(fc) => fc,

Err(e) => panic!("创建文件失败: {}", e),

},

\_ => panic!("创建文件失败: {}", e),

}

};

println!("{:?}", file);

}

这说明：对于不可恢复错误，程序不一定要立刻 `panic` ，而是根据情况采取补救措施。

## 3、Result的常用处理方式

### （1）直接匹配处理

如果希望当场处理错误，不向上传播时，最常见的方式仍然是 `match` ：

展开

自动换行

fn divide(a: i32, b: i32) -> Result<i32, String> {

if b == 0 {

Err("除数不能为0".to\_string())

} else {

Ok(a / b)

}

}

fn main() {

match divide(10, 2) {

Ok(v) => println!("结果：{}", v),

Err(e) => println!("错误：{}", e),

}

}

### （2）unwrap和expect

#### 1）unwrap

`unwrap()` 的行为很直接：

如果是 `Ok(v)` ，返回内部的成功值

如果是 `Err(e)` ，直接触发 `panic`

示例：

展开

自动换行

use std::fs::File;

fn main() {

let file = File::open("hello.txt").unwrap();

println!("{:?}", file);

}

如果文件不存在，就会因为在 `Err` 上调用 `unwrap()` 而发生 `panic` 。

#### 2）expect

`expect()` 和 `unwrap()` 类似，不同之处在于它允许自定义报错信息。

示例：

展开

自动换行

use std::fs::File;

fn main() {

#### 3）适用场景

`unwrap()` 和 `expect()` 虽然简单，但不能滥用，因为很容易导致程序直接结束。它们常见的使用场景有：

示例代码、实验代码、测试代码

非常确定这里不可能不错

错误发生时，直接终止程序是合理选择

例如：

展开

自动换行

use std::net::IpAddr;

fn main() {

let home: IpAddr = "127.0.0.1".parse().unwrap();

println!("{:?}",home);

}

这里的字符串是固定合法的IP地址，因此直接 `unwrap()` 是合理的。但如果输入来自用户，就不应该写，因为用户的输入可能出错，此时应进行正常错误处理。

### （3）常用辅助方法

`Result<T,E>` 还提供了其他便捷方法，用于简化一些常见场景。

`is_ok()` 和 `is_err()` ：用于判断结果是否成功或失败

`ok()` ：将 `Result<T, E>` 转位 `Option<T>` ，保留成功值，丢弃错误值

`err()` ：将 `Result<T, E>` 转位 `Option<E>` ，保留错误值，丢弃成功值

`unwrap_or(default)` ：成功时返回内部值，失败时返回默认值。

示例：

展开

自动换行

let res: Result<i32, &str> = Err("error");

let n = res.unwrap\_or(0);

println!("{}", n); // 0

`unwrap_or_else(f)` ：和 `unwrap_or()` 类似，但默认值通过 **闭包** 动态生成，只有在失败时才调用。它针对出现 `Ok` 的概率很大且计算回退值比较浪费耗时的情况。

示例：

展开

自动换行

let res: Result<i32, &str> = Err("error");

let n = res.unwrap\_or\_else(|\_| 0);

println!("{}", n); // 0

闭包是一种匿名函数，后续会详细介绍。

`as_ref()` ：将 `Result<T, E>` 转换为 `Result<&T, &E>`

`as_mut()` ：将 `Result<T, E>` 转换为 `Result<&mut T, &mut E>`

`as_ref()` 和 `as_mut()` 方法之所以有用，是因为前面介绍的除了 `is_ok()` 和 `is_err()` 以外的方法，都会消耗 `result` 。也就是说，它们接受按值 `self` 参数。然而，有时候不想消耗result，此时 `as_ref()` 和 `as_mut()` 方法就派上用场了。例如，假设想调用 `result.ok()` ，但还想让 `result` 存在，此时可以调用 `result.as_ref().ok()` ，结果返回的就是 `Option<&T>` 而不是 `Option<T>` 了。

### （4）忽略错误

有时只想完全忽略一个错误，不想做任何处理，如果这样写：

展开

自动换行

File::open("hello.txt");

则编译器编译时，会给出一个警告：

提示 `Result` 值必须使用。如果确定这里的错误对整个程序运行并无太大影响，那么可以使用 `let _ = ...`方式忽略该错误：

展开

自动换行

## 4、Result类型别名

在标准库和实际项目中，常常会给 `Result` 定义 **类型别名** ，以减少重复书写。

例如，标准库的 `std::io` 模块就有类似写法：

展开

自动换行

pub type Result<T> = result::Result<T, Error>;

这里定义了一个公共类型 `std::io::Result<T>` ，它是 `std::result::Result<T, E>` 的别名，但是将错误类型硬编码为了 `std::io::Error` 。因此，只要在当前crate中写下 `use std::io` ，那么Rust就会将 `io::Result<String>` 当作 `Result<String, io::Error` 的简写形式。

## 5、打印错误

**有时处理错误的一种方式是将其输出到终端并继续执行** 。

例如：

展开

自动换行

println!("error: {}", err);

这表示将错误值格式化成可读的文本并打印出来。

Rust标准库中许多常见错误类型如 `std::io::Error` 、 `std::fmt::Error` 、 `std::str::Utf8Error` 等都可以直接打印。这些错误都实现了 `std::error::Error` 特征。 `std::error::Error` 可看作是 **Rust错误类型的公共接口** ，它将 **各种错误统一到一个抽象层次** 下，方便用相似方法处理它们。

这意味着这些错误类型：

可以被显示为文本

可以提供更底层的错误来源

可以作为通用错误类型来传递和组织

因此，很多时候，不需要关注 `err` 的具体类型，也能先把它打印出来。

打印错误时，最常见的两种格式是：

展开

自动换行

println!("error: {}", err);

println!("error: {:?}", err);

它们的区别在于输出目标不同：

`{}` 是面向用户的简短信息，它使用的是 `Display` 格式，通常输出一条较短、较自然的错误信息。

`{:?}` 是面向调试的详细信息，它使用的是 `Debug` 格式，通常会输出更完整的内部结构，更适合开发时调试。

例如：

展开

自动换行

match File::open("hello.txt") {

Ok(f ) => println!("file: {:?}", f),

Err(e) => {

println!("{}", e);

println!("{:?}", e)

},

};

结果：

![[Image 64.png]]

如果不想直接打印，而是希望先把错误信息保存成字符串，可以使用 `to_string()` 方法：

展开

自动换行

let msg = err.to\_string();

它会返回一个 `String` 。

通常来说， `to_string()` 只能得到当前这一层错误的文字描述，不会把底层错误链一起展开打印。

很多错误不是单层的，而是由更底层的错误引起的。例如，读取配置文件失败，原因可能是文件不存在，而文件不存在又可能是路径错误或权限问题。为了表达 `一个错误是由另一个错误导致` 的关系， `Error` 特征提供了 `source()` 方法，它的返回值是 `Option<&(dyn Error + 'static)>` 。

也就是说：

如果当前错误还有更底层的原因，就返回 `Some(...)`

如果当前错误已经是最底层的错误，则返回 `None`

示例：

展开

自动换行

if let Some(source) = err.source() {

println!("caused by: {}", source);

}

如果想打印完整的错误链，一种较为通用的写法是：

展开

自动换行

use std::error::Error;

use std::fs::File;

use std::io::{self, Write};

fn print\_error(mut err: &dyn Error) {

let \_ = writeln!(io::stderr(), "error: {}", err);

while let Some(cause) = err.source() {

let \_ = writeln!(io::stderr(), "caused by: {}", cause);

err = cause;

}

}

fn main() {

match File::open("hello.txt") {

Ok(f ) => println!("file: {:?}", f),

Err(e) => {

print\_error(&e);

},

};

}

## 6、传播错误

当函数本身也返回 `Result` 时，很多时候不想在当前层处理错误，而是希望把错误继续交给调用者处理。这时就可以使用 `?` 运算符。可以在任何产生 `Result` 的表达式后面加上一个`?`，例如加在函数调用后面：

展开

自动换行

fn read\_username\_from\_file() -> Result<String> {

let mut file = File::open("hello.txt")?;

let mut s = String::new();

let \_ = file.read\_to\_string(&mut s)?;

Ok(s)

}

`?`的行为是：

如果结果是 `Ok(T)` ，就取出其中的值继续执行

如果结果是 `Err(E)` ，就立即返回该错误

因此，`?`本质上是错误传播的快捷写法，它有点类似于更冗长的 `match` 逻辑：

展开

自动换行

let mut file = match File::open("hello.txt") {

Ok(f) => f,

Err(e) => return Err(e),

};

但与 `match` 的区别点在于，`?`运算符不只是提前返回错误，在还会在需要时自动进行错误逻辑转换，其背后依赖的是 `From` 特征。

`?`支持链式使用：

展开

自动换行

fn read\_username\_from\_file() -> Result<String> {

let mut s = String::new();

let \_ = File::open("hello.txt")?.read\_to\_string(&mut s)?;

Ok(s)

}

在早期Rust中，常使用 `try!`传播错误，但现在已经使用`?`运算符替代：

展开

自动换行

let f = try!(File::open("hello.txt"));

`?`运算符除了可以作用于 `Result` ，也可作用于 `Option` 。对于 `Option` 而言：

如果是 `Some(T)` ，则取出值继续

如果是 `None` ，则提前返回 `None`

示例1：

展开

自动换行

fn first\_elem(arr: &\[i32\]) -> Option<&i32> {

let elem = arr.get(0)?;

Some(elem)

}

示例2:

展开

自动换行

fn last\_char\_of\_first\_line(text: &str) -> Option<char> {

text.lines().next()?.chars().last()

}

因此，`?`的核心作用可以统一理解为：在失败时提前返回。只是对 `Result` 来说返回的是 `Err` ，对 `Option` 来说返回的是 `None` 。

## 7、处理多种类型的错误

实际开发中，一个函数往往不止会遇到一种错误。例如，从文本中读取整数时，既可能发生IO错误，也可能发生解析错误：

展开

自动换行

use std::io::{self, BufRead};

fn read\_numbers(file: &mut dyn BufRead) -> Result<Vec<i64>, io::Error> {

let mut numbers = Vec::new();

for line\_res in file.lines() {

let line = line\_res?; // 读取各行可能失败

numbers.push(line.parse()?); // 解析整数可能失败

}

Ok(numbers)

}

编译报错：

![[Image 65.png]]

`?`运算符不能将 `std::num::ParseIntError` 转换为 `std::io::Error` 类型。 `line_res` 的类型是 `Result<String, std::io::Error>` ，而 `line.parse()` 的类型是 `Result<i64, std::num::ParseIntError` 。同一个函数会生成两种潜在的错误类型。

如果函数返回的错误类型过于具体，`?`可能无法自动传播所有错误。

有几种方法可以解决这个问题：

第一种方法是自定义自己的错误类型 `MyError` ，并实现从 `std：：io::Error` 和 `std::num::ParseIntError` 到 `MyError` 的转换。可以通过 `thiserror` crate完成。

第二种方法是使用Rust内置的特征 `std::error::Error` 。因为所有的错误类型都实现了该特好着呢个，所以标准库中的错误类型都可以转换为 `Box<dyn std::error::Error + Send + Sync + 'static>` 。 `dyn std::error::Error` 表示任何错误， `Send + Sync + 'static` 表示可以安全地在线程间传递。为了便于使用，可以定义类型别名：

展开

自动换行

type GenericError = Box<dyn std::error::Error + Send + Sync + 'static>;

type GenericResult<T> = Result<T, GenericError>;

此外还可以考虑使用 `anyhow` crate，它提供的错误类型和结果类型与 `GenericError` 和 `GenericResult<T>` 非常相似，但有一些不错的附加特性。

将 `read_numbers()` 的返回类型改为\`GenericResult<Vec<i64>>，如下：

展开

自动换行

use std::error::Error;

use std::io::{self, BufRead};

type GenericError = Box<dyn Error + Send + Sync + 'static>;

type GenericResult<T> = Result<T, GenericError>;

fn read\_numbers(file: &mut dyn BufRead) -> GenericResult<Vec<i64>> {

let mut numbers = Vec::new();

for line\_res in file.lines() {

let line = line\_res?; // 读取各行可能失败

numbers.push(line.parse()?)

}

Ok(numbers)

}

这样，`?`**运算符会根据需要自动将任意类型的错误转换为** `GenericError` 。本质上，是因为标准库为 `Box<dyn std::error::Error + Send + Sync 'static>` 实现了 `From` 特征，用于把任意实现了 `std::error::Error` 的类型转换为 `Box<dyn std::error::Error + Send + Sync 'static>` 。类似如下：

展开

自动换行

impl<E> From<E> for Box<dyn std::error::Error + Send + Sync 'static>

where

E: std::error::Error + Send + Sync 'static

{

fn from(err: E) -> Self {

Box::new(err)

}

}

## 8、main中的错误处理

### （1）直接使用expect

在较小程序中， `main` 一般返回 `()` ，如果产生 `Result` 错误，最简单的方式往往是直接在 `main` 中使用 `expect()` ：

展开

自动换行

use std::fs;

fn main() {

let context = fs::read\_to\_string("hello.txt").expect("读取文件失败");

println!("{}", content);

}

### （2）让main返回Result

更常见、更优雅的方式是让 `main` 直接返回 `Result` ，这样在 `main` 中就可以使用`?`了：

展开

自动换行

use std::error::Error;

use std::fs;

fn main() -> Result<(), Box<dyn Error>> {

let context = fs::read\_to\_string("hello.txt")?;

println!("{context}");

Ok(())

}

这种写法通常比 `expect()` 更整洁，也更适合稍复杂的程序。

### （3）手动打印错误并退出

如果希望错误输出更可控，也可以自己处理：

展开

自动换行

use std::fs;

use std::process;

fn main() {

if let Err(e) = run() {

eprintln!("error: {}", e);

process::exit(1);

}

}

fn run() -> Result<(), std::io::Error> {

let context = fs::read\_to\_string("hello.txt")?;

println!("{context}");

Ok(())

}

这种结构在命令行程序中很常见：真正的逻辑写在 `run()` 中， `main()` 只负责统一收口错误。

## 9、自定义错误类型

对于前面的 `read_numbers()` 函数，因为函数内会返回 `std::io::Error` 和 `std::num::ParseIntError` 两种错误，所以前面建议的第一种方式是自定义错误类型。

例如，针对上述这种情形，可以 **定义一个枚举，把不同的错误包起来** 。

展开

自动换行

#\[derive(Debug)\]

enum MyError {

Io(io::Error),

Parse(std::num::ParseIntError)

}

这样， `MyError` 就可以表示两类错误：IO错误和整数解析错误。

错误通过需要 `{}` 格式化打印，因此还需要为 `MyError` 实现 `Display` 特征：

展开

自动换行

impl Display for MyError {

fn fmt(&self, f: &mut Formatter<'\_>) -> std::fmt::Result {

match self {

MyError::Io(err) => write!(f, "io error: {}", err),

MyError::Parse(err) => write!(f, "parse error: {}", err)

}

}

}

所有的错误类型都要实现 `std::error::Error` ，通常下面这一行就够了：

展开

自动换行

impl Error for MyError {}

为了让 `std::io::Error` 和 `std::num::ParseIntError` 可以转换为 `MyError` ，还需要为 `MyError` 实现 `From<std::io::Error` 和 `From<std::num::ParseIntError>` ：

展开

自动换行

impl From<io::Error> for MyError {

fn from(value: io::Error) -> Self {

Self::Io(value)

}

}

impl From<std::num::ParseIntError> for MyError {

fn from(value: std::num::ParseIntError) -> Self {

Self::Parse(value)

}

}

完整示例：

展开

自动换行

use std::error::Error;

use std::io::{self, BufRead};

use std::fmt::{Display, Formatter};

#\[derive(Debug)\]

enum MyError {

Io(io::Error),

}

impl Display for MyError {

fn fmt(&self, f: &mut Formatter<'\_>) -> std::fmt::Result {

match self {

MyError::Io(err) => write!(f, "io error: {}", err),

MyError::Parse(err) => write!(f, "parse error: {}", err)

}

}

}

impl Error for MyError {}

impl From<io::Error> for MyError {

fn from(value: io::Error) -> Self {

Self::Io(value)

}

}

impl From<std::num::ParseIntError> for MyError {

fn from(value: std::num::ParseIntError) -> Self {

Self::Parse(value)

}

}

## 13、实用工具特征

## 一、概述

Rust标准库有一组非常重要但又不直接对应某个具体业务领域的特征。它们更多是在回答这样一个问题： **一个类型具备什么通用能力、能以什么方式参与语言机制、又如何与其他库协作** 。这一类特征称为 **实用工具特征** ，可以把它们理解为 **Rust为类型系统准备的一套通用接口约定** 。

Rust中常见的实用工具特征大致分为三类：

**（1）语言扩展特征**

**这类特征最贴近Rust的语言行为** 。它们的作用 **不是单纯提供一个普通方法，而是让自定义类型更自然融入语言本身** 。

除了之前介绍的和 **运算符重载相关** 的几个特征外，还有一些常见代表：

`Drop` ：定义值被销毁时的清理逻辑

`Deref` / `Deref Mut` ：让自定义类型在很多场景下像引用一样使用，并自动参与解引用

`From` / `Into` ：描述值之间的明确转换关系

这些trait的共同点是： **实现之后，类型在语言中的表现会发生变化** 。

**（2）标记特征**

这类特征通常 **不强调做什么** ，而 **强调它是什么样的类型** 。也就是说，它们 **更像是给类型贴标签，用来说明类型满足某种性质** 。

常见代表：

`Sized` ：类型的大小在编译期已知

`Copy` ：这个类型可以通过简单的按位复制产生独立值

**（3）通用约定特征**

这类特征归类为公共词汇类，它们 **不直接扩展语言语法，而是为大量库和API提供统一的表达方式** 。这类Trait的价值在于： **不同库只要遵守同样的trait约定，就更容易无缝协作** 。

常见代表：

`Default` ：提供一个合理的默认值

`AsRef` / `AsMut` ：提供廉价的借用转换

`Borrow` / `BorrowMut` ：也是借用转换，但语义要求更强

`TryFrom` / `TryInto` ：表示可能失败的转换

`ToOwned` ：从借用数据得到对应的拥有值

## 二、Drop

**当一个值的拥有者消失时，Rust会丢弃（drop）该值** 。丢弃一个值意味着释放该值拥有的任何其他值、堆存储和系统资源。

**大多数情况下，Rust会为自定义类型自动生成析构逻辑** 。例如：

展开

自动换行

struct Person {

name: String,

tags: Vec<String>,

}

`Person` 拥有用作字符串内容和向量元素缓冲区的堆内存。当 `Person` 类型的值被丢弃时，Rust会自动清理：

`String` 会自动释放自己的缓冲区

`Vec<String>` 先销毁每个元素，再释放自己的缓冲区

对于标准库类型 `String` 和 `Vec<String>` 来说，已经为它们实现了 `Drop` 特征，所以编译期会自动递归销毁字段。

当然，也可以为自定义类型手动实现 `std::ops::Drop` 特征，其原型如下：

展开

自动换行

trait Drop {

fn drop(&mut self);

}

说明：

`drop()` 接收的是 `&mut self` ，说明它是在 **值仍然存在，但即将被销毁的阶段** 被调用的，因此在方法的内部仍然可以访问它的字段。

Rust丢弃值时，先对值本身调用 `Drop::drop()` ，然后继续自动销毁这个值内部的字段。

下面的例子可以清楚看到 `Drop` 的触发时机：

展开

自动换行

struct Person {

name: String,

age: u32,

}

impl Drop for Person {

fn drop(&mut self) {

println!("cleaning person: name: {}, age: {}", self.name, self.age);

}

}

fn main() {

let mut p1 = Person { name: "Tom".to\_string(), age: 25 };

println!("before reassignment");

p1 = Person { name: "Jerry".to\_string(), age: 20 };

println!("before end");

}

结果：

![[Image 66.png]]

从中可以看出： **当旧值覆盖新值时，也会触发值的销毁** 。一个值被销毁，常见有以下时机：

**变量离开作用域**

**旧值覆盖新值**

**容器中的元素被移除**

**函数参数或局部变量生命周期结束**

需要注意的是，程序在 `panic` 时默认采用的是 **unwind** （调用栈展开）方式，此时栈上的值也会在展开过程中依次销毁；如果采用的是 **abort** 模式，则不会进行这种逐步析构。

大多数情况下，并不需要为了“释放内存”而自己实现 `Drop` ，因为标准库类型已经把这件事处理好了。Rust默认会递归地销毁字段所拥有的资源，程序员通常不必手动实现 `Drop` 。

真正需要自己实现 `Drop` 的场景通常是下面两类：

**某种类型拥有标准库不知道如何清理的外部资源**

**希望对象在销毁时执行额外逻辑，如记录日志、发出通知等**

**Rust保证一个值最多被销毁一次** 。因此，如果值的所有权已经移动到别处，原绑定不再参与销毁。

示例：

展开

自动换行

let s = String::from("Hello");

let t =s;

// s已经失效，不能再使用

// 最终只会drop t

原来s的那份值已经转移到 t，所以最终只会在t的生命周期结束时销毁一次。

关于 `Drop` 的几点注意事项：

**实现了** `Drop` **特征的类型不能实现** `Copy`

这是因为 `Copy` 和 `Drop` 天然冲突：

如果一个类型实现了 `Copy` ，表示它可以被隐式复制出多个副本

如果一个类型实现了 `Drop` ，通常说明它管理的是“唯一清理责任”的资源

例如，假设文件描述符既可以 `Copy` 又可以 `Drop` ，那么多个副本持有的是用一个文件描述符，最终都可能尝试关闭同一个文件描述符，这显然是错误的。

虽然 `drop()` 是 `Drop` 特征的方法，但它不能由程序员显式直接调用，下面的写法都是错误的：

展开

自动换行

value.drop();

Drop::drop(&mut value);

因为 `drop` 方法接收的是值的可变引用，如果手动显式调用，那么 `value` 值后面还可以继续使用（没有生效效），但其内部本身已经被释放和清理了。因此，作用域结束后，编译器又会自动调用一次，从而造成重复析构的问题。

如果确实想提前销毁一个值，通过标准库的函数 `std::mem::drop()` 完成。

`std::mem::drop` 按值接收，因此所有权转移到了 `drop` 函数内部，当离开该函数时，在触发该值的 `Drop::drop()` 方法。

示例：

展开

自动换行

let s = String::from("hello");

std::mem::drop(s);

调用之后，s的所有权被移动进 `drop` 函数，因此它会在函数结束时被销毁，之后不能再使用 `s` ，编译器就不会隐式调用 `Drop::drop()` 了。

从机制上看， `std::mem::drop` 其实很简单，本质就是一个空实现：

展开

自动换行

fn drop(\_x: T) {}

也就是说，它按值接收参数，不拿走所有权。函数体什么也不做，但参数\_x会在函数结束时离开作用域，于是自然触发释放。

## 三、Sized

**固定大小类型是指该类型的每个值在内存中都有相同的大小** 。Rust几乎所有类型都是固定大小的，如每个 `i32` 类型占用4字节，元组 `(f64, f64)` 占用16字节。枚举类型也有大小，无论是哪一个变体值，其所占内存都是枚举中最大变体值的大小。

**所有的固定大小类型都实现了** `std::marker::Sized` **特征，该特征没有方法或关联类型** 。 **Rust自动为所有适用的类型实现了** `Sized` **，不能手动实现它** 。

`Sized` 的唯一用途是 **用作类型参数的限界** ：像 `T: Sized` 表示 `T` 必须是在编译期已知的类型。由于Rust语言本身会使用这种类型的特征为具有某些特征的类型打上标记，因此将其称为 **标记特征** 。

Rust也有一些无固定大小类型，它们的值不尽相同。例如，字符串切片 `str` 、切片 `[T]` 以及 `dyn` 类型都是无固定大小的类型。

示例：

![[Image 67.png]]

**Rust不能将无固定大小的值存储在变量中或将它们作为参数传递** ，只能通过像 `&str` 或 `Box<dyn Write>` 这样的本身是固定大小的指针来处理它们。它们称为 **胖指针** ，包含两个字段：

对于切片引用，包含 **指向切片的指针和切片的长度**

对于特征对象，包含 **指向实际数据的指针和指向实现特征的方法的虚函数表指针**

无固定大小类型在使用上处处受限，因此大多数泛型类型变量被限制在固定大小的 `Sized` 类型。事实上，鉴 **于这种情况下的普遍性，** `Sized` **已成为Rust的默认隐式值** 。当写：

展开

自动换行

struct Person<T> {

...

}

Rust会将其理解为：

展开

自动换行

struct Person<T: Sized> {

...

}

但是，如果希望T没有这种约束，就需要明确写：

展开

自动换行

struct S<T:?Sized> {

...

}

`?Sized` 表示不要求固定大小，即可以是固定大小也可以是非固定大小。

例如，可以写成：

展开

自动换行

struct S<T:?Sized> {

b: Box<T>

}

那么Rust将允许写 `S<str>` 和 `S<dyn Write>` ，这样 `Box<str>` 和 `Box<dyn Write>` 就都是胖指针了。

## 四、Clone

## 1、Clone特征的定义

`std::clone::Clone` 特征适用于可复制自身的类型，其定义如下：

展开

自动换行

trait Clone: Sized {

fn clone(&self) -> Self;

fn clone\_from(&mut self, source: &Self) {

\*self = source.clone();

}

}

说明：

`clone` 方法为 `self` 构造了一个独立的副本并返回。由于此方法的返回类型为 `Self` 且函数也不可能返回无固定大小的类型，因此增加了限界 `Clone: Sized` ，即实现了 `Clone` 的类型也要实现 `Sized` 。

`clone_from` 方法会将 `self` 修改成 `source` 的副本，它是默认实现方法。本质上就是将 `source` 克隆出来的值转移给了 `*self` 。 **不过，某些类型可以重写** `clone_from` **以提高效率，例如标准库的** `String` 。

示例：

展开

自动换行

let s = String::from("hello");

let t = String::from("xyz");

s.clone\_from(&t);

假如使用的是 `clone_from` 的默认实现，那么上述代码等价于：

展开

自动换行

let s = String::from("hello");

let t = String::from("xyz");

s = t.clone();

这里涉及了一次堆分配和一次堆释放：分配了新的内存存放"xyz"，释放了原先s的缓冲区。最后让s指向新的堆内存。

但有时候这里的堆分配和堆释放不必要：这里的s的 `hello` 5个字节大于 `xyz` ，因此完全可以将 `xyz` 直接覆盖旧内存，然后调整长度即可。

因此 `String` 重写了 `clone_from` 进行了优化，先检查s当前缓冲区是否小于新数据的大小，如果小于再按原来的方式进行堆分配和释放；如果大于等于，则直接拷贝新数据到当前缓冲区，然后调整长度即可。

## 2、Clone注意事项

需要注意的是，克隆一个值通常还需要为它拥有的任何值分配副本，因此 `clone` 无论在时间消耗还是内存占用方面都是相当昂贵的。例如，克隆 `Vec<String>` 不仅会复制此向量，还会复制它的每个 `String` 元素。 **这就是为什么Rust不自动克隆值，而是要求程序员手动进行显式方法调用的原因** 。

**大多数情况下的** `clone()` **都是深拷贝的语义，但也有例外，比如** `Rc<T>` **和** `Arc<T>` **这样的引用计数智能指针，它们调用** `clone()` **只会增加一个引用计数，并不会深拷贝底层数据** 。

## 3、#\[derive(Clone)\]

如果一个类型的 `Clone` 实现只是对它的每个字段分别调用 `clone()` ，再用克隆结果构造出一个新值，并且 `clone_from` 的默认实现已经足够满足需求，那么通常情况下不需要手动实现 `Clone` ，可通过派生属性宏自动实现：

展开

自动换行

#\[derive(Clone)\]

标准库中，几乎所有可以合理复制的类型都实现了 `Clone` 。例如原始类型，如 `i32` 、 `bool` ，常见拥有型类型，如 `String` 以及容器类类型，如 `Vec<T>` 、 `HashMap<K, V>` 。这些类型之所以能实现 `Clone` ，是因为它们都能够 **明确地构造出一个新的、独立的副本** 。

**但并非所有的类型适合** `Clone` **，有些类型本身就不适合复制出另一个副本：**

`std::sync::Mutex` ： `Mutex` 本身的核心语义是管理独占/同步状态， `clone` 这个操作对它来说不自然，也容易让人误会。

`std::fs::File` ： `File` 在概念上可以复制，但这个复制不是普通内存复制，而是向操作系统申请一个新的文件句柄/描述符，指向同一个底层文件资源。但如果发生系统资源不足，就会失败， `clone` 不能返回 `Result` 代表错误情况。因此作为替代， `File` 提供了可能失败的克隆操作： `try_clone()` ，返回的是 `std::io::Result<File>` 。

## 五、Copy

对于大多数类型，赋值时会移动值而不是复制值。 **不过，对于不拥有任何资源的简单类型赋值会创建源的副本，而不会移动值并使得源变为未初始化状态** 。这些简单类型实现了 `std::marker::Copy` ，它也是一种 **标记特征** ，其定义如下：

展开

自动换行

trait Copy: Clone { }

**实现** `Copy` **的类型必须实现** `Clone` 。由于 `Copy` 是一种对语言有着特殊意义的标记特征，因此只有当类型需要一个浅层的逐字节赋值时，Rust才允许为它实现 `Copy` 。拥有任何其他资源（如堆缓冲区或操作系统句柄）的类型都无法实现 `Copy` 。

与 `Clone` 一样，可以使用 `#[derive(Copy)]` 让Rust派生 `Copy` 的实现。实际开发中，也会经常看到同时使用 `#[derive(Clone, Copy)]` 的代码。例如，自定义一个结构体，其中仅仅包含简单类型，那么就可以直接 `#[derive(Clone, Copy)]` ：

展开

自动换行

#\[derive(Clone, Copy)\]

struct Test {

a: i32,

b: i32,

}

然而，如果字段中包含拥有资源的类型，如 `String` ，那么只能使用 `#[derive(Clone)]` ：

展开

自动换行

#\[derive(Clone)\]

struct User {

name: String,

age: u8,

}

## 六、Deref与DerefMut

## 1、概述

通过实现 `std::ops::Deref` 和 `std::ops::DerefMut` ，可以让自定义类型在很多场景下表现得像“指向某个目标值的指针”。它们最直接影响的是Rust中的解引用行为，例如 `*` **运算符、方法调用时自动借用或解引用，以及引用类型之间的隐式转换** 。

像 `Box<T>` 、 `Rc<T>` 、 `Arc<T>` 这样的智能指针都实现了这些特征，因此使用起来 **很像Rust的内置引用类型** 。例如，对于一个 `Box<Complex>` 类型的值b， `*b` 表示访问 `b` 所指向的那个 `Complex` 的值， `b.re` 则可以直接访问它内部的字段，看起来像 `b` 本身就是一个 `Complex` 一样。

可以把这两个trait的作用概括为两点：

定义 `*` 运算符对某个类型的行为

让类型自动参与Rust的 **解引用强制转换** （deref coercion）

其中：

`Deref` 用于 **不可变解引用** ，得到共享引用

`DerefMut` 用于 **可变解引用** ，得到可变引用

**如果上下文只需要只读访问，那么实现** `Deref` **就够了；如果上下文还需要修改目标只，那么还需要实现** `DerefMut` 。

## 2、Deref和DerefMut的定义

这两个trait的定义如下：

展开

自动换行

trait Deref {

type Target =?Sized;

fn deref(&self) -> &Self::Target;

}

trait DerefMut: Deref {

fn deref(&mut self) -> &mut Self::Target;

}

说明：

`Deref` 关联了一个目标类型 `Target` ，这表示当前类型在解引用得到的类型。

`deref(&self) -> &Self::Target` 表示从 `&Self` 出发，取得一个 `&Target` 。

`DerefMut: Deref` 表示实现 `DerefMut` 的前提是先实现 `Deref` 。这很合理，因为如果一个类型能提供可变引用，那么它当然也应该能提供共享引用。

`deref_mut(&mut self) -> &mut Self::Target` 表示从 `&mut Self` 出发，缺的一个 `&mut Target` 。

这里的 `Target` 通常就是该类型“内部真正要访问的对象”。例如：

● 对于 `Box<Complex>` ， `Target` 是 `Complex`

● 对于 `Vec<T>` ， `Target` 是 `[T]`

● 对于 `String` ， `Target` 是 `str`

还需要注意一点： `Target` 写成了 `?Sized` ，说明目标类型不要求一定在编译期大小固定。因此像 `str` 、 `[T]` 这样的 DST（动态大小类型）也可以作为解引用目标。

## 3、Deref的本质：返回内部目标的引用

`deref` 和 `deref_mut` 的本质，并不是把自己变成另一个类型，而是 **把当前类型借用成它内部某个目标值的引用** 。

例如， `Box<T>` 内部拥有一个 `T` ，所以它可以返回 `&T` ， `String` 内部拥有一段UTF-8字节缓冲区，并且保证其内容始终是合法字符串，因此它可以返回 `&str` ， `Vec<T>` 内部持有连续元素缓冲区，因此可以返回 `&[T]` 。

由于返回的是引用，而且这个引用的生命周期与 `&self` 或 `&mut self` 绑定，所以只要返回值还活着，原对象就仍然处在借用状态中。这也保证了借用规则不会被破坏。

## 4、Deref参与的语言行为

### （1）\*运算符

实现了 `Deref` 之后，可以对类型使用 `*` 运算符； `DerefMut` 之后，也可以在需要可变访问的场景中使用可变解引用。

示例：

展开

自动换行

use std::ops::{Deref, DerefMut};

struct Selector<T> {

elements: Vec<T>,

current: usize,

}

impl<T> Deref for Selector<T> {

type Target = T;

fn deref(&self) -> &Self::Target {

&self.elements\[self.current\]

}

}

impl<T> DerefMut for Selector<T> {

fn deref\_mut(&mut self) -> &mut Self::Target {

&mut self.elements\[self.current\]

}

}

fn main() {

let mut s = Selector {

elements: vec!\[10, 20, 30\],

current: 2,

};

assert\_eq!(\*s, 30); // \*s == \*(s.deref())

\*s = 40; // \*s == \*(s.deref\_mut())

assert\_eq!(\*s, 40);

}

这里 `*s` 实际上会调用 `deref()` 或 `deref_mut()` ，从而访问 `current` 所指向的那个元素。

### （2）方法调用中的自动解引用

**Rust在调用方法时，会自动尝试借用和解引用，直到找到匹配的方法为止** 。因此，一个类型即使没有直接实现某个方法，只要它能通过 `Deref` 解引用到拥有该方法的目标类型，也可以直接调用。

示例：

展开

自动换行

use std::rc::Rc;

fn main() {

let r = Rc::new(String::from("hello?"));

assert\_eq!(r.find('?'), Some(5));

}

虽然 `find` 是定义在 `String` 或更准确地说最终可作用到 `str` 上的方法，但这里可以直接写 `r.find('?')` 。编译器会自动完成类似下面的过程：

先把 `r` 借用成 `&Rc<String>`

再通过 `Deref` 把它变成 `&String`

继续把 `&String` 变成 `&str`

最终在目标类型上找到合适的方法

因此，用户写起来非常自然，不必手动写出多层 `*` 或显式转换。

### （3）解引用强制转换（deref coercion）

`Deref` 和 `DerefMut` 还有一个非常重要的作用： **让引用类型在需要时自动转换为目标引用类型。**

也就是说，如果编译器发现：

● 当前拿到的是 `&Self` 但目标位置需要的是 `&Target`

并且 `Self: Deref<Target = Target>`

那么它就会自动插入一次 `deref()` 调用。

例如：

展开

自动换行

fn show(s: &str) {

println!("{}", s);

}

fn main() {

let s = String::from("hello");

show(&s);

}

这里传入的是 `&String` ，而函数需要的是 `&str` 。之所以能通过，就是因为 `String` 实现了 `Deref<Target = str>` ，编译器会自动把 `&String` 转换成 `&str` 。

## 5、Deref/DerefMut适合的类型

智能指针类型：典型代表有 `Box<T>` 、 `Rc<T>` 、 `Arc<T>` 以及某些自定义智能指针等。这些类型本身的语义就是 **持有并间接访问一个目标值** 。

拥有型容器到借用型视图的转换：典型代表有 `String -> str` 、 `Vec<T> -> [T]` 、 `PathBuf -> Path` 、 `OsString -> OsStr` 。这类类型的特点是：一个类型拥有数据，另一个类型是对这份数据的借用视图。实现 `Deref` 后，可以让 **拥有型类型自然复用借用型类型的方法集** 。

## 七、Default

## 1、概述

在实际开发中，很多类型都有一个自然且合理的初始值，例如：

向量(`Vec<T>`)或字符串(`String` ）默认是空的

数值类型默认为0

`Option<T>` 默认是 `None`

对于这类类型，Rust提供了 `std::default::Default` 特征，用来统一描述如何构造一个默认值。

其定义如下：

展开

自动换行

trait Default {

fn default() -> Self;

}

可以将其理解为为类型提供一个零配置的构造方式。

## 2、Default的基本语义

`default()` 方法的职责非常简单：

不接收任何参数

返回一个类型自身（ `Self` ）的默认实例

例如， `String` 的默认实现本质上就是创建一个空字符串：

展开

自动换行

impl Default for String {

fn default() -> Self {

String::new()

}

}

这些默认值通常具有“可安全使用、不会出错、语义合理”的特点。

## 3、Default在泛型中的作用

`Default` 的一个重要用途是：在泛型代码中提供“初始化能力”。标准库中的很多泛型算法会依赖这一点，例如 `Iterator::partition` 。

示例：

展开

自动换行

use std::collections::HashSet;

fn main() {

let squares = vec!\[2, 4, 9, 16, 25, 36, 49\];

let(powers\_of\_two, others): (HashSet<i32>, HashSet<i32>) = squares.iter().partition(|&x| (x & x - 1) == 0);

println!("{:?}", powers\_of\_two);

println!("{:?}", others);

}

这里的关键点是：

`partition` 并不关心具体使用哪种集合类型

它只要求目标类型满足：

`Default` ：用于创建空集合

`Extend<T>` ：用于向集合中添加元素

因此，只要类型满足这两个约定，就可以作为 `partition` 的输出容器。

## 4、Default + 结构体初始化

`Default` 的另一个非常常见的用途是：为配置型结构体提供默认值。

在实际项目中，经常会遇到参数很多的结构体，大部分字段都有合理默认值，调用者只需要修改少数几个字段。

示例：

展开

自动换行

let params = glium::DrawParameters {

line\_width: Some(0.02),

point\_size: Some(0.02),

..Default::default()

};

这里发生了三件事：

（1）调用 `Default::default()` 构造一个完整的默认值

（2）使用结构体更新语法`..`

（3）只覆盖需要修改的字段

这种写法的优点是：

● 避免手动填写大量默认字段  
● 保证未指定字段仍然是合法值  
● 可读性强，意图明确

## 5、自动实现Default（derive）

Rust不会自动为结构体实现 `Default` ，但如果满足条件，可以通过派生自动生成实现。

展开

自动换行

#\[derive(Default)\]

struct Config {

a: i32,

b: Option<String>,

}

前提条件是： **所有字段类型都必须实现了 Default** 。

生成的行为等价于：

展开

自动换行

impl Default for Config {

fn default() -> Self {

Self {

a: Default::default(),

b: Default::default(),

}

}

}

## 6、标准库中的自动传播

如果某个类型 `T` 实现了 `Default` ，那么很多包装类型也会自动实现 `Default` ，例如：

● `Box<T>`  
● `Rc<T>` / `Arc<T>`  
● `Cell<T>` / `RefCell<T>`  
● `Mutex<T>` / `RwLock<T>`  
● `Cow<T>`

这些类型的默认值通常是：包裹了 `T` 的默认值。

示例：

展开

自动换行

let x: Box<i32> = Default::default(); // Box::new(0)

此外，如果元组中所有元素都实现了 `Default` ，那么该元组也自动实现 `Default` 。

## 八、AsRef与AsMut

## 1、概述

`AsRef` 和 `AsMut` 是Rust中一组常用的转换类Trait，用来表示：

能否从一个值中廉价地借出某种目标类型的引用

这种借用是共享借用（ `AsRef` ）还是可变借用（ `AsMut` ）

它们的核心不是生成一个新的值，也不是转移所有权，而是 **把一个值临时看作另一种引用形式来使用** 。例如 `String` 可以借成 `&str` ， `Vec<T>` 可以借成 `&[T]` ， `PathBuf` 可以借成 `&Path` 。

这种转换不需要重新分配内存，也不会复制底层数据，因此通常成本更低。

## 2、定义

`AsRef` 和 `AsMut` 的定义如下：

展开

自动换行

trait AsRef<T:?Sized> {

fn as\_ref(&self) -> &T;

}

trait AsMut<T:?Sized> {

fn as\_mut(&mut self) -> &mut T;

}

说明：

`AsRef<T>` 表示可以从 `&Self` 得到一个 `&T`

`AsMut<T>` 表示可以从 `&mut Self` 得到一个 `&mut T`

`T: ?Sized` 说明目标类型 `T` 不一定要求固定大小类型，因此像 `str` 、 `[T]` 、 `Path` 这样的动态大小类型也可以作为目标。

示例：

展开

自动换行

let s = String::from("hello");

let r: &str = s.as\_ref();

这并不是把 `String` 变成了 `str` ，也不是创建了一个新的字符串，而只是： **从** `String` **中借出了一个** `&str` **视图。** 底层数据仍然是一份，只是访问方式变了。

## 3、常见实现示例

标准库里很多类型都是实现了 `AsRef` ，例如：

`String: AsRef<str>`

`String: AsRef<[u8]>`

`Vec<T>: AsRef<[T]>`

`PathBuf: AsRef<Path>`

`OsString: AsRef<OsStr>`

这些实现反映的都是一种很自然的关系： **拥有型类型可以借成对应的借用视图类型** 。

## 4、AsRef使用场景

`AsRef` 最常见的使用场景，是写出更通用、更好用的 **函数参数** 类型。

例如，标准库中的文件打开函数如下：

展开

自动换行

fn open<P: AsRef<Path>>(path: P) -> Result<File>;

这个签名的意思不是参数必须就是 `Path` ，而是只要这个参数能借用成 `&Path` 就接受。因此，调用者可以传入多种不同的类型：

&str

String

&String

PathBuf

&PathBuf

&Path

这里可能会有疑问， `String` 实现了 `AsRef<Path>` 可以理解，为什么这里传入 `&String` 也可以？

这是因为标准库有一个通用实现：

展开

自动换行

impl AsRef<U> for &T

where:

T: AsRef<U>

U:?Sized,

T:?Sized,

{

fn as\_ref(&self) -> &U {

(\*self).as\_ref()

}

}

换句话说，对于任意类型的 `T` 和 `U` ，只要满足 `T: AsRef<U>` ，那么就必然满足 `&T: AsRef<U>` 。

## 5、AsRef和Deref的区别

`AsRef` 和 `Deref` 看起来都能“得到另一个引用”，但它们的定位并不相同：

`Deref` 更强调 **类型像一个指针或包装器** ，参与 `*` 运算，参与方法调用时的自动解引用，参与解引用强制类型转换。

`AsRef` 更强调 **类型能否廉价地借成某种目标引用** ，主要用于 **函数参数和泛型约束** ，它是一种通用的API约定。

## 6、AsMut和AsRef的区别

`AsMut` 看起来只是 `AsRef` 的可变版本，但实际上它的 **实现条件往往更严格** 。

原因在于： **只读借用通常不会破坏对象的内部约束，而可变借用则可能让调用者任意修改底层数据，从而破坏类型必须保持的有效状态** 。

例如， `String` 实现了 `AsRef<[u8]>` ，这是合理的，因为字符串本来就可以按字节查看。但 `String` 不能轻易实现 `AsMut<[u8]>` ， 因为如果把底层字节数组作为 `&mut [u8]` 暴露出去，调用者就可能随意修改字节内容，进而破坏 `String` 必须始终保持 **合法UTF-8** 这一约束。

## 九、Borrow和BorrowMut

## 1、概述

`Borrow` 和 `BorrowMut` 也是一组借用转换类Trait。仅从签名上来看，它们和前面 `AsRef` 、 `AsMut` 非常相似：都是从一个值高效借出目标类型的引用，而不发生所有权的转移，也不进行额外分配与深拷贝。

然而， `Borrow` 与 `AsRef` 多了一层更强的语义。 `AsRef` 只保证这个类型可以借成 `&T` ，而 `Borrow` 除了可以借成 `&T` ，还要求 **借出来的** `&T` **，在比较、哈希和排序等行为上，应当与原值保持一致** 。也就是说， **也就是说，** `Borrow` **关心的不只是能不能借，还关心借出来之后，能不能把它当作原值本身来比较和查找** 。一个最经典的例子就是 `String` 和 `&str` ，比如：

展开

自动换行

use std::collections::HashMap;

fn main() {

let mut map = HashMap::new();

map.insert(String::from("alice"), 100);

println!("{:?}", map.get("alice"));

}

这里 `map` 的键类型为 `String` ，但是查找时传进去的是 `&str` 类型的"alice"。能这样做的原因是标准库为 `String` 实现了 `Borrow<str>` ， 这表示 `String` 可以借出一个 `&str` ，并且这个 `&str` 在哈希和相等性上，应当和这个 `String` 表示同一个键。

正因为语义更强， `Borrow` 最主要的用途并 **不是普通的参数借用** ，而是用于 `HashMap` 、 `BTreeMap` 、 `HashSet` 、 `BTreeSet` 这类关联容器的键查找。

## 2、定义

`Borrow` 和 `BorrowMut` 的定义如下：

展开

自动换行

trait Borrow<Borrowed:?Sized> {

fn borrow(&self) -> &Borrowed;

}

trait BorrowMut<Borrowed:?Sized>: Borrow<Borrowed> {

fn borrow\_mut(&mut self) -> &mut Borrowed;

}

说明：

● `Borrow<T>` 表示可以从 `&Self` 得到 `&T`  
● `BorrowMut<T>` 表示可以从 `&mut Self` 得到 `&mut T`  
● `BorrowMut` 继承自 `Borrow` ，因此一个类型如果能可变借出 `&mut T` ，它也必须能共享借出 `&T`  
● `T: ?Sized` 表示目标类型不一定要求是固定大小类型，所以 `str` 、 `[T]` 、 `Path` 等类型都可以作为借用目标

不过，真正要强调的并不是这两个方法本身，而是 `Borrow` 的语义约束： **只有当借出的目标类型在比较、排序、哈希等行为上与原类型一致，原类型才适合实现** `Borrow` 。

Rust 编译器并不会强制检查这件事，但这是 `Borrow` 被设计出来时的核心意图。也正因为如此， `Borrow` 不是一个“只要能借就随便实现”的 trait， **它的适用范围比** `AsRef` **更窄，也更讲究语义是否匹配** 。

例如， `String` 可以实现多种 `AsRef` ：

`AsRef<str>`

`AsRef<[u8]`

`AsRef<Path>`

因为从“借用视图”的角度看，这些都说得通：字符串既可以看成文本切片，也可以看成字节切片，还可以在某些场景下看成路径。

但 `Borrow` 不一样。对 `String` 来说，真正适合实现的是：

展开

自动换行

String: Borrow<str>

因为 `String` 和 `str` 的相等性判断与哈希方式都是 **基于文本** 内容的，语义完全一致。

## 3、Borrow的典型用途

`Borrow` 最重要也是最典型的应用场景，是 **解决关联容器查找时“键类型过于死板”的问题** 。

假设有这样一个哈希表：

展开

自动换行

use std::collections::HashMap;

let mut map = HashMap::new();

map.insert("apple".to\_string(), 3);

map.insert("banana".to\_string(), 5);

这里的键类型是 `String` 。那么，查找函数应该怎么设计呢？

如果最开始写成这样：

展开

自动换行

fn get(&self, key: K) -> Option<&V>

问题就很明显：查找一个元素，居然还必须按值传入一个完整的键。这意味着对于 `HashMap<String, V>` ，每次查找都要构造一个 `String` ，这既不自然，也很浪费，因为查找本身只需要“看一下键”，根本不需要拿走所有权。

于是可以退一步，改成接收引用：

展开

自动换行

fn get(&self, key: &K) -> Option<&V>

这确实比前一种好一些，但仍然不够理想。因为如果 `K = String` ，那查找参数就必须是 `&String` 。这会导致一种很别扭的写法：

展开

自动换行

map.get(&"apple".to\_string())

这里只是想查找一个 `"apple"` ，却还得临时在堆上分配一个 `String` ，把文本复制进去，再借成 `&String` 用来查表，最后马上丢弃。这显然也是不高效的。

理想情况是：哈希表里虽然存的是 `String` ，但查找时应该允许直接传 `&str` 。因为对于查找而言，真正重要的不是“你传进来的值是不是 `String` ”，而是 **它能不能以与** `String` **一致的方式参与比较和哈希。** 这正是 `Borrow` 发挥作用的地方。因此，标准库中 `HashMap::get()` 的思路大致如下：

展开

自动换行

fn get<Q:?Sized>(&self, key: &Q) -> Option<&V>

where

K: Borrow<Q>

Q: Eq + Hash

对于这里的 `HashMap<String, i32>` 来说， `K = String` ， `V = i32` ，因为 `String: Borrow<str>` ，因此调用 `get` 时可以直接写：

展开

自动换行

assert\_eq!(map.get("apple"), Some(&3));

## 4、BorrrowMut注意事项

`BorrowMut` 是 `Borrow` 的可变版本，它表示一个类型不仅可以借出共享引用，还可以借出可变引用 `&mut T` 。不过，相比 `Borrow` ， `BorrowMut` 的使用场景通常少得多，因为它的要求更严格。

原因在于：一旦把内部数据以 `&mut T` 的形式暴露出去，调用者就可以修改它，而这些修改不能破坏原类型的基本语义。 **对于** `BorrowMut` **来说，这不仅意味着不能破坏内存安全，还意味着修改之后，这个类型在比较、排序、哈希等行为上仍然要和借出的目标类型保持一致** 。

因此， `BorrowMut` 只有在“借出的可变视图不会破坏原类型不变性”的前提下才适合实现。这个要求和前面 `AsMut` 的限制有些类似，只不过 `BorrowMut` 还额外涉及比较和哈希语义的一致性，因此更需要谨慎。

## 5、补充

标准库还提供了一些方便使用的通用实现，最重要的有两点：

**任意类型** `T` **都可以从自身借用** ，即 `T: Borrow<T>`

`&mut T` 也可以借成 `&T` ，因此某些需要 `Borrow<T>` 的场景，传入可变引用也同样成立

## 十、From和InTo

## 1、概述

`From` 和 `Into` 是Rust中用于 **值转换** 的一组通用Trait。 它们表示， **把一个类型的值按值接收过来，转换成另一个类型的值，再发转换结果返回出去** 。

这一点和前面讲过的 `AsRef` 、 `AsMut` 很不一样。 `AsRef` / `AsMut` 做的是借用转换，本质只是临时借出一个引用，而 `From` / `Into` 做的是所有权转换，它们拿走原值，并返回一个新的目标值。

## 2、定义

`From` 和 `Into` 的定义如下：

展开

自动换行

trait From<T> {

fn from(value: T) -> Self;

}

trait Into<T> {

fn into(self) -> T;

}

从形式上看，这两个 trait 很对称：

`From` 站在目标类型的角度，表示我可以由 `T` 构造出来

`Into` 站在源类型的角度，表示我可以变成 `T`

虽然它们看起来像是在做同一件事，但实际使用时分工并不一样：写实现时通常以 `From` 为主，设计泛型参数时通常以 `Into` 为主。

## 3、核心语义与基本用法

`From` 和 `Into` 的共同点在于： **它们都会接管参数的所有权** 。这意味着， **转换过程中，原值将被消耗，它拥有的资源也可以直接拿过来复用，而不一定需要重新分配或复制** 。

示例：

展开

自动换行

let s = "hello".to\_string();

let bytes: Vec<u8> = s.into();

这里 `s` 的类型是 `String` ，转换结果是 `Vec<u8>` 。这不是借用，也不是从 `String` 中拷贝出一份字节数据出来，而是直接把 `String` 内部那块UTF-8缓冲区的所有权转移给 `Vec<u8>` 使用。由于两者底层都可以使用同一种堆缓冲区表示，因此这个转换可以非常高效，不需要重新分配，也不需要复制内容。

当然，并不是所有 `From` / `Into` 转换都这么便宜，它们只是表示可以转换，并不承诺一定是零开销，例如：

展开

自动换行

let s = String::from("hello");

let t = String::from(&s\[..\]);

这里 `String: From<&str>` 的实现就必须重新分配新的缓冲区，并把字符串内容复制进去。因为 `&str` 只是借用视图，不拥有底层数据，不能像前面的 `String -> Vec<u8>` 那样直接搬走资源。

## 4、使用习惯

`From` 和 `Into` 在使用习惯上有明显分工： `From` 常用于构造目标值， `Into` 常用于函数参数。

当编写一个函数，并希望它的参数更灵活时，通常会用 `Into<T>` 。因为从函数的角度来看，它只关心一件事：传进来的值最后能不能转换成我需要的 `T` 。

示例：

展开

自动换行

use std::net::Ipv4Addr;

fn ping<A>(addr: A)

where

A: Into<Ipv4Addr>,

{

let ipv4 = address.into();

println!("{:?}", ipv4);

}

这个函数真正想要的是一个 `Ipv4Addr` ，但它没有把参数类型写死成 `Ipv4Addr` ，而是写成了 `A: Into<Ipv4Addr>` 。于是只要某种类型能够转换成 `Ipv4Addr` ，它都可以作为参数传进来：

展开

自动换行

ping(Ipv4Addr::from(127, 0, 0, 1));

ping(\[127, 0, 0, 1\]);

ping(0x7f000001\_u32);

这些调用之所以都成立，是因为相关类型都可以转换成 `Ipv4Addr` 。

而 `From<T>` 更常扮演“泛型构造函数”的角色，也就是从目标类型的角度说： **我可以由某种值构造出来。**

示例：

展开

自动换行

use std::net::Ipv4Addr;

let addr1 = Ipv4Addr::from(\[127, 0, 0, 1\]);

let addr2 = Ipv4Addr::from(0x7f000001\_u32);

这种写法很自然，因为它强调的是“构造一个 `Ipv4Addr` ”。所以可以把两者的使用习惯简单记成：

● **函数收参数时，常用** `Into<T>`  
● **为某个类型提供转换构造能力时，常用** `From<T>`

另外，标准库还规定：如果实现了 `From<T> for U` ，就会自动得到 `Into<U> for T` 。所以给自定义类型写转换时，通常只需要实现 `From` ，不必再手动实现对应的 `Into` 。

## 5、价值和典型场景

`From` / `Into` 的价值，首先在于它们 **给类型转换提供了一套统一约定** 。过去如果没有这两个 trait，不同类型之间的转换往往只能靠各种专用构造函数或专用 trait，接口风格会很不统一。现在只要一个类型能自然地由另一个类型构造出来，就可以优先考虑实现 `From` 。

它们的另一个重要价值，是能让一些“受约束的值”安全地转换成“更一般的值”。例如， `String` 保证自己内部始终是合法 UTF-8，而一旦把它转换成 `Vec<u8>` ，这种 UTF-8 约束就不再存在了，得到的只是普通字节序列。由于 `into()` 会拿走原值的所有权，原来的 `String` 已经失效，因此转换后就可以自由处理这块字节缓冲区，而不会破坏任何现存的 `String` 值。

此外， `From` / `Into` 在错误处理里也非常重要。`?`**运算符之所以能把不同的错误类型自动统一到函数返回类型，背后依赖的正是** `From` 。

示例：

展开

自动换行

use std::error::Error;

type GenericError = Box<dyn Error + Send + Sync + 'static>;

type GenericResult<T> = Result<T, GenericError>;

fn parse\_i32\_bytes(b: &\[u8\]) -> GenericResult<i32> {

Ok(std::str::from\_utf8(b)?.parse::<i32>()?)

}

这里 `from_utf8` 可能返回 `Utf8Error` ， `parse::<i32>()` 可能返回 `ParseIntError` ，但函数最终返回的是 `Result<i32, Box<dyn Error>` 。之所以`?`能工作，是因为标准库提供了合适的 `From` 实现，即为 `Box<dyn Error>` 实现了通用的泛型 `From` ，能把这些具体类型自动转换成 `Box<dyn Error>` 。

展开

自动换行

impl<E> From<E> for Box<dyn std::error::Error + Send + Sync + 'static>

where

E: std::error::Error + Send + Sync + 'static,

{

fn from(err: E) -> Self {

Box::new(err)

}

}

把 `err: E` 放进堆里，变成 `Box<E>` ，然后再利用 trait object 强制转换成 `Box<dyn Error + Send + Sync + 'static>` 。

## 6、使用原则与注意事项

虽然 `From` / `Into` 很常用，但它们也有明确边界。最重要的一点是： `From` **和** `Into` **表达的是不会失败的转换** 。因为它们的方法签名直接返回目标类型，而不是 `Result` ，这意味着一点实现这种转换，应当保证它总能成功，而且语义应当明确、自然、合理。

例如：

`String -> Vec<u8>` 是合理的

`&str -> String` 是合理的

`[u8; 4] -> Ipv4Addr` 是合理的

但有些转换显然可能失败，就不适合放进 `From` / `Into` 。例如把一个很大的 `i64` 转成 `i32` ：

展开

自动换行

let huge = 2\_000\_000\_000\_000i64;

let smaller = huge as i32;

println!("{}", smaller);

这里如果直接做普通整数截断，结果往往不是我们真正想要的值。这种转换要么可能溢出，要么需要额外规则（例如饱和、截断、报错），因此不适合简单地实现为 `From<i64> for i32` 。这类可能失败的转换，应该交给 `TryFrom` / `TryInto` 处理，而不是 `From` / `Into` 。

## 十一、TryFrom和TryInTo

## 1、概述

前面讲过， `From` 和 `Into` 表示不会失败的值转换。如果一种转换天然总能成功，而且语义明确，就适合用它们来表达。

但现实中并不是所有转换都这么简单。有些转换可能会失败，最典型的例子就是 **大范围数值转小范围数值** 。例如， `i64` 能表示的数值范围远大于 `i32` ，因此并不是每个 `i64` 都能安全转换成 `i32` 。如果直接提供 `From<i64> for i32` ，那么就会让人误以为这种转换总是可靠的，而这显然不对。

因此，Rust 提供了 `TryFrom` 和 `TryInto` ，把“可能失败的转换”也纳入统一的 trait 约定中。它们可以看作是 `From` 和 `Into` 的容错版本： **转换成功时返回目标值，转换失败时返回错误信息** 。

## 2、定义

`TryFrom` 和 `TryInto` 的定义如下：

展开

自动换行

trait TryFrom<T>: Sized {

type Error;

fn try\_from(value: T) -> Result<Self, Self::Error>;

}

trait TryInto<T>: Sized {

type Error;

fn try\_into(self) -> Result<T, Sefl::Error>;

}

它们和 `From` 、 `Into` 的整体结构很像，只不过返回值不再是目标类型本身，而是： `Result<目标类型, 错误类型>` 。

因此，它们表达的含义分别是：

`TryFrom<T> for U` ： `U` 可以由 `T` 尝试转换而来，但转换可能失败

`TryInto<U> for T` ： `T` 可以尝试转换为 `U` ，但转换可能失败。

和 `From` / `Into` 一样，这两者也是配套使用的。通常只要实现了 `TryFrom<T> for U` ，标准库就会自动提供对应的 `TryInto<U> for T` 。因此，给自定义类型设计失败型转换时，通常也是 **优先实现** `TryFrom` 。

## 3、典型用途：处理可能失败的数值转换

`TryFrom` / `TryInto` 最常见的使用场景，就是处理可能越界的数值转换。

例如：

展开

自动换行

let huge = 2\_000\_000\_000\_000i64;

let smaller: i32 = huge.try\_into().unwrap\_or(i32::MAX);

println!("{}", smaller);

这里 `huge` 可能超出 `i32` 的表示范围，因此不能直接用 `From` / `Into` 。 `try_into()` 会返回一个 `Result<i32, _>` ：

● 如果转换成功，就得到对应的 `i32`  
● 如果转换失败，就进入错误分支

## 十二、ToOwned

## 1、概述

学习 `Clone` 知道， `clone()` 的作用是 **为一个值构造出它的副本** 。对于大多数普通类型，这很好理解： `String` 克隆出另一个 `String` ， `Vec<T>` 克隆出另一个 `Vec<T>` 。

但问题是，经常使用的 **“借用类型”本身** 并不是拥有型类型，例如：

`str`

`[T]`

`Path`

它们通常只能通过 **引用形式** 出现，比如 `&str` 、 `&[T]` 、 `&Path` 。这时候如果手里拿的是借用值，想得到一份对应的拥有型副本，就无法通过 `Clone` 做到。因为 `Clone` 的语义要求是克隆出同类型的值，而 `str` 、 `[T]` 这类动态大小类型本身不是普通意义上可直接返回的值，也不是我们真正想要的结果类型。例如对于 `&str` ，通常想要的是 `String` ；对于 `&[i32]` ，通常想要的是 `Vec<i32>` 。

为了解决上述问题，标准库提供了 `ToOwned` 特征。它表达的不是克隆出同类型的副本，而是 **从一个借用值，构造出它对应的拥有型值** 。

## 2、定义

`ToOwned` 的定义如下：

展开

自动换行

trait ToOwned {

type Owned: Borrow<Self>;

fn to\_owned(&self) -> Self::Owned;

}

这个定义最关键的就是 **关联类型** `Owned` ，它表示 **当前这个借用类型对应的拥有型是什么** 。同时要求 `Owned: Borrow<Self>` ， **这表示** `to_owned()` **返回的拥有型值，必须能再借回当前这个借用类型** 。这个约束非常自然，因为 `ToOwned` 的本意就是“为某个借用视图生成对应的拥有型版本”。

例如：

`str` 的拥有型是 `String` ，而 `String` 可以借成 `&str`

`[T]` 的拥有型是 `Vec<T>` ，而 `Vec<T>` 可以借成 `&[T]`

`Path` 的拥有型是 `PathBuf` ，而 `PathBuf` 可以借成 `&Path`

所以， `ToOwned` 的核心语义是： **从借用视图得到对应的拥有型对象，并且这个拥有型对象仍然保留原来那种借用语义。**

## 3、Cow

`ToOwned` 的一个应用场景就是 `Cow` 类型， `Cow` 是标准库中的 **一个枚举类型** ，全称表示 `Copy On Write` ，即写时复制。

`Cow` 的核心思想是：一开始先借用数据，只有在明确需要修改或取得所有权时，才将自己复制成对应的拥有型数据。

因此， `Cow` 同时兼顾两件事：

**如果只是读，就尽量不分配、不复制，直接借用原数据**

**如果后面必须改，或者必须变成拥有型，就再进行克隆**

`Cow` 定义在 `std::borrow` 中，它的定义可以简化理解为：

展开

自动换行

enum Cow<'a, B>

where

B: ToOwned +?Sized

{

Borrowed(&'a B),

Owned(B::Owned)

}

这里最关键的是两个变体：

`Borrowed(&'a B)` ：表示当前只是借用了一份数据

`Owned(B::Owned)` ：表示当前持有这份数据的拥有型副本

例如， `Cow<'a, str>` 的借用形态为 `&'a str` ，拥有形态为 `String` 。 `Cow` **之所以知道借用类型对应的拥有类型是什么，依赖的就是** `ToOwned` 。

示例1：

展开

自动换行

use std::borrow::Cow;

fn main() {

let s1: Cow<'\_, str> = Cow::Borrowed("hello");

let s2: Cow<'\_, str> = Cow::Owned(String::from("hello"));

println!("{}", s1);

println!("{}", s2);

}

示例2：

展开

自动换行

use std::borrow::Cow;

fn main() {

let mut s: Cow<'\_, str> = Cow::Borrowed("hello");

s.to\_mut().push\_str(" world");

println!("{}", s);

}

注意： `to_mut()` 是 `Cow` 的方法，它会做两件事，调用 `to_owned()` ，把 `"hello"` 复制成一个新的 `String` ，把自身从 `Borrowed` 变成 `Owned` ，然后再返回 `&mut String` 。

`Cow` 最大的价值是可以应用在在有时需要读，有时需要修改的场景中，避免不需要的分配和复制。

例如，编写一个函数，对字符串做按需修改：

展开

自动换行

use std::borrow::Cow;

fn ensure\_prefix(s: &str) -> Cow<'\_, str> {

if s.starts\_with("prefix\_") {

Cow::Borrowed(s)

} else {

Cow::Owned(format!("prefix\_{}", s))

}

}

使用时：

展开

自动换行

fn main() {

let s1 = ensure\_prefix("prefix\_hello");

let s2 = ensure\_prefix("world");

println!("{:?}", s1);

println!("{:?}", s2);

}

这里：

● 对 `"prefix_hello"` ，函数发现它已经满足要求，于是直接返回借用，不做任何分配  
● 对 `"world"` ，函数必须构造新字符串，于是返回拥有型值

`Cow` 的几个常见方法：

`into_owned()`

作用：无论当前是借用态还是拥有态，最终都变成拥有型值返回。

展开

自动换行

let c: Cow<'\_, str> = Cow::Borrowed("hello");

let s = c.into\_owned(); // String

`to_mut()`

作用：确保当前数据变成拥有型，并返回 **可变引用** 。

展开

自动换行

let mut c: Cow<'\_, str> = Cow::Borrowed("hello");

c.to\_mut().push\_str("!");

一个最常见的应用是 `String::from_utf8_lossy(&buf[..n])` 方法，它返回的就是 `Cow<'a, str>` 。

传入一个 `&[u8]` ，如果这段字节本身就是合法的UTF-8，那么它返回借用型： `Cow::Borrowed(&str)` ；如果这段字节里有非法UTF-8，它会把非法部分替换成 `�` ，然后返回拥有型： `Cow::Owned(String)` ，此时会新建 `String` 。因此，到底是 `Borrowed` 还是 `Owned` ，取决于输入数据是否是合法UTF-8。

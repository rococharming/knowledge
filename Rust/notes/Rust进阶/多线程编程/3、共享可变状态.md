# 一、概述

在并发程序中，除了使用 **消息传递（通道）** 这种思路外，还有一种非常常见的协作方式： **多个线程共同访问同一份数据** 。

例如，在多人游戏服务器中，很多线程可能同时处理玩家请求，而这些线程需要共同维护“等待进入下一局游戏的玩家列表”。这个列表有两个鲜明特点：

**共享的** ：多个线程都要访问它

**可变的** ：玩家会不断加入，满足条件后又会被取出并开始游戏

这就引出了并发编程中的一个核心主题： **共享可变状态** 。

Rust并不是不支持共享可变状态，而是要求你必须通过 **显式的并发原语** 来安全地完成它。标准库中处理这类问题的底层工具主要包括：

`Mutex` （互斥锁）

`RwLock` （读写锁）

`Condvar` （条件变量）

原子类型（atomic types）

# 二、互斥锁

## 1、互斥锁的概念

互斥锁（mutex）的作用是 **让多个线程访问某段共享数据时，必须轮流进行** 。

同一时刻：

**最多只能有一个线程进入临界区**

**其他线程如果也想进入，就必须等待**

这里的“临界区”是指安全访问共享数据的代码区域。

因此，互斥锁的核心价值在于：

**防止数据竞争**

数据竞争是指多个线程在没有同步的情况下，同时访问同一块内存，并且至少有一个在写。这会导致未定义行为，结果完全不可预测。

**防止操作无序交错**

即使底层读写没有直接撞车，不同线程的操作也可能以非常混乱的顺穿插执行。

3、 **支持不变条件**

受保护的数据往往需要始终满足某些规则。例如等待列表中的人数不能超过上限，一旦凑够8人，就必须立即取出并开局，列表中的玩家不应重复出现。使用互斥锁后，可以把这些规则看成临界区必须维护的 **不变条件（invariant）** 。每次进入临界区时，数据已经是合法的；离开前，也必须恢复为合法状态。

## 2、Mutx<T>

### （1）传统语言互斥锁的问题

在C++、Java等传统语言中，锁和数据是两个分离的概念。例如在C++中可能会写这样的注释：

`waitlingList` 这份数据必须由 `mutex` 保护

任何线程访问 `waitingList` 前都要先获取 `mutex` ，使用完后都必须释放 `mutex`

然而， **这只是约定，不是编译器能强制执行的规则** 。

程序员在编写代码时，很可能：

没有加锁就使用数据

加错锁

使用完数据忘记释放锁等

编译器不会阻止这些行为，因此代码执行就可能出问题。

Rust的设计思路则更加直接： **既然某份数据必须由锁保护，那么直接将数据放进锁里面** 。为此，标准库提供了 `Mutex<T>` 类型。

```rust
use std::sync::Mutex;

type PlayerId = u32;

const GAME\_SIZE: usize = 8;

type WaitingList = Vec<PlayerId>;

struct GameApp {

waiting\_list: Mutex<WaitingList>,

}
```
这里的 `waiting_list` 不是一个 `Vec` 加一个单独的锁，而是一个被 `Mutex` 包裹起来的 `Vec` 。

这意味着不拿到锁就拿不到里面的数据，访问路径被类型系统收紧了，因此Rust相比传统语言更加可靠。
### （2）Arc与Mutex的配合使用

`Mutex` 通常配合 `Arc` 使用，因为多个线程都要共享同一份数据（Mutex）的所有权，并且能修改内部数据。

示例：

```rust
let app = Arc::new(GameApp {

waiting\_list: Mutex::new(vec!\[\])

});
```
这里两者的分工非常清晰：

`Arc<T>` ：让多个线程共享同一个值的所有权

`Mutex<T>` ：让多个线程安全地共享同一份数据
### （3）lock()访问互斥锁中的数据

通过 `Mutex::lock()` 方法访问互斥锁中的数据：

```rust
let mut guard = app.waiting\_list.lock().unwrap();
```
这行代码做了几件事：

获取锁

如果当前没有其他线程持有锁，那么当前线程立即获取锁。如果锁已经被其他线程持有，当前线程会阻塞等待。

返回守卫值 `MutexGuard<T>`

`lock()` 不会直接把 `T` 交给你，而是返回一个守卫（guard）： `MutexGuard<WaitingList>` ，它是对内部数据的受控访问凭证。

守卫在作用域结束时自动释放锁

这一点非常重要。Rust通过RAII机制保证：

**守卫创建时表示锁已经拿到**

**守卫被丢弃时表示锁自动释放**

因此，临界区就是guard的生命周期。

示例：
```rust
use std::sync::{Arc, Mutex};

type PlayerId = u32;

const GAME\_SIZE: usize = 8;

type WaitingList = Vec<PlayerId>;

struct GameApp {

waiting\_list: Mutex<WaitingList>,

}

impl GameApp {

fn join\_waiting\_list(&self, player: PlayerId) {

let mut guard = self.waiting\_list.lock().unwrap();

guard.push(player);

if guard.len() == GAME\_SIZE {

let players = guard.split\_off(0);

drop(guard); // 提前释放锁

self.start\_game(players);

}

}

fn start\_game(&self, players: WaitingList) {

println!("Starting game with {:?}", players);

}

}

fn main() {

let app = Arc::new(GameApp {

waiting\_list: Mutex::new(vec!\[\])

});

}
```
说明：

这里注意当游戏满8人时，会先把玩家列表取出来，然后显式 `drop(guard)` 提前释放锁，最后再启动游戏。因为启动游戏可能是一个比较重的逻辑，没必要在整个开局过程中一直占着等待列表锁。

`join_waiting_list` 的方法签名只需要 `&self` ，但里面却调用了 `guard.push(player)` 修改了数据。这是因为 `Mutex` 提供了内部可变性。正常情况下，只有你已经拥有外层对象的 `&mut` ，才能进一步拿到里面字段的 `&mut` ， **但** `Mutex` **是一个特殊容器，它通过运行时加锁的方式，动态地保证内部数据的独占访问** 。
### （4）死锁

使用互斥锁时，一个非常经典，也非常危险的问题就是死锁（dead lock）。所谓死锁，就是线程因为彼此等待对方释放资源，导致所有相关线程都无法继续执行，程序就卡住了。

最简单的一种死锁情况，就是同一个线程重复去获取自己已经持有的同一把锁：

```rust
let mut guard1 = self.waiting\_list.lock().unwrap();

let mut guard2 = self.waiting\_list.lock().unwrap(); // 死锁
```
假设第一次调用 `self.waiting_list.lock()` 时成功获得了锁，那么第二次调用时，互斥锁会发现这把锁已经处于占用状态，于是当前线程就会阻塞，等待锁被释放。

但问题在于：持有这把锁的线程，正是当前这个被阻塞的线程自己。

这就形成了一个无法解除的等待关系：它在等自己释放锁，而它又必须先继续执行，才能走到释放锁的那一步。于是程序会永远卡在那里。

这说明 Rust 标准库中的 `Mutex` **不是递归锁** （recursive mutex）。也就是说，同一个线程不能在未释放前一次锁的情况下，再次锁住同一个 `Mutex` 。

上面这个例子中的问题看起来比较明显，但在实际项目里，死锁往往没有这么容易看出来。更常见的情况是：

两次 `lock()` 分散在不同的方法里

一个方法内部又调用了另一个方法

每个方法单独看没什么问题，但组合起来就会重复加锁。

**除了同一个线程重复加锁之外，死锁还可能以更隐蔽地形式出现，例如：多个线程相互等待** ：线程A持有锁1，等待锁2；线程B持有锁2，等待锁1。两边都不肯先释放，于是彼此永久阻塞。

Rust 的借用检查器无法帮助我们避免死锁，借用系统可以防止数据竞争，但死锁属于运行时的同步逻辑问题，不是类型系统能够静态消除的错误。

因此， **避免死锁更依赖良好的并发设计** 。实践中通常有几条重要原则：

临界区尽可能小

尽量减少嵌套加锁

多把锁时各个线程保持一致的加锁顺序
### （5）互斥锁中毒

前面已经发现， `Mutex::lock()` 的返回值不是直接的 `MutexGuard<T>` ，而是一个 `LockResult<MutexGuard<T>>` 。也就是说，它返回的是一个 `Result<MutexGuard<T>, PoisonErr<MutexGuard<T>>>` 。这背后有一个重要原因： **互斥锁可能发生了中毒（poisoning）** 。

所谓互斥锁中毒，是指某个线程在持有 `Mutex` 锁期间发生了 `panic` ，Rust就会把互斥锁标记为“ **已中毒** ”。之后，其他线程再去调用 `lock()` 试图获取这把锁时，就不会得到一个普通的成功结果，而是得到一个错误值。

这就是为什么经常会写这样的代码：

```rust
let guard = mutex.lock().unwrap();
```
这段写法的含义其实是：如果锁正常可用，就继续执行；如果这把锁因为别的线程 `panic` 而中毒，那么当前线程也立刻 `panic` ，从而把错误继续传播出去。

这种处理方式和 `join().unwrap()` 的思路很像：它不是悄悄忽略错误，而是明确告诉程序，既然另一个执行路径已经发生严重问题，那这里也不应假装一切正常。

Rust的 `panic` 不会导致未定义行为，因此即使某线程 `panic` 了其他线程也可以继续执行。但互斥锁中毒的问题在于：临界区的操作可能只进行了一半，这会导致数据的不一致的。因此，中毒防范的不是未定义行为，而是悄悄基于错误状态继续计算。

`mutex.lock().unwrap()` 是最常见的处理互斥锁中毒的方式，适合大多数不一致状态不可接受场景。但如果说即使某个线程由于 `panic` 导致互斥锁中毒，当前线程仍然希望检查或恢复里面的数据，也可以在匹配错误后，通过 `PoisonError::into_inner()` 取出守卫，强行继续访问里面的数据。
### （6）使用互斥锁的多消费者通道

前面提到过，Rust标准库的 `mpsc` 通道是多生产者、单消费者（multiple producer， single comsumer）的通道。

可以有多 `Sender<T>` 同时向通道发送消息，但一个通道内部只对应一个真正的 `Receiver<T>` ，这个 `Receiver<T>` 不能像 `Sender<T>` 那样直接克隆给多个线程共同消费。

因此，如果想把通道当成共享任务队列，并让线程池中的多个工作线程一起从中取任务，就会遇到限制：标准库中的 `mpsc::Receiver<T>` 本身不是多消费者设计，不能直接让多个线程同时持有并消费它。

不过，这并不意味着做不到“多个线程共同消费同一个接收端”。一种常见而直接的做法是：将 `Receiver<T>` 放到 `Mutex` 中，再通过 `Arc` 把这份受互斥锁保护的接收端共享给多个线程。

这样，多个线程虽然都能访问接受端，但每次真正调用 `recv()` 时，仍然只能有一个线程拿到锁并完成接收操作。

下面是一个典型的封装方式：

```rust
pub mod shared\_channel {

use std::sync::{Arc, Mutex};

use std::sync::mpsc::{channel, Sender, Receiver};

#\[derive(Clone)\]

pub struct SharedReceiver<T>(Arc<Mutex<Receiver<T>>>);

impl<T> Iterator for SharedReceiver<T> {

type Item = T;

fn next(&mut self) -> Option<Self::Item> {

let guard = self.0.lock().unwrap();

guard.recv().ok()

}

}

pub fn shared\_channel<T>() -> (Sender<T>, SharedReceiver<T>) {

let (sender, receiver) = channel();

(sender, SharedReceiver(Arc::new(Mutex::new(receiver))))

}

}
```
# 三、读写锁

## 1、概述

`std::sync` 中的另一种常用同步工具是读写锁 `RwLock<T>` 。

在服务器程序中，常常会有这样的一类数据： **启动时加载一次，运行期间大多数时候只读，但偶尔也需要更新** 。例如程序配置就是典型代表。多数线程只需要查询配置，只有少数情况才重新加载配置文件并更新内容。

这种场景当然可以使用 `Mutex<T>` ，但互斥锁的限制是：无论是读还是写，同一时刻只能有一个线程访问数据。如果大多数操作只是读取，那么这会带来不必要的等待，因为读取并不会导致数据竞争。这时候更适合用读写锁 `RwLock<T>` 。

## 2、RwLock<T>基本特点

`RwLock<T>` 与 `Mutex<T>` 一样，都是用来保护共享数据的同步原语。不同之处在于，它区分了读和写两种访问方式：

`read()` ：获取只读访问

`write()` ：获取可写访问

它的规则是：

可以同时有多个读者

但写着必须独占

只要写者存在，就不能再有读者和其他写者

所以， `RwLock<T>` 适合读多写少的场景。

## 3、使用方式

假设程序中有一个配置对象，需要被多个线程共享读取，偶尔重新加载：

```rust
use std::sync::RwLock;

struct AppConfig {

debug\_mode: bool,

max\_connections: usize,

}

struct AppState {

config: RwLock<AppConfig>

}

impl AppState {

fn debug\_mode(&self) -> bool {

let guard = self.config.read().unwrap();

guard.debug\_mode

}

fn set\_max\_connections(&self, value: usize) {

let mut config = self.config.write().unwrap();

config.max\_connections = value;

}

}
```
# 四、条件变量

## 1、概述

有些线程需要一直等待，直到某个条件成立后才能继续执行。例如：

主线程等待所有工作线程结束后再退出

工作线程在没有任务时进入等待，直到有新任务到来

某些并发流程要等到共享状态满足特定条件后才能继续

有时，标准库已经提供了现成的阻塞式API，例如 `JoinHandle::join()` 。但如果程序要等待的是某个由共享数据决定的条件，就需要自己建立等待机制，这时候可以使用条件变量 `Condvar` 。

`Condvar` 是 `std::sync` 中提供的条件变量类型，通常与 `Mutex` 配合使用。它的作用不是直接保护数据，而是让线程在“条件未成立”时睡眠，并在条件可能已经成立时被唤醒重新检查。

## 2、特点

条件变量总是围绕某个受 `Mutex` 保护的共享数据来使用。也就是说，线程真正关心的不是等一个通知，而是等某个条件变为真。

因此，条件变量通常和互斥锁一起出现：

`Mutex<T>` ：保护共享状态

`Condvar` ：在条件不满足时挂起线程，在条件满足时唤醒线程

例如，一个工作线程可能需要等待 **队列非空** 这个条件，才能继续取任务。

## 3、通知等待线程

当某个线程修改了共享状态，并使等待条件可能成立时，就应该调用：

```rust
let mut ready = lock.lock().unwrap();

\*ready = true;

condvar.notify\_one();
```
这段代码表示：

先获得互斥锁

把共享状态改为true

然后调用 `notify_one()` ，唤醒一个正在等待的线程

如果希望唤醒所有等待线程，也可以使用：
```rust
condvar.notify\_all();
```
需要注意的是， `notify_one()` 或 `notify_all()` 的含义并不是“条件一定已经满足”，而是：条件现在可能成立了，等待线程应该醒来重新检查。
## 4、等待条件成立

等待条件变量的标准写法是：

```rust
let mut ready = lock.lock().unwrap();

while!\*ready {

ready = condvar.wait(ready).unwrap();

}
```
这段代码的含义是：

先检查条件是否成立

如果条件不成立，就调用 `wait()` 进入等待

被唤醒后也能再次检查条件

只有条件真的成立时，才退出循环继续执行

这里必须使用 `while` ，而不是 `if` 。因为线程即使被唤醒，也不能直接假定条件已经满足，仍然必须重新检查共享状态。

注意，Condvar::wait()的签名比较特别，它会按值接收一个 `MutexGuard` ，并在返回时再交一个新的 `MutexGuard` 。

这表示它内部完成了这样几个步骤：

当前线程原本持有互斥锁

调用 `wait(guard)` 时，把守卫交给条件变量

条件变量先释放这把锁，再让线程睡眠

线程被唤醒后，重新获取锁

最后把新的守卫返回出来

示例：
```rust
use std::sync::{Arc, Condvar, Mutex};

struct SharedState {

job: Option<String>

}

fn main() {

let shared = Arc::new((

Mutex::new(SharedState {

job: None,

}),

Condvar::new()

));

let shared\_clone = Arc::clone(&shared);

let worker = std::thread::spawn(move || {

let (lock, condvar) = &\*shared\_clone;

let mut guard = lock.lock().unwrap(); // 获取锁

while guard.job.is\_none() {

guard = condvar.wait(guard).unwrap(); // 等待锁

}

let job = guard.job.take().unwrap();

drop(guard);

println!("Get job: {}", job);

});

// 故意休眠1s让子线程阻塞

std::thread::sleep(std::time::Duration::from\_secs(1));

let (lock, convar) = &\*shared;

let mut guard = lock.lock().unwrap();

guard.job = Some("hello".to\_string());

convar.notify\_one(); // 通知子线程

drop(guard); // 释放锁
```
# 五、原子类型

## 1、概述

原子类型是Rust提供的一类 **无锁（lock-free）** 并发原语，它们适合表示 **非常简单、需要被多个线程共享的小型状态** ，例如计数器、开关标志、状态标志、指针值等。

普通变量上的运算，如 `n += 1` ，在并发环境下通常不是一个不可分割的整体。它往往会分成“读取旧值、计算新值、写回结果”几个步骤，因此多个线程同时操作时可能发生冲突。原子类型基于原子操作， **原子操作是指在CPU级别执行的一系列不可分割或中断的指令** 。

当然， **原子类型适用于处理单个值的简单并发操作** ，并不能在所有场合都能替换锁。对于需要维护复杂不变条件的共享数据，通常仍然应该使用 `Mutex<T>` 、 `RwLock<T>` 。

## 2、Rust原子类型

Rust原子类型都位于 `std::sync::atomic` 模块中，提供了一系列存储布尔值、整数值和指针的原子类型。

`AtomicBool` ：布尔原子类型，常用于开关、状态标志。例如是否停止线程、是否完成初始化操作。

`AtomicI8` / `AtomicU8` / `AtomicI16` / `AtomicU16` / `AtomicI32` / `AtomicU32` / `AtomicI64` / `AtomicU64` / `AtomicIsize` / `AtomicUsize` ：整数布尔类型，常用于计数器、索引等。

`AtomicPtr<T>` ：指针原子类型，常用于底层并发结构，原子地读写裸指针

## 3、基本用法

以 `AtomicUsize` 为例，展示原子类型用法。

创建

```rust
use std::sync::atomic::{AtomicUsize, Ordering};

let counter = AtomicUsize::new(0);
```
读值
```rust
let v = counter.load(Ordering::SeqCst);
```
写值
```rust
counter.store(10, Ordering::SeqCst);
```
原子加减
```text
counter.fetch\_add(1, Ordering::SeqCst);

counter.fetch\_sub(1, Ordering::SeqCst);
```
交换值
```rust
let old = counter.swap(5, Ordering::SeqCst);

CAS操作
```
```text
counter.compare\_exchange(

expected,

new,

Ordering::SeqCst,

Ordering::SeqCst,

);
```
如果当前的值等于expected，设为new，并返回Ok(old\_value)

如果当前的值不等于expected，返回Err(actual\_value)

完整使用示例：
```rust
use std::sync::atomic::{AtomicUsize, Ordering};

fn main() {

// 创建值

let counter = AtomicUsize::new(0);

// 写值

counter.store(10, Ordering::SeqCst);

// 读值

println!("value: {}", counter.load(Ordering::SeqCst));

// 加

counter.fetch\_add(10, Ordering::SeqCst);

println!("value: {}", counter.load(Ordering::SeqCst));

// 交换

let old = counter.swap(30, Ordering::SeqCst);

println!("old: {}, new: {}", old, counter.load(Ordering::SeqCst));

// CAS操作

let res = counter.compare\_exchange(20, 100, Ordering::SeqCst, Ordering::SeqCst);

assert\_eq!(res, Err(30));

}
```
注意：

`counter` 没有声明为 `mut` ，但是可以修改内部值。因为原子类型和 `Mutex<T>` 和 `RwLock<T>` 一样，都属于内部可变性的一种形式。因此，它的方法通常接收的都是共享引用 `&self` 而不是 `&mut self` 。这意味着多个线程可以持有同一个原子值（配合 `Arc` ），并通过方法安全地修改它。

原子操作的方法通常都需要额外传入一个 `Ordering` 参数，它是一个枚举类型，例如 `Ordering::SeqCst` 。这个参数表示 **内存顺序** ，也就是系统应当如何看待 **这个原子操作与其他内存访问之间的先后关系** 。内存顺序会影响并发程序的正确性，也会影响优化空间。它是原子编程中最难理解的部分之一。在入门阶段，可以先记住一个实用原则：如果拿不准，就使用 `Ordering::SeqCst` 。
## 4、内存序

内存序（memory ordering）要解决的是 **本次原子操作与周围其他读写之间的顺序到底怎样被其他线程观察到** 。

假如线程A做两件事：

```text
data = 42;

flag.store(true,...);
```
线程B做的事是：
```rust
if flag.load(...) {

println!("{}", data);

}
```
从个人直觉看，好像只要B看到了 `flag` 为 `true` ，那么它就能看到 `data` 为42。

然而机器执行时，没有那么简单，因为：

编译器可能重排指令

CPU可能乱序执行

各个核心看到内存变化的时间也可能不同

所以如果没有额外约束，就可能出现这种现象：B已经看到了 `flag` 为 `true` ， 但还没看到 `data` 为42，于是打印出旧值。

因此，内存序本质上是在问：当一个线程看到某个原子操作时，它能顺带相信哪些其他内存写入也已经可见了？

`Ordering` 枚举提供了五种变体：

`Relaxed`

`Release`

`Acquire`

`AcqRel`

`SeqCst`

可以把它们理解成从 **最宽松到最严格** 的一组规则。

**（1）Relaxed**

这是 **最弱** 的一种，它只保证这次原子读写本身是原子的，不会读到半截数据，也不会形成数据竞争。 **但它几乎不保证和周围普通读写之间的先后顺序** 。

例如：
```rust
flag.store(true, Ordering::Relaxed);
```
它只保证“把 `flag` 写成 `true` ”这件事是原子的。但它 **不保证** 这个写入与前后的其他内存操作之间，会按你写代码的顺序被别的线程观察到。

**（2）Release**

`Release` 一般用于写入方。它的直觉是：在这次 `store(Release)` 之前，线程里已经做完的普通写入，不会被拖到这次发布之后。看一个典型模式：
```text
data = 42;

flag.store(true, Ordering::Release);
```
这里 `Release` 的含义可以理解成：我现在把 `flag = true` 这件事发布出去，并且承诺：在这之前我做过的写入，比如 `data = 42` ，也应该先准备好。

**（3）Acquire**

`Acquire` 一般用于读取方。例如：
```rust
if flag.load(Ordering::Acquire) {

println!("{}", data);

}
```
它的直觉是：如果我通过这次 `Acquire` 读取，看到了另一个线程 `Release` 发布出来的值，那么那个线程在发布之前做的写入，我这边也应该能看到了。所以它和 `Release` 是成对工作的。

因此，最经典的组合就是：

线程A：先写数据，再 `store(Release)`

线程B： `load(Acquire)` 看到这个值后，再读数据

**（4）AcqRel**

`AcqRel` 常用于那些 **既要读旧值、又要写新值** 的原子操作，比如：

`fetch_add` / `fetch_sub`

`compare_exchange` 的成功路径

`swap`

因为这些操作不是单纯读取，也不是单纯写入，而是两边都有。

它的意思就是：

在“读”的这边，具备 `Acquire` 效果

在“写”的这边，具备 `Release` 效果

例如：
```rust
counter.fetch\_add(1, Ordering::AcqRel);
```
这表示这个操作既不能让后续读写乱跑到前面，也不能让前面的写入乱跑到后面。

**（5）SeqCst**

这是最严格、也最容易理解的一种。所有使用 `SeqCst` 的原子操作，好像都进入了一个全局统一的先后顺序表里，而且每个线程都同意这张表。当然，底层实现没有这么朴素，但可以先这么理解。
## 5、简单例子：取消标志

原子类型的一个常见用途：是在线程之间共享一个简单的“是否取消”的标志。例如，一个工作线程正在执行长时间任务，主线程希望在必要时通知他提前停止。这时候可以使用 `AtomicBool` 。

示例：

```rust
use std::sync::Arc;

use std::sync::atomic::{AtomicBool, Ordering};

use std::thread;

use std::time::Duration;

fn main() {

let cancel\_flag = Arc::new(AtomicBool::new(false));

let cancel\_flag\_clone = Arc::clone(&cancel\_flag);

let worker\_thread = thread::spawn(move|| {

for i in 0..10 {

println!("worker step: {}", i);

if cancel\_flag\_clone.load(Ordering::SeqCst) {

println!("worker cancelled");

return;

}

thread::sleep(Duration::from\_millis(200));

}

println!("worker finished");

});

thread::sleep(Duration::from\_millis(500));

cancel\_flag.store(true, Ordering::SeqCst);

worker\_thread.join().unwrap();

}
```

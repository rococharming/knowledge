## 4、全局变量

## 一、静态变量

## 1、static

在Rust中，使用 `static` 定义静态变量，它是一个真正存在于内存中的全局对象。静态变量具有以下特征：

**在整个程序运行期间都存在**

**拥有唯一、稳定的内存地址**

生命周期是 `'static`

示例：

展开

自动换行

use std::thread;

static COUNTER: u32 = 0;

fn main() {

let handle = thread::spawn(||{

println!("spawned thread: {}", COUNTER);

println!("addr: {:p}", &COUNTER);

});

println!("main thread: {}", COUNTER);

println!("addr: {:p}", &COUNTER);

handle.join().unwrap();

}

说明：

`static` 的初始化表达式通常必须是 **编译期可求值** 的，如字面量、常量表达式或是 `const fn` 的调用结果。

多个线程可以只读地访问静态变量，静态变量的地址是唯一的。

## 2、static mut与不安全性

默认情况下，静态变量是不可变的。如果要声明可变静态变量，可以用 `static mut` 。

示例：

展开

自动换行

static mut COUNTER: u32 = 0;

但 `static mut` 几乎总是危险的，原因在于：它绕开了Rust对共享可变状态的常规保护机制。

如果多个线程同时访问并修改同一个 `static mut` ，就可能产生数据竞争。因此，Rust规定，对 `static mut` 的访问必须放在 `unsafe` 中。

示例：

展开

自动换行

static mut COUNTER: u32 = 0;

fn main() {

unsafe {

COUNTER += 1;

println!("{}", COUNTER);

}

}

不过，这里还要进一步说明一件事： `unsafe` 的含义不是“这样就安全了”，而是编译器把正确性责任交给你自己。也就是说，编译器不再替你保证这段代码满足 Rust 的别名规则和线程安全规则。

不过，上述的代码并不一定编译通过，在最新的 **Edition 2024** 规则下， `static mut` 的使用又被进一步收紧。像下面这种代码：

展开

自动换行

static mut COUNTER: u32 = 0;

fn main() {

unsafe {

println!("{}", COUNTER);

}

}

看起来只是“读一下”，但它实际上会隐式地产生对 `COUNTER` 的共享引用 `&u32` 。而对于 `static mut` ，这种引用是有问题的，因为编译器无法证明此时不会同时存在其他可变访问。

报错信息：

![[Image 100.png]]

创建了一个可变静态变量的共享引用是错误的。因为Edition 2024 更明确地限制了：不能随意从 `static mut` 直接创建 `&T` 或 `&mut T` 。

但如果确实要操作 `static mut` ，更底层、也更符合规则的方式是通过裸指针：

展开

自动换行

static mut COUNTER: u32 = 0;

fn main() {

unsafe {

let p: \*mut u32 = std::ptr::addr\_of\_mut!(COUNTER);

println!("{}", \*p);

\*p += 1;

println!("{}", \*p);

}

}

但即便如此， `static mut` 仍然不推荐作为日常方案。现代Rust中更推荐的替代方案是：原子类型、 `Mutex<T>` 、 `RwLock<T>` 等。这些类型都属于内部可变性 + 并发原语的组合，它们能在共享环境下安全地表达“可变全局状态”。

## 3、使用Mutex和原子类型

Mutex

展开

自动换行

use std::sync::Mutex;

use std::thread;

static COUNT: Mutex<u32> = Mutex::new(0);

fn main() {

let threads = 8;

let iters\_per\_thread = 5\_000;

let mut handles = vec!\[\];

for \_ in 0..threads {

handles.push(thread::spawn(move || {

for \_ in 0..iters\_per\_thread {

let mut guard = COUNT.lock().unwrap();

\*guard += 1;

}

}));

}

for handle in handles {

handle.join().unwrap();

}

println!("final\_value: {}", COUNT.lock().unwrap());

println!("expected\_value: {}", threads as u32 \* iters\_per\_thread as u32);

}

原子类型

因为共享状态是一个简单计数器，因此使用原子类型更轻量。

示例：

展开

自动换行

use std::sync::atomic::{AtomicU32, Ordering};

use std::thread;

static COUNT: AtomicU32 = AtomicU32::new(0);

fn main() {

let threads = 8;

let iters\_per\_thread = 5\_000;

let mut handles = vec!\[\];

for \_ in 0..threads {

handles.push(thread::spawn(move || {

for \_ in 0..iters\_per\_thread {

COUNT.fetch\_add(1, Ordering::SeqCst);

}

}));

}

for handle in handles {

handle.join().unwrap();

}

println!("final\_value: {}", COUNT.load(Ordering::SeqCst));

println!("expected\_value: {}", threads as u32 \* iters\_per\_thread as u32);

}

注意事项：

`const` 和 `static` 初始化要求表达式可以在编译期求值，这里虽然出现了 `Mutex::new()` 和 `AtomicU32::new()` 函数调用，但它们属于 `const fn` ，允许在编译期调用，编译期求值。

## 二、运行期初始化

## 1、问题引入

有些全局值虽然也想写成 `static` ，但它们的初始化过程不是纯编译期可完成的。例如：

展开

自动换行

use std::sync::Mutex;

static NAME: Mutex<String> = Mutex::new(String::from("Tom"));

fn main() {

println!("{}", NAME.lock().unwrap());

}

这段代码不能通过编译。问题不在 `Mutex::new()` ，而在于：

展开

自动换行

String::from("Tom")

这不是一个可以在编译期完成的常量表达式。 `String` 是拥有堆内存的类型，构造它通常涉及运行期分配。

所以问题的本质是：全局变量想要存在，但它的内部值必须等到运行时才能真正构造出来，这时就需要使用 **延迟初始化** 。

## 2、懒初始化的核心思想

所谓懒初始化是指： **变量不是在程序启动时立刻构造完成，而是在第一次使用它时才初始化，并且只初始化一次** 。

这特别适合下面这些全局值：

`String`

`Vec<T>`

`HashMap<K, V>`

需要运行时逻辑构造的额配置对象

初始化成本较高的全局资源

## 3、lazy\_static

在标准库还没有完善懒初始化方案工具时， `lazy_static` **crate曾经是非常主流的方案** 。

它通过 `lazy_static!`宏定义懒初始化的静态值：

展开

自动换行

use lazy\_static::lazy\_static;

use std::sync::Mutex;

lazy\_static! {

static ref NAME: Mutex<String> = Mutex::new(String::from("Hello"));

}

这里的 `static ref` 不是普通的Rust关键字，而是 `lazy_static!`宏提供的专用语法。它表达的含义不是一个普通的 `static T` ，而更接近于：

一个全局入口，第一次访问时完成初始化，此后始终返回同一个全局实例。 **从使用者角度看，可以把它当作一个可获得** `&'static T` **的全局值** 。

示例（全局日志字符串）：

展开

自动换行

use lazy\_static::lazy\_static;

use std::sync::Mutex;

use std::thread;

lazy\_static! {

static ref LOG: Mutex<String> = Mutex::new(String::from("start\\n"));

}

fn main() {

let threads = 3;

let iter\_per\_threads = 20;

let mut handles = vec!\[\];

for tid in 0..threads {

handles.push(thread::spawn(move || {

for i in 0..iter\_per\_threads {

{

let mut guard = LOG.lock().unwrap();

guard.push\_str(&format!("thread: {}, i: {}\\n", tid, i));

}

thread::yield\_now();

}

}))

}

for handle in handles {

handle.join().unwrap();

}

println!("{}", LOG.lock().unwrap());

}

说明： `lazy_static!`的开销

`lazy_static!` 的底层通常依赖“一次性初始化”机制。初始化完成后，后续每次访问仍然需要经过一次很短的“是否已初始化”的检查路径，因此会比普通 `static` 多出极小开销。不过在大多数应用场景中，这点开销通常可以忽略。

这里重点关注多线程环境下常用的： `OnceLock` 和 `LazyLock` 。

### （1）OnceLock<T>

`OnceLock<T>` 表示一个最多初始化一次的存储槽。它的特点是：

初始为空

初始化方式：

`set(value)` ：外部显式写入

`get_or_init(f)` ：按需初始化

并发下，初始化只会成功发生一次

常用接口：

`const fn new() -> OnceLock<T>`

`fn get(&self) -> Option<&T>`

`fn set(&self, value: T) -> Result<(), T>`

`fn get_or_init<F>(&self, f: F) -> &T`

示例：

展开

自动换行

use std::sync::Mutex;

use std::sync::OnceLock;

static NAME: OnceLock<Mutex<String>> = OnceLock::new();

fn name() -> &'static Mutex<String> {

NAME.get\_or\_init(|| {

Mutex::new(String::from("Tom"))

})

}

这里的意思是：

`NAME` 一开始为空

第一次调用 `name()` 时完成初始化

以后所有调用都返回同一个 `Mutex<String>`

完整示例

展开

自动换行

use std::sync::Mutex;

use std::thread;

use std::sync::OnceLock;

static LOG: OnceLock<Mutex<String>> = OnceLock::new();

fn log() -> &'static Mutex<String> {

LOG.get\_or\_init(|| Mutex::new(String::from("start\\n")))

}

fn main() {

let threads = 3;

let iter\_per\_threads = 20;

let mut handles = vec!\[\];

for tid in 0..threads {

handles.push(thread::spawn(move || {

for i in 0..iter\_per\_threads {

{

let mut guard = log().lock().unwrap();

guard.push\_str(&format!("thread: {}, i: {}\\n", tid, i));

}

thread::yield\_now();

}

}))

}

for handle in handles {

handle.join().unwrap();

}

println!("{}", log().lock().unwrap());

}

### （2）LazyLock<T>

`LazyLock<T>` 表示一个 **携带初始化闭包的惰性值** 。它和 `OnceLock<T>` 的区别在于：

`OnceLock<T>` 更像“一个空槽，之后往里放值”

`LazyLock<T>` 更像“创建时就把初始化方式一起带上，等第一次访问时自动执行”

初始化模型：

构造时就给出初始化闭包

第一次使用时执行闭包

并发下只允许一个线程执行初始化

初始化完成后，后续访问都共享同一个值

示例：

展开

自动换行

use std::collections::HashMap;

use std::sync::LazyLock;

static MAP: LazyLock<HashMap<u32, &'static str>> = LazyLock::new(|| {

let mut m = HashMap::new();

m.insert(1, "foo");

m.insert(2, "bar");

m.insert(3, "baz");

m

});

这个例子说明：即使初始化逻辑比较复杂，也可以很自然地写在闭包里。

完整示例：

展开

自动换行

use std::sync::Mutex;

use std::thread;

use std::sync::LazyLock;

static LOG: LazyLock<Mutex<String>> = LazyLock::new(|| {

Mutex::new(String::from("start\\n"))

});

fn main() {

let threads = 3;

let iter\_per\_threads = 20;

let mut handles = vec!\[\];

for tid in 0..threads {

handles.push(thread::spawn(move || {

for i in 0..iter\_per\_threads {

{

let mut guard = LOG.lock().unwrap();

guard.push\_str(&format!("thread: {}, i: {}\\n", tid, i));

}

thread::yield\_now();

}

}))

}

for handle in handles {

handle.join().unwrap();

}

println!("{}", LOG.lock().unwrap());

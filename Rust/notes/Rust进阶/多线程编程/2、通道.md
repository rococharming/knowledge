## 2、通道

## 一、消息通道

多线程之间有多种方式可以共享、传递数据，一种安全并发的方式就是 **消息传递** （message passing）。为了实现消息传递并发，Rust 标准库提供了一个 **通道** （channel）实现。通道是一个通用的编程概念，表示 **数据从一个线程发送到另一个线程** 。

通道通常由两部分组成：

发送端（sender）

接收端（receiver）

代码中的一部分调用发送端的方法发送数据，另一部分调用接收端的方法接收数据。

## 二、mpsc通道

## 1、概念

标准库提供了通道 `std::sync::mpsc` 。其中的 `mpsc` 是 `Multiple Producer, Single Consumer` 的缩写，即多生产者单消费者。

通过 `std::sync::mpsc::channel()` 创建一个异步通道，返回一对 `(Sender<T>, Receiver<T>)` ，分别表示 **发送端和接收端** 。

原型：

展开

自动换行

fn channel<T>() -> (Sender<T>, Receiver<T>);

`Sender` 可以通过 `clone()` 方法克隆，这样多个线程可以同时像同一个线程发送消息，但 `Receiver` 只能有一个，因此该通道称为多生产者单消费者模型。

发送线程通过 `Sender::send(msg)` 发送消息，接收线程通过 `Receiver::recv()` 接收消息。

错误情况：

如果接收端的 `Receiver` 被丢弃，那么发送端 `Sender` 调用 `send()` 方法会失败，返回 `SendError<T>` 。注意， **发送失败时会将原本要发送的数据一并返回出来** 。

如果所有发送端的 `Sender` 都被丢弃并且通道中也没有剩余消息，那么接收端的 `Receiver` 调用 `recv()` 方法会失败，返回 `RecvError` 。

## 2、特点

通过 `Sender` 发送到通道中的数据，会按照 **发送顺序** 被 `Receiver` 接收到。更准确来说： **同一个发送者发送的消息顺序会保留** 。如果存在多个发送者（克隆出来），那么不同发送者之间消息的最终到达顺序是不确定的。

`channel` 创建的是 **异步通道** 。所谓异步，是指 `send` **通常情况下不会因为接收者未调用** `recv` **而阻塞当前线程** 。 **发送的数据会先进入通道内部缓冲区** 。因此，该通道可看成 **逻辑上的无界缓冲区** ，只要系统内存允许，发送就可以继续进行。

当前通道中没有消息可读取且至少还有一个 `Sender` 存活时， `Receiver` 调用 `recv` 就会阻塞。一旦消息到达， `recv` 会立即返回该消息。

## 3、基本使用

### （1）典型用法

展开

自动换行

use std::sync::mpsc;

use std::thread;

fn main() {

let (tx, rx) = mpsc::channel();

thread::spawn(move || {

let msg = String::from("hello");

tx.send(msg).unwrap();

// println!("Send msg: {}", msg); 错误，因为msg已经转移

});

// 主线程此时可以做一些其他工作

let msg = rx.recv().unwrap();

println!("Receive msg: {}", msg);

}

说明：

由于历史原因， `tx` 和 `rx` 通常分别作为发送端（transmitter）和接收端（receiver）的缩写。

发送端 `tx` 使用 `send` 方法将数据发送出去。 `send` 方法返回一个 `Result<T, SendError<T>>` ：

如果发送成功，返回 `Ok(())`

如果接收端已经被丢弃，则返回错误，因为消息已经没有接收目标了

**特别注意：所有权转移**

所有权在消息传递中扮演着非常重要的角色。上面的发送端 `tx` 将 `msg` 发送出去后， `msg` 就不能再继续使用了。这是因为 `send` 会把值的 **所有权** 转移给接收端。这样设计可以避免多个线程同时访问同一份数据带来的不一致问题。

也就是说：

`send(msg)` 之后，发送方不再拥有 `msg`

接收方拿到的是这份数据的所有权

这正是 Rust 保证线程安全的重要方式之一。

### （2）try\_recv

接收端 `rx` 常用的接收方法由两个： `recv()` 和 `try_recv()` 。

`recv` 会阻塞当前线程，直到通道中有消息可取。它返回 `Result<T, RecvErr>` ：

有消息时返回 `Ok(T)`

如果所有发送端的 `Sender` 都已经关闭，并且通道也没有剩余消息，则返回 `Err`

`try_recv()` 不会阻塞，它立即返回。返回值也是 `Result<T, TryRecvError>` ：

有消息时返回 `Ok(T)`

没有消息且发送端仍然存在时，返回 `Err(TryRecvError::Empty)`

所有发送端都已经关闭且没有消息时，返回 `Err(TryRecvError::Disconnected)`

如果线程在等待消息期间还有别的事情要做，那么 `try_recv`

很有用： **可以在循环中周期性检查消息，其余时间做其他工作** 。

示例：

展开

自动换行

use std::sync::mpsc;

use std::thread;

use std::time::Duration;

fn main() {

let (tx, rx) = mpsc::channel();

thread::spawn(move || {

thread::sleep(Duration::from\_secs(5));

let msg = String::from("hello");

tx.send(msg).unwrap();

});

loop {

match rx.try\_recv() {

Ok(msg ) => {

println!("received: {}", msg);

break;

}

Err(mpsc::TryRecvError::Empty) => {

println!("暂时还没有消息，做一些其他事情...");

thread::sleep(Duration::from\_secs(1)); // 模拟做其他事

}

Err(mpsc::TryRecvError::Disconnected) => {

println!("发送端已经关闭，后续不会再有消息");

break;

}

}

}

}

### （3）for循环接收

对于接收端的 `Receiver` ，可以使用 `for` 循环阻塞地从迭代器中接收消息。

当所有发送端被 `drop` ，并且通道中的消息已经被取完时， `for` 循环自动结束。

示例：

展开

自动换行

use std::sync::mpsc;

use std::thread;

fn main() {

let (tx, rx) = mpsc::channel();

thread::spawn(move || {

let msgs = vec!\[

String::from("hello"),

String::from("world"),

String::from("rust"),

\];

for msg in msgs.into\_iter() {

tx.send(msg).unwrap();

}

// 这里tx将被drop

});

for msg in rx {

println!("{}", msg);

}

}

### （4）多发送者单接收者

通过对 `Sender` 调用 `clone()` 可以克隆一个独立的发送端，这样多个线程可以同时像单个线程发送消息，实现多生产者单消费者模型。

示例：

展开

自动换行

use std::sync::mpsc;

use std::thread;

fn main() {

let (tx, rx) = mpsc::channel();

let tx\_clone = tx.clone();

thread::spawn(move || {

let msgs = vec!\[

String::from("hello"),

String::from("world"),

String::from("rust"),

\];

for msg in msgs {

tx.send(msg).unwrap();

}

});

thread::spawn(move || {

let msgs = vec!\[

String::from("你好"),

String::from("世界"),

String::from("Rust"),

\];

for msg in msgs {

tx\_clone.send(msg).unwrap();

}

});

for msg in rx {

println!("{}", msg);

}

结果：

![[Image 92.png]]

需要注意： **多线程发送时，消息到达顺序通常是不确定的** ，这取决于线程调度顺序。因此这里不要假定输出顺序一定固定。但同一个线程发送的消息一定是有顺序的。

### （5）传输多种类型的数据

一个消息通道只能传输同一种类型的数据。如果想要传输多种类型，可以有两种方式：

为每一种类型创建一个通道

使用枚举统一不同类型的数据

通常来说，使用枚举会更加自然一些。

示例：

展开

自动换行

use std::sync::mpsc;

use std::thread;

#\[derive(Debug)\]

enum Message {

Num(i32),

Text(String),

}

fn main() {

let (tx, rx) = mpsc::channel();

thread::spawn(move || {

let msgs = vec!\[

Message::Num(10),

Message::Text("Hello".to\_string()),

\];

for msg in msgs {

tx.send(msg).unwrap();

}

});

for msg in rx {

println!("{:?}", msg);

}

}

结果：

## 4、通道关闭的条件

通道的关闭要分情况理解：

对于发送端来说：如果接收端已经被丢弃，那么再度调用 `send()` 返回 `Err(SendErr(T))`

对于接收端来说：如果所有发送端杯丢弃且通道中也没有剩余消息，那么再度调用 `recv()` 返回 `Err(RecvErr)` ；而 `for val in rx` 这样的迭代也会结束。

错误代码：

展开

自动换行

use std::sync::mpsc;

use std::thread;

fn main() {

let (tx, rx) = mpsc::channel();

for i in 0..3 {

let tx\_clone = tx.clone();

thread::spawn(move || {

tx\_clone.send(i).unwrap();

});

}

for num in rx {

println!("{}", num);

}

println!("Finish recv");

}

结果：

![[Image 94.png]]

可以看到程序没有如期望般打印 `Finish recv` ，而是卡死了。显然此时阻塞在 `for num in rx` 。

这是因为尽管子线程中的 `tx_clone` 会在各自线程结束后被丢弃，但主线程里最开始创建的 `tx` 还一直存在，因此对于接收端来说，只要还有一个发送端活着， `for val in rx` 就会继续等待后续消息，而不会结束。

因此在接收消息前需要主线程需要主动丢弃 `tx` ：

展开

自动换行

use std::sync::mpsc;

use std::thread;

fn main() {

let (tx, rx) = mpsc::channel();

for i in 0..3 {

let tx\_clone = tx.clone();

thread::spawn(move || {

tx\_clone.send(i).unwrap();

});

}

drop(tx); // 很关键：丢弃主线程手里的发送端

for val in rx {

println!("Got: {}", val);

}

println!("Finish Recv");

}

结果：

## 5、异步通道和同步通道

### （1）异步通道

`std::sync::mpsc::channel()` 创建的通道是 **异步通道** （无界通道）。对于异步通道，发送端 `Sender` 调用 `send()` 不会因为接收端 `Receiver` 没有调用 `recv()` 就阻塞。这是因为发送端会先把消息放进通道里，如果系统内存没有限制，理论上这个通道就是无界的。

示例：

展开

自动换行

use std::sync::mpsc;

use std::thread;

use std::time::Duration;

fn main() {

let (tx, rx) = mpsc::channel();

thread::spawn(move || {

println!("Before send...");

tx.send(100).unwrap();

println!("After send...");

});

// 故意休眠

thread::sleep(Duration::from\_secs(1));

println!("{}", rx.recv().unwrap());

}

结果：

### （2）同步通道

然而，虽然异步通道很方便，但是也有风险： **如果生产速度远远大于消费速度，消息可能不断累积，造成内存增长** 。

因此， `mpsc` 还提供了 **同步通道（也称有界通道）** ，通道 `std::sync::mpsc::sync_channel(bound)` 创建。

函数原型：

展开

自动换行

fn sync\_channel<T>(bound: usize) -> (SyncSender<T>, Receiver<T>);

可以看到，与 `channel()` 不同， `sync_channel` 需要指定一个缓冲区的大小，同时返回的发送端是 `SyncReceiver` 类型。

和异步通道相同的是， `SyncSender` 发送的所有数据将按顺序被 `Receiver` 端接收。同时，如果通道中没有消息且 `SyncSender` 没有被丢弃，则 `Receiver` 接收消息时会阻塞。

同步通道具有缓冲区：

当缓冲区未满时，则 `SyncSender` 调用 `send` 立即完成

当缓冲区已满，则后续发送会阻塞，直到接收端取出一条消息腾出空间

如果 **缓冲区大小指定为0** ，则该通道变为一个零容量通道，发送端和接收端必须一一碰头，这种模式也称为 **会合通道** 。

#### 1）零容量同步通道

展开

自动换行

use std::{sync::mpsc, thread, time::Duration};

fn main() {

let (tx, rx) = mpsc::sync\_channel(0); // 零容量同步通道

thread::spawn(move || {

println!("Before send...");

tx.send(1).unwrap(); // 会阻塞，直到有接收者真正来接收

println!("After send...");

});

for i in 0..5 {

thread::sleep(Duration::from\_secs(1));

println!("main thread wait: {}", i);

}

println!("Start recv");

println!("Recv: {}", rx.recv().unwrap());

}

结果：

![[Image 97.png]]

结果中会看到：

`Before send...` 很快打印

`After send...` 会等到主线程真正执行接收后才打印

这说明 `sync_channel(0)` 下，发送和接收必须同时配合。

#### 2）带缓冲区同步通道

展开

自动换行

use std::{sync::mpsc, thread, time::Duration};

fn main() {

let (tx, rx) = mpsc::sync\_channel(1); // 缓冲区容量为 1

thread::spawn(move || {

println!("send first...");

tx.send(1).unwrap(); // 第一次发送不会阻塞，因为缓冲区还有空间

println!("send second...");

tx.send(2).unwrap(); // 第二次发送会阻塞，因为缓冲区满了

println!("finish sending");

});

thread::sleep(Duration::from\_secs(2));

println!("recv = {}", rx.recv().unwrap()); // 取走 1，腾出空间

thread::sleep(Duration::from\_secs(2));

println!("recv = {}", rx.recv().unwrap());

}

在实际项目中：

如果希望发送尽量流畅，考虑使用异步通道

如果希望通过背压（backpressure）限制消息堆积，避免内存无限增长，更适合使用带缓冲区的同步通道。

# 一、启动与联结线程

## 1、创建新线程

Rust 通过 `std::thread::spawn()` 函数创建新线程。 `spawn` 需要传入一个待执行的函数或闭包，实际开发中更常用闭包。

示例：

```rust
use std::thread;

use std::time::Duration;

fn main() {

thread::spawn(|| {

for i in 0..10 {

println!("spawn thread: {}", i);

thread::sleep(Duration::from_secs(1));

}

});

for i in 0..5 {

println!("main thread: {}", i);

thread::sleep(Duration::from_secs(1));

}

}
```
结果：

![[Image 82.png]]

注意事项：

主线程结束时，整个进程会终止，因此其他仍在运行的线程也会被一并终止。所以如果不等待子线程，子线程的输出可能会被中断。（上述子线程后续输出就中断了）

`std::thread::sleep()` 的作用是让当前线程休眠一段时间。在这段时间内，操作系统调度器可以去调度其他线程执行。因此，即使在单核 CPU 上，上述代码也仍然可以表现出并发执行的效果。

线程的执行顺序是不确定的，输出结果通常具有随机性，不能依赖具体的打印顺序。
## 2、等待线程结束

前面的例子已经看到，如果主线程先结束，则其他子线程也会被迫结束。因此，大多数情况下，主线程需要等待子线程执行完成。

实际上，调用 `std::thread::spawn()` 时，会返回一个 `JoinHandle<T>` 类型的线程句柄，其中 `T` 是子线程闭包的返回值类型。调用者线程可以通过 `JoinHandle::join()` 阻塞地等待对应子线程结束，并获取其返回值。

示例：

```rust
use std::thread;

use std::time::Duration;

fn main() {

let handle = thread::spawn(|| {

for i in 0..10 {

println!("spawn thread: {}", i);

thread::sleep(Duration::from_secs(1));

}

});

handle.join().unwrap();

for i in 0..5 {

println!("main thread: {}", i);

thread::sleep(Duration::from_secs(1));

}

}
```
结果：

![[Image 83.png]]

主线程会先等待子线程全部执行完毕，然后再继续向下执行。

注意： `join()` 返回的 `Result<T, Box<dyn Any + Send + 'static>>` ，返回 `Result` 的原因是：子线程执行过程中可能发生 `panic` ：

如果子线程正常结束，则 `join()` 返回 `Ok(T)` ，其中 `T` 就是子线程的返回值类型；

如果子线程 `panic` ，则返回 `Err(...)` 告知调用线程，再由调用线程决定怎么处理，而不是直接让整个程序 `panic` 。
## 3、spawn对闭包的要求

`thread::spawn()`

对传入的闭包有一定的要求：

闭包要能够在线程间安全转移，因此实现 `Send`

闭包需要满足 `'static` 生命周期要求，因此新线程的执行时间可能比创建它的作用域更长

# 二、线程panic的行为

在Rust中，线程 `panic` 的行为取决于是哪一个线程发生了 `panic` 。

## 1、主线程panic

当主线程发生 `panic` 时，默认行为是整个进程直接终止。

示例：

```rust
use std::thread;

use std::time::Duration;

fn main() {

thread::spawn(|| {

for i in 0..5 {

thread::sleep(Duration::from_secs(1));

println!("spawn thread: {}", i);

}

});

thread::sleep(Duration::from_secs(2));

panic!("main thread panic!");

}
```
结果：

![[Image 84.png]]

主线程 `panic` 会导致整个进程退出，进程一旦退出，其他所有线程都会结束。
## 2、子线程panic

当子线程发生 `panic` 时，默认不会直接导致整个进程崩溃。子线程的 `panic` 信息会保存在它对应的 `JoinHandle` 中，只有在其他线程对该 `JoinHandle` 调用 `join()` 时，才会感知到该错误。

示例：

```rust
use std::thread;

use std::time::Duration;

fn main() {

let _handle = thread::spawn(|| {

thread::sleep(Duration::from_secs(2));

panic!("spawn thread panic!");

});

for i in 0..5 {

println!("main thread: {}", i);

thread::sleep(Duration::from_secs(1));

}

}
```
结果：

![[Image 85.png]]

即使子线程发生了 `panic` ，主线程仍然会继续执行。

注意：

子线程 `panic` 时，通常仍然会在控制台打印 panic 信息。

如果主线程既不 `join()` ，也不依赖该线程的执行结果，那么主线程本身不会因此主动崩溃

现在对子线程进行 `join()` ：
```rust
use std::thread;

use std::time::Duration;

fn main() {

let handle = thread::spawn(|| {

thread::sleep(Duration::from_secs(2));

panic!("spawn thread panic!");

});

handle.join().unwrap();

for i in 0..5 {

println!("main thread: {}", i);

thread::sleep(Duration::from_secs(1));

}

}
```
结果：

![[Image 86.png]]

现在能看到控制台打印了子线程和主线程都 `panic` 的信息。说明，通过 `join()` ，子线程的 `panic` 传递给了主线程，而 `unwrap()` 遇到 `Err` 后导致主线程也发生 `panic` ，最终程序终止。
# 三、线程间移动所有权

先看下面例子：

```rust
use std::thread;

fn main() {

let s = String::from("hello");

let handle = thread::spawn(|| {

println!("spawn thread: s = {}", s);

});

handle.join().unwrap();

println!("main thread: s = {}", s);

}
```
编译报错：闭包可能比当前函数活得更久，但它借用了当前函数拥有的变量 `s` 。这是因为当s离开作用域了，闭包还在运行，此时会产生悬垂引用的问题。

![[Image 87.png]]

尽管在这个例子中，主线程创建好子线程后立刻调用 `join()` 等待子线程结束，看起来似乎是安全的，但编译器不会根据这种“看起来没问题”的运行顺序做特殊放行。

在一般情况下，主线程可能在子线程执行期间修改 `s` ，释放 `s` ，或者提前结束作用域，这会导致子线程中对 `s` 的引用失效，进而产生悬垂引用或违反借用规则。 **为了从根源上杜绝这类问题，Rust 要求传入新线程的数据必须满足更严格的条件** 。 **因此，在Rust中，需要使用** `move` **关键字将外部变量的所有权转移到新线程中** 。

示例：
```rust
use std::thread;

fn main() {

let s = String::from("hello");

let handle = thread::spawn(move || {

println!("spawn thread: s = {}", s);

});

handle.join().unwrap();

}
```
注意：

使用 `move` 后， `s` 的所有权被转移到子线程中，因此主线程后续不能再使用 `s` 。

● `thread::spawn()` 对闭包有 `'static` 要求，本质上是为了保证新线程中使用的数据在线程生命周期内始终有效，即 **闭包捕获的所有数据，必须不依赖某个更短生命周期的借用** 。使用 `move` 往往是满足这一要求的常见方式之一。
# 四、Arc<T>共享所有权

多线程场景下往往需要共享数据，现在问题来了：使用 `move` 闭包将 `s` 的所有权转移进了新线程，主线程仍然想使用 `s` 怎么办呢?

前面学习过的 `Rc<T>` 智能指针可以让同一份数据拥有多个所有者， 但只能用于单线程场景，因为：

`Rc<T>` 没有实现 `Send` 和 `Sync` 特征，不能安全地在线程间传递或共享。

`Rc<T>` 的引用计数增减不是原子操作，在多线程环境下会产生数据竞争。

而 `Arc<T>` （Atomic Reference Counted）是 `Rc<T>` 的多线程版本，它的引用计数操作是原子的，因此可以安全地在多个线程之间共享同一份数据。

示例：

```rust
use std::thread;

use std::sync::Arc;

fn main() {

let s = Arc::new(String::from("hello"));

let s_clone = Arc::clone(&s);

let handle = thread::spawn(move || {

println!("spawn thread: s = {}", s_clone);

});

println!("main thread: s = {}", s);

handle.join().unwrap();

}
```
结果：

![[Image 88.png]]

主线程和子线程都可以访问同一份字符串数据。

注意：

`Arc<T>` 只解决“共享所有权”的问题，它并不自动提供“可变共享”。更准确地说， `Arc<T>` 允许多个线程共享同一个值；至于这个值能否被修改，取决于 `T` 本身是否支持线程安全的内部可变性。

如果要在多线程中共享并修改数据，通常需要将 `Arc<T>` 与 `Mutex<T>` 、 `RwLock<T>` 等同步原语配合使用，例如 `Arc<Mutex<T>>` 。
# 五、互斥器Mutex<T>

## 1、互斥器的概念

互斥器（Mutex）可以保证 **同一时刻，只允许一个线程访问被保护的数据** 。

线程如果想访问互斥器中的数据，首先必须先获取锁。锁是互斥器的一部分，它记录了 **当前哪一个线程拥有对这份数据的独占所有权** 。

当线程需要访问互斥器中的数据时，过程如下：

获取锁。如果锁当前被其他线程持有，则当前线程会阻塞等待。

获得锁后，访问或修改被保护的数据

使用完数据后，释放锁，以便其他线程继续竞争这把锁

## 2、Mutex<T>的使用

Rust使用 `Mutex<T>` 作为互斥锁来保护共享数据。使用时需要将 `Mutex` 引入当前作用域：

```rust
use std::sync::Mutex;
```
为了先理解其基本用法，这里先在单线程环境下演示:
```rust
use std::sync::Mutex;

fn main() {

// 定义互斥锁

let m = Mutex::new(5);

// 访问互斥锁

{

let mut guard = m.lock().unwrap();

*guard += 1;

}

println!("data: {}", m.lock().unwrap());

}
```
结果：

![[Image 89.png]]

注意：

使用 `Mutex::new()` 创建一个互斥锁，内部包裹一个 `T` 类型的数据。

使用 `Mutex::lock()` 获取锁。如果锁被其他线程持有，则当前线程会阻塞等待。

`Mutex::lock()` 返回的是 `LockResult<MutexGuard<T>>` ，也就是一个 `Result` 。这是因为如果某个线程在持有锁期间发生了 `panic` ，那么这个互斥锁会进入 **中毒（poisoned）** 状态。此后其他线程再去 `lock()` 就返回 `Err` ，提示该锁保护的数据可能处于不一致状态。

若正常获取到锁，对 `lock()` 的结果调用 `unwrap()` 后，得到的是一个 `MutexGuard<T>` 。它本质上也是一个智能指针：

实现了 `Deref` 和 `DerefMut` ，因此可以像操作普通引用一样操作内部数据

实现了 `Drop` ，当 `guard` 离开作用域时，会自动释放锁。因此上面用花括号显式限制了 `guard` 的作用域，方便尽早释放锁。

`Mutex<T>` 提供内部可变性。即使 `Mutex` 本身没有声明为 `mut` ，也仍然可以通过锁来修改其内部的数据。

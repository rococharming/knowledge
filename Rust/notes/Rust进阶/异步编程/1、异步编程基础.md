# 一、异步编程概述

## 1、并发编程模型

异步编程是一种并发编程模型，目前主流语言基本都支持，但实现方式各不相同。它的核心目标是： **用更少的线程承载更多的并发任务（尤其是I/O密集性任务）** ，通过 **等待时让出执行权** 的方式提高资源利用率。

现代并发编程模型大致可分为如下几类：

### （1）OS线程（Thread / 1.1线程）

**OS线程由操作系统调度，每个线程有独立的栈和调度状态** 。对开发者来说直观、简单，一般不改变同步代码的结构。

优点：

编程模型自然：同步/阻塞方式均可，无特殊语法

通用性强：几乎所有语言都支持

适合作为系统语言基础能力（Rust的 `std::thread` ）

缺点：

创建/销毁成本较大

上下文切换开销大

大量并发时扩展性差：IO密集任务会浪费大量线程在阻塞等待上

同步复杂：锁/条件变量容易写出死锁、竞态条件

### （2）事件驱动（Event Loop + 回调）

**事件驱动的核心是事件循环（event loop）：I/O就绪、定时器触发等事件进入队列，再由回调处理** 。

优点：

性能高：上下文切换开销小

单线程可处理大量I/O（典型如 `Node.js` 风格）

缺点：

回调地狱：控制流分散，可读性差

错误传播/组合困难：异常处理复杂

复杂逻辑容易变得难维护

说明：现代 `async/await` 往往是为了解决事件驱动+回调在可维护性上的问题。

### （3）协程（Coroutine / 绿线程）

**协程是用户态可调度的执行单元** 。很多语言的协程是有栈协程或由运行时实现的“绿线程”。

优点：

写法接近同步代码，逻辑直观

切换成本低于OS线程

适合高并发：大量协程并存

缺点：

抽象层较高

需要语言/运行时深度支持

### （4）Actor

**程序由多个Actor构成，Actor之间通过消息传递通信，通常不共享可变状态** 。

优点：

天然减少共享内存问题：弱化锁竞争

分布式友好：结构贴近现实系统

模块化强

缺点：

流控复杂

错误处理、重试、消息顺序/一致性策略设计难

消息传递带来额外的开销与复杂度

### （5）async/await

async/await是一种让异步代码写起来接近于同步风格的语法糖。 **底层通常由状态机 + 轮询（poll）机制驱动，实现用户态的协作式调度** 。

优点（以Rust为代表）：

性能高：用户态调度 + 零成本抽象

语法接近同步，避免回调地狱

适合底层编程：runtime可自定义（tokio、smol等）

适用于大规模I/O并发

缺点：

实现机制复杂（Future、Pin、Waker、执行器等）

学习成本较高

## 2、Rust并发编程模型

Rust在并发上做了权衡：同时支持 `OS线程` 和 `async编程`:

`OS线程` 通过标准库 `std::thread` 实现，适合并行计算和少量并发任务

`async编程` 通过 `语言特性（async/await）+ 标准库基础trait + 第三方运行时` 实现。大量I/O并发时优先选择它。

`OS线程` 在多线程编程中已经具体介绍，本节主要围绕 `async编程` 。Rust异步编程具有以下关键特点：

Rust每个异步任务都实现了 `Future` 特征。 `Future` 是惰性的（lazy）：创建任务并不会立即执行，只有被执行器（executor）不断 `poll` 才会推进。具体来说，通过`.await` 语法，会把控制权交还给执行器，由执行器去 `poll` 这个 `Future` 直到完成。

**Rust async零成本** 。async/await会在编译期被展开为状态机，没有额外的运行时解释开销。但异步代码的性能具体取决于开发者使用的runtime、任务调度以及程序逻辑是否引入了阻塞。

`Future` 的运行需要 `异步运行时（runtime）` 支持：Rust标准库不内置完整runtime，社区提供成熟运行时库（如Tokio）。 **异步运行时通常支持单线程执行器（curret-thread）和线程池执行器（multi-thread）** 。

## 3、Rust异步编程生态与组成

Rust异步通常由以下层次共同组成：

语言层：async/await关键字，编译器生成状态机

标准库基础： `std::future::Future` 、 `core::task::{Context, Poll, Waker}` 等异步原语

futures crate：异步生态常用的基础工具与抽象（如 `Stream` 、 `Sink` 、 `FutureExt` 等）

第三方运行时（runtime）：提供执行器、I/O驱动、定时器、任务调度等：

`Tokio` ：最成熟流行，功能全面（调度器、网络、定时器、同步原语等）

`async-std` ：API风格接近std

`smol` ：轻量，简洁

## 4、同步/异步

同步（synchronous）：调用者发起操作后，要等结果出来才能继续向下走。

异步（asynchronous）：调用者发起操作后，先拿到一个将来会完成的东西（在Rust里就是 `Future` 、 `JoinHandle` ），当前流程可以继续向下做别的，等未来某个时刻再取结果。

## 5、阻塞/非阻塞

阻塞（Blocking）：线程在等待时OS挂起/睡眠，此时线程什么也做不了

非阻塞（Non-blocking）：等待不会让线程睡死，线程可以继续执行别的任务（例如async的任务调度）

## 6、OS线程与异步编程的选择

OS线程和async都能实现并发，但两者的代码结构和依赖不同， **切换成本高** ，所以项目选型很重要。一般经验如下：

**大量I/O并发（网络服务、爬虫等）** ：优先选async

**少量并发 + 逻辑简单：** OS线程更直接（需要时还可使用线程池）

**大量CPU密集并行计算** ：优先OS线程，线程数接近CPU核心数更合理

在async中执行CPU重活或阻塞操作时需要额外注意：

如果是CPU密集型任务（如大量图片/音视频处理、压缩、解压等），它们会持续占用CPU时间片。此时不适合直接将这类代码写到async任务中，否则会长时间占用异步运行时的工作线程，导致其他异步任务得不到及时调度

如果某些操作会阻塞线程（如同步文件I/O、同步网络I/O、 `std::thread::sleep` 或某些阻塞式第三方库调用），也不应该直接写到async任务中，因为线程直接休眠挂起，破坏了运行时的任务调度。通常应该使用 `spawn_blocking` 。

下面分别给出使用OS线程和使用async编程实现两个并发下载的伪代码，来对两者进行一个更深刻的理解：

OS线程

```rust
fn get\_two\_sites() {

let thread\_one = std::thread::spawn(|| download("https://www.foo.com"));

let thread\_two = std::thread::spawn(|| download("https://www.bar.com"));

thread\_one.join().expect("thread one panicked");

thread\_two.join().expect("thread two panicked");

}
```
如果是两个下载任务倒没什么，但一旦下载请求变多，线程数量膨胀：I/O等待期间线程阻塞，浪费资源且可能触达系统上限。

异步方式：
```rust
fn get\_two\_sites\_async() {

// 创建两个Future, 注意仅仅是创建，没有执行

let future\_one = download\_async("https://www.foo.com")

let future\_two = download\_async("https://www.bar.com");

// 并发推进两个Future

join!(future\_one, future\_two);

}
```
# 二、async/await入门

## 1、为什么需要async/await

来看这一段同步代码：

```rust
let data = read\_from\_network(); // 阻塞

process(data);
```
当 `read_from_network()` 在等待网络时，线程无法执行其他工作。

异步的目标是： **当遇到I/O等待时，让出当前执行权，让线程可以去推进其他任务，等I/O就绪后再回来继续执行原任务** 。
## 2、Future Trait

在Rust中，异步任务用 `Future` 表示，它表示将来某个时刻会完成的计算。

标准库中 `Future` 的简化定义如下：

```rust
trait Future {

type Output;

fn poll(self: Pin<&mut Self>, cx: &mut Context<'\_>) -> Poll<Self::Output>;

}
```
这里重点关注：

`type Output` ：Future完成时产生的结果类型

`Poll` 枚举：
```rust
enum Poll<T> {

Ready(T), // 完成

Pending, // 未完成

}

`poll` 方法语义：
```
返回 `Poll::Ready(val)` ：表示 `Future` 已经完成，得到val

返回 `Poll::Pending` ：表示暂未完成，需要稍后再 `poll`

当 `poll` 方法返回 `Pending` 时，含义不只是当前没有完成，更完整地说是：这个 `Future` 目前还不能继续推进，但它借助 `cx` 中的 `Waker` 注册好了一个唤醒机制；等到相关条件成熟时，底层资源调用 `Waker` 通知执行器，再由执行器重新 `poll` 这个 `Future` 。

一个最简单的 `Future` 示例：
```rust
async fn foo() -> i32 {

100

}

// impl Future<Output=i32>

let f = foo(); // 这里不是100，而是一个实现了Future的类型
```
## 2、async关键字

`async` 的作用，是把一段原本按顺序书写的逻辑（函数或代码块）封装成一个 `Future` 。它并不会在写下来的那一刻立即执行，而是先 **构造出一个将来可以被执行和推进的异步任务** 。

这说明，创建一个 `Future` ，并不会立刻运行它。 `Future` 本身是惰性的，只有在被 `poll` 时，才真正开始推进内部逻辑。

### （1）async块

示例：

```rust
let f = async {

println!("run");

10

};
```
这里的 `f` 不是10，是一个实现了 `Future` 的值（即 `impl Future<Output = i32>` ）。

但要注意， **上面这段代码本身并不会立刻打印** `run` 。因为 `async` 块只是创建了一个 `Future` ，真正的执行要等这个 `Future` 被`.await` ，或者被执行器反复 `poll` 之后才会发生，

**注意：** `f.await` **只能写在异步上下文中，也就是** `async fn` **或** `async {}` **内部** 。普通同步函数不能直接用`.await` 。
### （2）async fn

示例：

```text
async fn foo() -> i32 {

42

}
```
可以近似看作：
```rust
fn foo() -> impl Future<Output = i32> {

aysnc {

42

}

}

`async fn` 的含义是：这个函数被调用时，不会立即执行函数体并返回结果，而是先返回一个 `Future` ；等这个 `Future` 被执行器推进完成后，才得到 `i32` 类型的结果。
```
注意：

`foo()` 返回的是 `impl Future<Output = i32>`

`foo().await` 的结果才是 `i32`

常见变体：

`async move { ... }` ：把异步块捕获到的外部变量按值移动进 `Future` 内部。它的典型用途是让 `Future` 拥有自己所需的数据，从而避免依赖外部借用关系。这在需要把任务交给执行器独立运行时尤为常见，例如 `tokio::spawn` 。

`|| async { ... }` ：这种写法本质上还是一个闭包，只是这个闭包返回值恰好是一个异步任务。
## 3、.await

### （1）.await 的基本语义

`.await` 的作用是：在这里等待某个 `Future` 完成；如果它还没完成，就先暂停当前的异步任务，等将来可以继续时再恢复执行。

因此，`.await` 并不是一个普通方法调用，它更像是异步函数中的暂停点。

示例：

```rust
async fn example() {

let x = foo().await;

let y = bar().await;

println!("{x}, {y}");

}
```
这段代码的执行顺序可以理解为：

`foo()` 先返回一个 `Future`

`foo().await` 等待这个 Future 完成，并在完成后得到结果 `x`

然后再调用 `bar()` ，得到另一个 `Future`

`bar().await` 等待它完成，得到结果 `y`

最后继续执行 `println!`

所以，`.await` 的结果不是 Future 本身，而是这个 Future 完成后产出的值。
### （2）.await背后发生了什么

当执行到：

```text
foo().await
```
本质上，当前异步任务会把 `foo()` 返回的 `Future` 交给执行器去推进，也就是不断对它进行 `poll` 。

执行过程中会出现两种情况：

如果返回 `Poll::Ready(x)` ，说明这个 `Future` 已经完成，那么`.await` 表达式的值就是x，当前异步任务继续向下执行

如果返回 `Poll::Pending` ，说明这个 `Future` 暂时还不能完成，这时当前异步任务会在这个位置挂起，并把执行权交还给执行器。等底层条件成熟、任务被唤醒后，执行器再次 `poll` 它，随机再从这个`.await` 之后继续往下执行。
### （3）.await不阻塞线程

当 `.await` 等待的 Future 尚未完成时，发生的是： **当前异步任务被挂起，而不是当前 OS 线程被阻塞** 。

也就是说：

当前任务先让出执行权

runtime 可以利用这段时间去推进其他任务

等待的条件成熟后，再回来继续执行这个任务

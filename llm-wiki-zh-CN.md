# LLM Wiki

一种利用大语言模型构建个人知识库的模式。

这是一份概念性文档，设计初衷是供你复制粘贴到自己的 LLM Agent（如 OpenAI Codex、Claude Code、OpenCode / Pi 等）中使用。它的目标是传达高层思想，具体实现细节由你和你的 Agent 协作完善。

## 核心思想

大多数人与 LLM 和文档的交互体验类似于 RAG（检索增强生成）：你上传一堆文件，LLM 在查询时检索相关片段，然后生成答案。这能工作，但 LLM 每次都在从头重新发现知识，没有积累效应。当你问一个需要综合五份文档的微妙问题时，LLM 必须每次都找到并拼凑相关片段。什么都没有沉淀下来。NotebookLM、ChatGPT 文件上传和大多数 RAG 系统都是这样运作的。

这里的思路不同。与其仅在查询时从原始文档中检索，不如让 LLM **增量式地构建和维护一个持久化的 wiki**——一个结构化的、相互链接的 markdown 文件集合，位于你和原始素材之间。当你添加新素材时，LLM 不只是为后续检索而索引它。它会阅读它，提取关键信息，并将其整合进现有的 wiki 中——更新实体页面、修订主题摘要、标注新数据与旧说法的矛盾之处、强化或挑战不断演进的综合论述。知识被编译一次，然后*持续保持最新*，而非每次查询都重新推导。

这就是关键区别：**wiki 是一个持久化、复合增长的产物。** 交叉引用早已就位。矛盾之处已被标记。综合论述已经反映了你所阅读过的一切。wiki 随着你添加的每一个素材、提出的每一个问题而愈发丰富。

你几乎从不（或很少）亲自撰写 wiki——LLM 代劳所有工作。你负责素材筛选、探索挖掘和提出正确的问题。LLM 处理所有繁琐事务——正是这些让知识库真正随时间变得有用的总结、交叉引用、归档和 bookkeeping 工作。在实践中，我会在一侧打开 LLM Agent，另一侧打开 Obsidian。LLM 根据我们的对话进行编辑，我实时浏览结果——跟随链接、查看图谱视图、阅读更新后的页面。Obsidian 是 IDE；LLM 是程序员；wiki 是代码库。

这一模式适用于多种场景：

- **个人**：追踪你自己的目标、健康、心理学、自我提升——归档日记条目、文章、播客笔记，并随时间构建一幅关于你自己的结构化图景。
- **研究**：在数周或数月内深入研究某个主题——阅读论文、文章、报告，并增量式地构建一个带有演进论点的综合 wiki。
- **阅读书籍**：边读边归档每一章，为角色、主题、情节线索及其关联方式构建页面。到结束时，你会拥有一个丰富的 companion wiki。想想像 [Tolkien Gateway](https://tolkiengateway.net/wiki/Main_Page) 这样的粉丝 wiki——数千个相互链接的页面，涵盖角色、地点、事件、语言，由一群志愿者在数年间构建。你可以在阅读时亲自构建类似的东西，由 LLM 完成所有交叉引用和维护工作。
- **商业/团队**：由 LLM 维护的内部 wiki，素材来源包括 Slack 线程、会议记录、项目文档、客户电话。可能由人类审核更新。wiki 保持最新，因为 LLM 做了团队中没有人愿意做的维护工作。
- **竞争分析、尽职调查、旅行规划、课程笔记、爱好深挖**——任何你在随时间积累知识并希望它有条理而非散落各处的情况。

## 架构

共有三层：

**raw sources** —— 你精心策划的源文档集合。文章、论文、图片、数据文件。这些是不可变的——LLM 从中读取但从不修改。这是你的真相来源。

**wiki** —— 一个由 LLM 生成的 markdown 文件目录。摘要、实体页面、概念页面、对比分析、概览、综合论述。LLM 完全拥有这一层。它创建页面，在新素材到达时更新它们，维护交叉引用，并保持一切一致。你阅读它；LLM 撰写它。

**schema** —— 一份文档（如 Claude Code 的 CLAUDE.md 或 Codex 的 AGENTS.md），告诉 LLM wiki 的结构方式、约定规范是什么，以及在 ingest 素材、回答问题或维护 wiki 时应遵循哪些工作流。这是关键的配置文件——它让 LLM 成为有纪律的 wiki 维护员，而非普通的聊天机器人。你和 LLM 在探索适合你的领域的过程中共同演进这份文档。

## 操作

**Ingest.** 你将新素材放入 raw sources 集合并告诉 LLM 处理它。一个示例流程：LLM 阅读素材，与你讨论关键要点，在 wiki 中撰写摘要页面，更新 index，更新 wiki 中相关的实体和概念页面，并在 log 中追加条目。单个素材可能触及 10-15 个 wiki 页面。我个人倾向于一次 ingest 一个素材并保持参与——我阅读摘要、检查更新，并引导 LLM 决定重点强调什么。但你也可以批量 ingest 多个素材，减少监督。由你开发适合自己风格的工作流，并将其记录在 schema 中供未来会话使用。

**Query.** 你向 wiki 提问。LLM 搜索相关页面，阅读它们，并综合出一个带引用的答案。答案可以根据问题采取不同形式——markdown 页面、对比表格、幻灯片（Marp）、图表（matplotlib）、画布。重要的洞察是：**好的答案可以作为新页面归档回 wiki 中。** 你要求的一次对比、一次分析、一个你发现的关联——这些都有价值，不应消失在聊天记录中。这样，你的探索就像 ingest 的素材一样，在知识库中复合增长。

**Lint.** 定期让 LLM 对 wiki 进行健康检查。查找：页面之间的矛盾、被新素材取代的陈旧说法、没有入站链接的孤立页面、被提及但缺少独立页面的重要概念、缺失的交叉引用、可以通过网络搜索填补的数据空白。LLM 善于建议新的调查问题和新素材来源。这能让 wiki 在增长过程中保持健康。

## Index 与 Log

两个特殊文件帮助 LLM（和你）在 wiki 增长时进行导航。它们服务于不同目的：

**index.md** 以内容为导向。它是 wiki 中所有内容的目录——每个页面都带有链接、一行摘要，以及可选的元数据如日期或素材数量。按类别组织（实体、概念、素材等）。LLM 在每次 ingest 时更新它。回答 query 时，LLM 先阅读 index 找到相关页面，然后深入阅读。这在中等规模（~100 个素材，~数百个页面）下出奇地有效，避免了基于嵌入的 RAG 基础设施的需求。

**log.md** 以时间为导向。它是按时间顺序的、仅追加的记录——记录了发生了什么、何时发生——ingest、query、lint。一个有用的技巧：如果每个条目以一致的前缀开头（例如 `## [2026-04-02] ingest | 文章标题`），log 就可以用简单的 Unix 工具解析——`grep "^## \[" log.md | tail -5` 就能给出最近 5 条条目。log 提供了 wiki 演进的时间线，并帮助 LLM 理解最近做了什么。

## 可选：CLI 工具

在某些时候，你可能想构建一些小工具来帮助 LLM 更高效地操作 wiki。最明显的需求是 wiki 页面的搜索引擎——在小规模下 index 文件就足够了，但随着 wiki 增长，你需要真正的搜索。[qmd](https://github.com/tobi/qmd) 是一个不错的选择：它是一个本地 markdown 文件搜索引擎，带有混合 BM25/向量搜索和 LLM 重排序，全部在本地运行。它既有 CLI（因此 LLM 可以调用它），也有 MCP 服务器（因此 LLM 可以将其作为原生工具使用）。你也可以自己构建更简单的工具——LLM 可以在需要时帮你快速开发一个朴素搜索脚本。

## 技巧与窍门

- **Obsidian Web Clipper** 是一个浏览器扩展，可以将网页文章转换为 markdown。对于快速将素材放入你的 raw sources 集合非常有用。
- **本地下载图片。** 在 Obsidian 设置 → 文件与链接中，将"附件文件夹路径"设为固定目录（例如 `raw/assets/`）。然后在设置 → 快捷键中，搜索"Download"找到"为当前文件下载附件"并绑定一个快捷键（例如 Ctrl+Shift+D）。剪辑文章后，按下快捷键，所有图片都会下载到本地磁盘。这是可选但有用的——它让 LLM 可以直接查看和引用图片，而非依赖可能失效的 URL。注意 LLM 不能原生地在一次阅读中读取带有内嵌图片的 markdown——解决方法是让 LLM 先阅读文本，然后单独查看部分或全部引用的图片以获得额外上下文。这有点笨拙，但足够好用。
- **Obsidian 的图谱视图** 是查看 wiki 形状的最佳方式——什么与什么相连，哪些是枢纽页面，哪些是孤立页面。
- **Marp** 是一种基于 markdown 的幻灯片格式。Obsidian 有它的插件。对于直接从 wiki 内容生成演示文稿很有用。
- **Dataview** 是一个 Obsidian 插件，可以在页面 frontmatter 上运行查询。如果你的 LLM 为 wiki 页面添加 YAML frontmatter（标签、日期、素材数量），Dataview 可以生成动态表格和列表。
- wiki 就是一个 markdown 文件的 git 仓库。你免费获得了版本历史、分支和协作功能。

## 为什么这能工作

维护知识库的繁琐之处不在于阅读或思考——而在于 bookkeeping。更新交叉引用、保持摘要最新、标注新数据何时与旧说法矛盾、在数十个页面之间保持一致性。人类放弃 wiki 是因为维护负担的增长速度超过了价值。LLM 不会感到无聊，不会忘记更新交叉引用，并且可以一次触及 15 个文件。wiki 得以保持维护，因为维护成本近乎为零。

人类的工作是策划素材、指导分析、提出好问题、思考这一切意味着什么。LLM 的工作是其他一切。

这一思想在精神上与 Vannevar Bush 的 Memex（1945）相关——一个个人的、精心策划的知识存储库，带有文档之间的关联轨迹。Bush 的愿景更接近于此，而非后来的万维网：私密的、主动策划的，文档之间的连接与文档本身同样有价值。他没能解决的部分是谁来做维护工作。LLM 处理了这部分。

## 说明

本文档故意保持抽象。它描述的是思想，而非具体实现。确切的目录结构、规范约定、页面格式、工具——所有这些都取决于你的领域、偏好和所选的 LLM。上面提到的所有内容都是可选和模块化的——挑选有用的，忽略不适用的。例如：你的素材可能只有文本，所以你根本不需要图片处理。你的 wiki 可能小到 index 文件就够用了，不需要搜索引擎。你可能不关心幻灯片，只想要 markdown 页面。你可能想要一套完全不同的输出格式。正确的使用方式是将它与你的 LLM Agent 分享，共同协作实例化一个适合你的版本。本文档的唯一职责是传达模式。你的 LLM 可以搞定其余部分。

# 一、HTTP简介

## 1、HTTP基本概念

`HTTP`，全称 Hyper Text Transfer Protocol，即超文本传输协议。

`HTTP`是一种运行在**应用层**的网络协议，主要用于客户端与服务器之间传输资源。它最初主要服务于万维网，也就是浏览器向Web服务器请求网页资源。

这里的资源可以包括：

- 网页相关资源，例如HTML、CSS、JavaScript
- 媒体资源，例如图片、音频、视频
- 数据资源，例如JSON、XML、API接口返回结果
- 文件资源，例如文档、压缩包、下载文件

现在，HTTP已经不只是浏览器访问网页时使用的协议，也广泛应用于：

- 前后端接口通信
- 移动端与服务端通信
- 第三方开放 API 调用
- 微服务之间通信
- Webhook 回调
- 爬虫程序抓取网页
- AI服务接口调用

因此，对于程序员来说，HTTP不是一个单纯的“浏览器知识”，而是现代 Web 开发中非常基础的一类网络通信知识。


## 2、HTTP核心作用

HTTP的核心作用是：**让客户端和服务器之间能够按照统一格式交换数据**。

用户在浏览器中访问一个网页：

```text
浏览器 / 客户端                         Web 服务器
     │                                      │
     │  发送 HTTP 请求                      │
     │ ───────────────────────────────────▶ │
     │                                      │
     │  返回 HTTP 响应                      │
     │ ◀─────────────────────────────────── │
     │                                      │
     │  浏览器解析响应内容并展示页面              │
```

前端页面调用后端登录接口：

```text
前端页面 / 浏览器                         后端服务
      │                                      │
      │  POST /api/login                    │
      │ ──────────────────────────────────▶ │
      │                                      │
      │  返回登录结果 JSON                  │
      │ ◀────────────────────────────────── │
      │                                      │
      │  根据响应结果更新页面状态            │
      │
```

后端调用第三方 API 服务：

```text
业务后端服务                              第三方 API 服务
      │                                      │
      │  发送 HTTP API 请求                 │
      │ ──────────────────────────────────▶ │
      │                                      │
      │  返回 HTTP API 响应                 │
      │ ◀────────────────────────────────── │
      │                                      │
      │  解析响应并继续处理业务逻辑          │
      │
```

HTTP 规定了一套通用规则：

- 客户端如何发起请求
- 服务器如何返回响应
- 请求和响应的格式是什么
- 请求方法如何表达操作意图
- 状态码如何表达处理结果
- Header 如何携带附加信息
- Body 如何传输实际数据

> HTTP 本质上是一套客户端和服务器之间交换信息的约定。


## 3、HTTP 的应用场景

只要进行Web开发，就很难绕开HTTP。

前端开发中，常见场景包括：

- 浏览器加载页面资源
- 使用 fetch 或 axios 请求后端接口
- 处理 Cookie、跨域、缓存等问题
- 分析 DevTools Network 面板中的请求

后端开发中，常见场景包括：

- 编写 HTTP API
- 接收请求参数
- 返回JSON响应
- 处理状态码
- 处理认证、鉴权、跨域、缓存
- 部署服务到 Nginx、网关或负载均衡后面

移动端、爬虫、云服务、AI 应用开发中，也经常需要直接或间接使用 HTTP。

学习 HTTP 的目的不是背概念，而是为了以后能看懂和处理这些问题：

- 为什么接口返回404？
- 为什么登录后能保持状态？
- 为什么浏览器报跨域错误？
- 为什么有些请求返回304？
- 为什么接口需要 Authorization 头？
- 为什么 HTTPS 比 HTTP 安全？


# 二、HTTP的基本模型

## 1、客户端-服务器模型

HTTP 是一种**客户端-服务器**协议。

基本模型如下：

![[Pasted image 20260512210519.png|300]]

**客户端主动发起请求，服务器负责接收请求、处理请求，并返回响应**。

常见的客户端包括：浏览器、curl、Postman、手机App、前端JavaScript程序、后端服务、爬虫程序、自动化测试脚本。

服务器负责提供资源或处理业务逻辑：

- 返回HTML页面
- 返回CSS、JavaScript、图片等静态资源
- 查询数据库后返回JSON
- 处理登录请求
- 返回文件下载内容
- 返回AI模型生成结果

> 客户端不一定是浏览器，服务器也不一定是传统意义上的网站服务器。


## 2、请求-响应模型

HTTP 最核心的交互模式是**请求-响应**模型。

![[Pasted image 20260512214119.png|300]]

客户端发送请求：

```http
GET /index.html HTTP/1.1
Host: Example.com
```

服务器返回响应：

```http
HTTP/1.1 200 OK
Content-Type: text/html

<html>
  <body>Hello HTTP</body>
</html>
```

这就是一次最基本的 HTTP 通信。

其中：

- `GET /index.html HTTP/1.1` 表示客户端想获取 `/index.html` 这个资源；
- `Host: example.com` 表示请求的目标主机；
- `HTTP/1.1 200 OK` 表示服务器成功处理请求；
- `Content-Type: text/html` 表示响应体是 HTML 文档；
- 空行后面的内容就是响应体。

HTTP 通信不是客户端和服务器之间随意发送数据，而是通过一个个结构化的消息完成。

客户端发出的消息称为 **请求**，服务器返回的消息称为 **响应**。


## 3、面向资源基本思想

HTTP 不是围绕“远程函数调用”设计的，而是围绕**资源**设计的。

这个资源可以是：

- 一个网页
- 一张图片
- 一个用户
- 一篇文章
- 一个订单
- 一个JSON数据

资源通常通过 `URL` 标识：

```text
/users/1  
/articles/100  
/orders/20260512001  
/images/logo.png
```

客户端通过 **HTTP方法** 说明要对资源执行什么操作：

```text
GET    /users/1       获取用户
POST   /users         创建用户
PUT    /users/1       整体更新用户
PATCH  /users/1       局部更新用户
DELETE /users/1       删除用户
```

各要素之间的关系：

- URL 表示资源
- HTTP 方法表示操作
- 状态码表示结果
- Header 表示附加信息
- Body 表示具体数据


## 4、应用层协议定位

HTTP 位于网络分层模型中的**应用层**。下图是经典的五层协议：

![[Pasted image 20260512214427.png|200]]

它不直接负责底层数据如何传输，而是建立在更底层的协议之上。

HTTP关注的是请求是什么、响应是什么、资源是什么、状态码是什么、Header是什么、Body是什么。

而底层协议关注的是：数据怎么分片、数据怎么传输、数据丢了怎么办、连接怎么建立、网络地址怎么寻找。

## 5、HTTP 与 TCP、TLS、QUIC的关系

传统HTTP/1.1建立在TCP协议之上，HTTP/2.0也建立在TCP协议之上。

HTTPS 在 HTTP 和 TCP 之间增加了 TLS 层：

![[Pasted image 20260512215314.png|200]]

HTTPS 仍然使用 HTTP 请求、响应、方法、状态码、Header、Body 等概念，**只是在 HTTP 和 TCP 之间增加了 TLS，用于加密、完整性校验和身份验证**。

HTTP/3则基于 QUIC：
![[Pasted image 20260512215555.png|200]]

QUIC 虽然运行在 UDP 之上，但它实现了可靠传输、加密、拥塞控制等能力。

# 三、HTTP系统的组成

## 1、客户端

客户端是发起 HTTP 请求的一方。

最常见的客户端是浏览器，如 Chrome、Edge、Firefox、Safari 等。但客户端不一定是浏览器，只要一个程序能构造 HTTP 请求并接收 HTTP 响应，就可以作为 HTTP 客户端。

使用 `curl` 命令请求一个网页：

```shell
curl https://example.com
```

这里的 `curl` 就是 HTTP 客户端。

前端代码中使用 `fetch` 请求接口：

```JavaScript
fetch('/api/users')
  .then(response => response.json())
  .then(data => console.log(data));
```

这里运行 JavaScript 的浏览器就是客户端。

在后端服务之间调用接口时，发起请求的后端服务也可以看作是 HTTP 客户端。例如，订单服务向用户服务发送 HTTP 请求，用户服务返回 HTTP 响应，这里的订单服务就是 HTTP 客户端。


## 2、用户代理

用户代理，英文名叫 User Agent。它指的是**代表用户发起 HTTP 请求的客户端程序**。常见用户代理包括浏览器、搜索引擎爬虫、curl、Postman、移动端APP、自动化测试工具。

HTTP 请求中常见的 `User-Agent` 请求头，就是用来描述客户端身份的：

```http
User-Agent: Mozilla/5.0
```

服务器可以根据 `User-Agent` 大致判断请求来自哪类客户端。

> `User-Agent` 可以被伪造，不应作为严格的安全凭证，更适合用于统计、兼容性处理和日志记录。

## 3、服务器

服务器是接收请求并返回响应的一方。

服务器负责提供客户端请求的资源，或者根据请求执行业务逻辑。

这些资源可以是**静态文件**（HTML、CSS、JavaScript、图片、字体、视频、下载文件等），也可以是**动态生成的结果**：

- 查询数据库后返回用户信息；
- 登录接口返回Token
- 电商网站返回商品列表；
- 后端接口返回JSON
- AI模型返回生成结果

在实际系统里，服务器可能是一台机器，也可能是多台机器组成的集群。如下图所示：

![[Pasted image 20260512222206.png|500]]

负载均衡器可以理解为一个“前台接待员”或“分流器”。用户的请求先到负载均衡器，负载均衡器再决定把请求转发到哪一台后端服务器。而这些后端服务器一般连接共享资源，如数据库、缓存、消息队列等。

## 4、Web服务器和应用服务器

在实际开发中，经常会听到两个概念：Web服务器和应用服务器。它们有时会混用，区别如下：

- **Web服务器**：更偏向处理 HTTP 层面的请求和静态资源。常见的Web服务器包括Nginx、Apache、Caddy。
- **应用服务器**：更偏向于运行业务代码，如 Node.js 服务、Java Spring Boot 服务、Rust Web 服务等。

![[Pasted image 20260512223046.png]]


## 5、代理

在客户端和服务器之间，可能还会经过一些中间实体，这些实体叫做**代理**。

![[Pasted image 20260512233510.png|500]]


代理本质上位于通信中间，负责转发、处理或控制 HTTP 消息。

CDN 就是一种常见的缓存代理。如果用户请求了一个图片资源，而 CDN 节点已经缓存了该图片，那么 CDN 就可以直接返回资源，不必每次都回源到真实服务器。

企业内部网络也可能使用代理控制员工访问外部网络的行为。

在后端系统中，API网关、负载均衡器、Nginx反向代理，也都可以看作HTTP 通信链路中的中间层。

## 6、正向代理和反向代理

代理可以分为很多类型，其中最常见的就是**正向代理**和**反向代理**。

正向代理是站在客户端一侧的代理。

请求路径类似：

反向代理是站在服务器一侧的代理。

# 四、HTTP基本性质

1. 简单易读

HTTP 的设计相对简单，尤其是 HTTP/1.1 以及更早版本中，请求和响应报文都具有较强的文本可读性。

请求示例：

```http
GET / HTTP/1.1  
Host: example.com  
Accept: text/html
```

响应示例：

```http
HTTP/1.1 200 OK
Content-Type: text/html

<html>Hello HTTP</html>
```

虽然 HTTP/2 和 HTTP/3 的底层传输格式已经不再是简单文本报文，但 HTTP 的核心语义没有变化，仍然有请求、响应、方法、状态码等。

2. 灵活

HTTP 可以传输各种类型的数据。具体数据类型通过 `Content-Type` 表示：

- `text/html`：响应体是HTML
- `application/json`：响应体是JSON
- `image/png`：响应体是PNG图片
- `multipart/form-data`：响应体是表单和文件上传数据

3. 可扩展

HTTP是一种可扩展协议，它可以通过 Header 扩展很多能力。

```http
Cache-Control: max-age=3600
```

可以控制缓存。

```http
Authorization: Bearer token123
```

可以携带认证信息。

```http
Set-Cookie: session_id=abc123
```

可以让浏览器保存 Cookie。

4. 无状态

HTTP是无状态协议。所谓无状态是指，HTTP本身不会自动记住前一次请求的信息。

服务器收到两个请求：

```text
请求 1：POST /login
请求 2：GET /profile
```

HTTP 协议本身不会自动知道这两个请求是不是同一个用户发来的。

这会带来一些问题：

- 用户登录后，服务器怎么知道后续请求还是这个用户？
- 用户把商品加入购物车后，服务器怎么知道购物车属于谁？
- 用户切换页面时，网站怎么保持登录状态？

不过，虽然HTTP本身无状态，但Web应用可以借助 Cookie / Session /Token建立有状态的会话。

5. 依赖可靠传输

传统 HTTP 主要依赖 TCP，因为 TCP 提供可靠传输能力。HTTP/3 则依赖 QUIC。QUIC 虽然基于 UDP，但它自己提供可靠传输、安全加密、拥塞控制等能力。

因此，HTTP 对底层传输的核心要求不是“必须是 TCP”，而是：

> 底层传输应该能够可靠地交付 HTTP 消息，或者在失败时明确报告错误。

# 五、一次完整的HTTP访问过程

## 1、URL解析

以访问这个地址为例：

```text
https://www.example.com/articles/1
```

浏览器首先会解析URL：

- 协议：https
- 主机：www.example.com
- 端口：443
- 路径：/articles/1

如果 URL 中没有显式写端口，则使用协议默认端口。

- HTTP 默认为 80
- HTTPS 默认为 443

**URL 是客户端定位资源的重要方式**。其中，协议决定使用 HTTP 还是 HTTPS，主机决定访问哪台服务器，端口决定连接服务器上的哪个服务，路径决定请求服务器上的哪个资源。

## 2、DNS解析

浏览器需要把域名解析成 IP 地址：

```text
www.example.com → 93.184.216.34
```

因为真实网络通信需要依靠 IP 地址找到目标服务器。

域名是给人看的地址，IP 是给网络通信用的地址。

如果浏览器或系统缓存中已经有对应解析结果，就可以直接使用缓存；否则就需要向 DNS 服务器查询。

## 3、连接建立

浏览器拿到服务器IP地址之后，需要和服务器建立连接。

如果是 HTTP/1.1 或 HTTP/2，通常需要建立 TCP 连接。如果是 HTTPS，还需要进行 TLS 握手。

![[Pasted image 20260513003914.png]]

如果是 HTTP/3，则底层使用 QUIC。

连接建立完成后，客户端和服务器之间才可以正式传输 HTTP 请求和响应。


## 4、HTTP请求发送

连接建立之后，浏览器构造 HTTP 请求并发送给服务器。


```http
GET /articles/1 HTTP/1.1
Host: www.example.com
Accept: text/html
User-Agent: Mozilla/5.0
```



## 5、服务器处理请求

服务器收到请求后，会根据请求内容进行处理。如请求 `/articles/1`，服务器可能会去数据库中查询 ID 为 1 的文章，然后生成 HTML 页面或 JSON 数据返回给客户端。

如果请求的是静态资源，如 `/logo.png`、`style.css`，服务器会直接从文件系统或缓存读取资源并返回。

如果请求的是动态接口，如 `/api/users/1`，服务器通常会执行业务代码，查询数据库，然后返回 JSON 响应。

## 6、HTTP响应返回

服务器处理完成后，会返回 HTTP 响应：

```http
HTTP/1.1 200 OK
Content-Type: text/html
Cache-Control: max-age=600

<html>
  <body>文章内容</body>
</html>
```

## 7、客户端解析响应

浏览器收到 HTTP 响应后，解析响应内容。如果响应体是 HTML，浏览器会解析页面。如果 HTML 中引用了其他资源，则继续发送更多 HTTP 请求。

HTML 示例：

```html
<link rel="stylesheet" href="/style.css">
<script src="/main.js"></script>
<img src="/logo.png">
```

浏览器会继续发起请求：

```text
浏览器 / 客户端                         Web 服务器
     │                                      │
     │  GET /style.css                      │
     │ ───────────────────────────────────▶ │
     │  返回 CSS 文件                       │
     │ ◀─────────────────────────────────── │
     │                                      │
     │  GET /main.js                        │
     │ ───────────────────────────────────▶ │
     │  返回 JavaScript 文件                │
     │ ◀─────────────────────────────────── │
     │                                      │
     │  GET /logo.png                       │
     │ ───────────────────────────────────▶ │
     │  返回图片资源                        │
     │ ◀─────────────────────────────────── │
     │                                      │
```

所以，打开一个网页通常不是一次 HTTP 请求完成的，而是很多个 HTTP 请求共同完成的。

这也是为什么浏览器 DevTools 的 Network 面板中，经常能看到几十个甚至上百个请求。
